[[thinking]]
The atomicity question matters here. Read-then-write from Python means two concurrent requests can both read the count below the limit and both proceed. Redis pipelines don't fix that on their own — I need either INCR with expiry or a Lua script so the check and increment happen as one operation.

[[response]]
Implemented in `app/mw/ratelimit.py`. The core is a Lua script so the check and increment are atomic:

```lua
local n = redis.call('INCR', KEYS[1])
if n == 1 then
  redis.call('PEXPIRE', KEYS[1], ARGV[1])
end
return n
```

A Python-side read-then-write would let two concurrent requests both observe `n < limit` and both pass. This can't.

The middleware itself:

```python
class RateLimit:
    def __call__(self, request):
        key = f"rl:{request.user.id}" if request.user else f"rl:ip:{request.client_ip}"
        n = self.script(keys=[key], args=[WINDOW_MS])
        if n > LIMIT:
            return JsonResponse({"error": "rate_limited"}, status=429,
                                headers={"Retry-After": str(WINDOW_S)})
        return self.next(request)
```

`Retry-After` is included because clients that respect it back off on their own, which is most of the benefit of returning 429 at all.

Two things I decided against, both worth your input:

- **No limit headers on success** (`X-RateLimit-Remaining`). They're useful, but they'd mean a Redis read on every request that's currently free.
- **IP keying trusts `request.client_ip`.** If you're behind a proxy that isn't stripping `X-Forwarded-For`, that's spoofable and anonymous users could evade the limit. Worth checking your ingress config.
