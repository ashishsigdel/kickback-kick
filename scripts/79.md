# Script 79 — optimistic locking that silently drops concurrent updates

[[thinking]]
"Silently drops" with optimistic locking is suspicious on its face — JPA's `@Version` is supposed to throw `OptimisticLockException` on a stale write, which is loud, not silent. If updates are vanishing without an exception anywhere, either the version check isn't actually engaging (wrong field, wrong scope) or something is catching and swallowing the exception downstream. I need to see both the entity and the save path before guessing which.

  description: Find the versioned entity and anywhere the optimistic lock exception is handled

[[response]]
The entity is set up correctly — `InventoryItem.java:14`:

```java
@Version
private Long version;
```

The problem is the save path, `InventoryService.java:88`:

```java
@Transactional
public void adjustStock(Long itemId, int delta) {
    try {
        InventoryItem item = repository.findById(itemId).orElseThrow();
        item.setQuantity(item.getQuantity() + delta);
        repository.save(item);
    } catch (OptimisticLockException e) {
        log.warn("stock adjustment conflict for item {}, skipping", itemId);
    }
}
```

It does throw — and it's caught, logged at `WARN`, and swallowed. The caller (`OrderFulfillmentJob.java:41`) calls `adjustStock` and treats a normal return as success. Under concurrent updates to the same item, the loser of the race gets its stock adjustment silently discarded, and the job that called it believes the adjustment happened. There's no retry, no re-fetch, no propagation — the WARN log is the only trace, and nobody alerts on WARN.

[[thinking]]
The two real options are retry-on-conflict (re-fetch the current version and reapply the delta) or propagate the failure so the caller decides. For a numeric delta like stock adjustment, retry is almost always correct — the operation is commutative and idempotent to retry (adjust by +/-N again against fresh state), unlike a "set to absolute value" write where blind retry could double-apply. I want to confirm `adjustStock` really is always a relative delta before assuming retry is safe, not an absolute set in disguise somewhere.

[[response]]
Checked every call site of `adjustStock` (`rg -n "adjustStock\(" -g "*.java"`) — all four callers pass a relative delta (`+qty`, `-qty`), never an absolute target. Retry-on-conflict is safe here: reapplying `delta` against a freshly-read `version` doesn't double-count anything, it's the same operation against current state.

```diff
 @Transactional
 public void adjustStock(Long itemId, int delta) {
-    try {
-        InventoryItem item = repository.findById(itemId).orElseThrow();
-        item.setQuantity(item.getQuantity() + delta);
-        repository.save(item);
-    } catch (OptimisticLockException e) {
-        log.warn("stock adjustment conflict for item {}, skipping", itemId);
-    }
+    RetryTemplate.builder()
+        .maxAttempts(3)
+        .retryOn(OptimisticLockException.class)
+        .build()
+        .execute(ctx -> {
+            InventoryItem item = repository.findById(itemId).orElseThrow();
+            item.setQuantity(item.getQuantity() + delta);
+            repository.save(item);
+            return null;
+        });
 }
```

Three attempts before giving up — at that point it's no longer a transient race, it's persistent contention worth surfacing, so I let the exception propagate on final failure instead of catching it a second time.

[[thinking]]
Retrying inside the same `@Transactional` method is worth double-checking — Spring's proxy-based transaction management means `@Transactional` wraps the whole method in one transaction by default, and if `RetryTemplate` re-runs the lambda inside that same outer transaction, a failed attempt's changes might not actually roll back between retries the way I'm assuming.

[[response]]
Checked — this matters and needed fixing. `repository.save(item)` inside the retry lambda was running under the single outer `@Transactional` transaction spanning all three attempts, not a fresh transaction per attempt. On a stale-version failure, the transaction doesn't automatically clear its persistence context between retries, so the second attempt's `findById` could return the same stale entity from Hibernate's first-level cache instead of re-reading the current row.

Fixed by giving each retry attempt its own transaction, so a failed attempt fully rolls back and the next one reads fresh:

```diff
-@Transactional
 public void adjustStock(Long itemId, int delta) {
     RetryTemplate.builder()
         .maxAttempts(3)
         .retryOn(OptimisticLockException.class)
         .build()
-        .execute(ctx -> {
-            InventoryItem item = repository.findById(itemId).orElseThrow();
-            item.setQuantity(item.getQuantity() + delta);
-            repository.save(item);
-            return null;
-        });
+        .execute(ctx -> adjustStockOnce(itemId, delta));
 }
+
+@Transactional(propagation = Propagation.REQUIRES_NEW)
+public void adjustStockOnce(Long itemId, int delta) {
+    InventoryItem item = repository.findById(itemId).orElseThrow();
+    item.setQuantity(item.getQuantity() + delta);
+    repository.save(item);
+}
```

`REQUIRES_NEW` on the inner method forces a fresh transaction (and fresh persistence context) per attempt, so a stale read can't survive across retries.
