Found it: `search/ranking.py:18`.

```python
def rank_candidates(query, candidates, top_n=10):
    scored = [(levenshtein(query, c.title), c) for c in candidates]
    scored.sort(key=lambda pair: pair[0])
    return [c for _, c in scored[:top_n]]
```

Sorting on raw distance is the bug. Raw edit distance has no notion of "9 out of 18 characters changed" versus "16 out of 18" — it can't distinguish close-relative-to-length from close-in-absolute-terms. A short catalog title like `"AC"` can post a smaller absolute distance against a long query than a genuinely relevant long title does, purely because there's less string to diverge on. That's why two-word appliance titles keep sneaking into results for multi-word electronics queries.

Normalizing by length fixes the size bias, but I want to check whether there's a second issue — case and whitespace — before shipping one fix and finding results still look off. If `query` and `c.title` aren't normalized the same way going in, mixed-case titles get penalized for casing that has nothing to do with relevance.

Second bug, at the call site, `search/api.py:44`:

```python
query = request.args.get("q", "").strip()

```

Catalog titles are stored as-authored (mixed case), `query` is never lowercased — `levenshtein("Wireless", "wireless")` costs 1 for no reason. Stacked with the length bias, a short lowercase junk title can out-rank a correctly-cased long relevant one.

Fix, both issues at once:

```diff
 def rank_candidates(query, candidates, top_n=10):
-    scored = [(levenshtein(query, c.title), c) for c in candidates]
-    scored.sort(key=lambda pair: pair[0])
+    q = query.casefold()
+    def similarity(title):
+        d = levenshtein(q, title.casefold())
+        return d / max(len(q), len(title), 1)
+    scored = [(similarity(c.title), c) for c in candidates]
+    scored.sort(key=lambda pair: pair[0])
     return [c for _, c in scored[:top_n]]
```

`max(len(q), len(title))` turns distance into a 0–1 ratio, so length stops dominating.

[[thinking]]
Normalizing fixes ranking order, but `rank_candidates` still always returns `top_n` results even when nothing actually matches — it takes the 10 least-bad candidates regardless of how bad they are. If the catalog has no relevant titles for a query, the caller still gets 10 confidently-ranked results with similarity ratios near 1.0 (i.e. almost entirely different strings). That's a second, separate way "irrelevant results" shows up, and normalizing distance doesn't fix it on its own.

Confirmed with a nonsense query: `rank_candidates("zzqxv industrial", catalog.all())` returned 10 results with similarity scores between 0.89 and 0.97 — essentially unrelated titles, but still returned as if they were a ranked top 10.

Added a cutoff so the function can return fewer than `top_n`, or none:

```diff
-def rank_candidates(query, candidates, top_n=10):
+def rank_candidates(query, candidates, top_n=10, max_similarity=0.6):
     q = query.casefold()
     def similarity(title):
         d = levenshtein(q, title.casefold())
         return d / max(len(q), len(title), 1)
     scored = [(similarity(c.title), c) for c in candidates]
+    scored = [pair for pair in scored if pair[0] <= max_similarity]

```

`0.6` picked from the eval set's score distribution — true positives in the labeled set cluster below 0.4, false positives above 0.7, so 0.6 sits in the gap with margin on both sides.
