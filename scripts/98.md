[[thinking]]
A Bloom filter in front of a DB lookup only pays for itself if the false-positive rate is tuned to the actual key cardinality and if the filter is kept in sync with the underlying data — a stale or undersized filter either does nothing (too many false positives, falls through to the DB anyway) or, worse, causes real misses if anyone mistakes "maybe present" for "definitely present." I want to see how it's sized and how it's populated before judging the logic that consumes it.

[[response]]
Diff adds a Bloom filter in front of `SeenBefore(eventID string) bool`, previously a straight `SELECT 1 FROM processed_events WHERE id = $1`. Goal is avoiding that DB round-trip for the common case (most event IDs are new, so most existence checks return false, and a Bloom filter answers "definitely not present" without touching the DB). Sound approach for this access pattern. One correctness issue and one sizing issue.

**1. Correctness — `internal/dedupe/check.go:19`, filter result treated as authoritative on the positive side:**

```go
func SeenBefore(eventID string) bool {
    if !filter.Test([]byte(eventID)) {
        return false // definitely not present, skip DB
    }
    return true // <-- wrong: filter says "maybe", not "yes"
}
```

A Bloom filter has no false negatives but *does* have false positives — `filter.Test` returning true means "maybe present," not "present." Treating that as a confirmed "yes" means some genuinely new event IDs will be incorrectly reported as duplicates and silently dropped, purely based on a hash collision in the filter, never touching the DB to confirm. That's a correctness regression, not just a missed optimization — the previous DB-only version never had false positives.

**2. Sizing — `internal/dedupe/bloom.go:11`:**

```go
filter := bloom.NewWithEstimates(1_000_000, 0.01)
```

Fixed at 1M expected items, 1% target false-positive rate, set once at startup and never resized. `processed_events` is queried elsewhere in the same PR's description as "growing ~2M rows/month" — at 6 months this filter is sized for half the actual item count it'll hold, and the real false-positive rate degrades well past 1% as more items are added beyond what it was sized for (Bloom filters get *leakier* than their target rate once you exceed the `n` they were built for, not just proportionally worse).

[[thinking]]
Of the two, the correctness bug is the one that has to block merge — it silently drops real data. The sizing issue degrades gracefully rather than corrupting anything, so it's worth separating the two in my recommendation rather than treating them as equally urgent.

[[response]]
Two fixes, in order of urgency.

**Correctness (required before merge):** a filter hit must fall through to the DB to confirm, not short-circuit as true. This is the entire point of a Bloom filter used for this pattern — it only saves work on the negative case:

```diff
 func SeenBefore(eventID string) bool {
     if !filter.Test([]byte(eventID)) {
-        return false // definitely not present, skip DB
+        return false // definitely not present, skip DB
     }
-    return true // <-- wrong: filter says "maybe", not "yes"
+    return db.Exists(eventID) // filter said "maybe" — confirm against source of truth
 }
```

This still gets the performance win for the common case (new events skip the DB entirely) and adds a DB call only for the minority the filter flags as possible duplicates — which, at a well-tuned false-positive rate, is close to the true duplicate rate plus a small collision margin, not close to 100% of traffic.

**Sizing (should fix before this sees real growth, not blocking today):** size for a growth horizon, not current count, and note when it needs rebuilding:

```go
filter := bloom.NewWithEstimates(30_000_000, 0.01) // ~15 months of growth at 2M/mo before re-sizing needed
```

Recommend a comment or a startup metric logging the filter's estimated current fill ratio, so a future degradation shows up as a dashboard trend rather than a silent creep in DB load as more "maybe" hits fall through.

Not blocking on sizing since it degrades gracefully (worst case, more DB fallback calls, not incorrect behavior) — but the correctness bug is a straightforward miscorrect-use-of-Bloom-filter and should be fixed in this diff before it merges, since it changes the failure mode from "occasional wasted DB call" to "occasional silently dropped real event."
