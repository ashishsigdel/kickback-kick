# Script 54 — reviewing a new Postgres connection pool

[[thinking]]
Connection pool reviews live or die on two numbers most PRs don't justify: max size relative to what the database can hold across all callers, and what happens to a connection when a request panics or times out mid-query. A pool sized without reference to `max_connections`, or that can leak on error, won't show up in a normal test run — it shows up in production under load. Read the diff, then check both specifically rather than trusting the numbers as given.

[[response]]
`max_connections` is 100 on this instance. The diff sets up the pool like this:

```go
func NewPool(dsn string) (*pgxpool.Pool, error) {
    cfg, err := pgxpool.ParseConfig(dsn)
    if err != nil {
        return nil, err
    }
    cfg.MaxConns = 50
    cfg.MinConns = 10
    return pgxpool.NewWithConfig(context.Background(), cfg)
}
```

50 looks reasonable in isolation, but this service runs 4 replicas — `deploy/k8s/api-deployment.yaml:8` sets `replicas: 4`. That's 4 × 50 = 200 possible connections against a 100-connection ceiling, before counting the reporting service and migration runner on the same database. This diff alone doesn't cause an outage, but it makes one likely under simultaneous replica load — connections start failing with `too many clients already` rather than degrading gracefully.

[[thinking]]
The right ceiling isn't "50 sounds like a lot" — it's `max_connections` divided across every consumer of this database, with headroom for `superuser_reserved_connections` and anything else that connects (migrations, admin tools, the reporting replica). I should enumerate everyone who actually connects before proposing a number, rather than just cutting this one service's pool in half arbitrarily.

I'd also flag that `MinConns = 10` means each replica holds 10 idle connections open permanently regardless of load — 4 replicas × 10 idle = 40 connections consumed at all times even at zero traffic, which is worth being deliberate about rather than a default someone copied from an example config.

Added `MaxConnLifetime` and `MaxConnIdleTime`, which weren't in the original diff at all — without them, connections live forever once established, which matters if Postgres itself gets restarted or fails over: a pool that never cycles connections can hold stale ones after a failover and error on every query until something forces a reconnect.

[[thinking]]
Sizing is one axis; leak handling is the other, and it's the one that fails silently. I want to check every place a connection or transaction is acquired from the pool and confirm it's released on every path, including panics — a `defer conn.Release()` placed after error-handling code rather than immediately after acquisition is a common way this goes wrong, since an early return before the defer is registered leaks the connection permanently.

[[response]]
Checked every `pool.Acquire` / `pool.Begin` call site in the diff — 6 total. 5 of 6 follow the correct pattern (`defer release()` immediately after acquire, before any error handling). One doesn't:

`internal/db/reports.go:71`:

```go
func (r *ReportRepo) GenerateSummary(ctx context.Context, id string) (*Summary, error) {
    conn, err := r.pool.Acquire(ctx)
    if err != nil {
        return nil, err
    }
    summary, err := computeSummary(ctx, conn, id)
    if err != nil {
        return nil, err   // <- returns without releasing conn
    }
    conn.Release()
    return summary, nil
}
```

If `computeSummary` returns an error — which it will on any bad input or query failure — this function returns without ever calling `conn.Release()`. That connection is gone from the pool's available set until the process restarts; under a sustained error condition (a bad report ID hit repeatedly, or a downstream failure making every summary fail), this leaks the pool down to zero over time, and a pool at zero available connections looks identical to a database outage from the caller's side.

```diff
     conn, err := r.pool.Acquire(ctx)
     if err != nil {
         return nil, err
     }
+    defer conn.Release()
     summary, err := computeSummary(ctx, conn, id)
     if err != nil {
-        return nil, err
+        return nil, err
     }
-    conn.Release()
     return summary, nil
```

`defer` immediately after a successful acquire is the pattern that makes this safe regardless of how many return paths get added later — the other 5 call sites already follow it, which is presumably why this one stood out as the exception rather than the rule.

Ran `go test ./internal/db/... -run TestReportRepo -race` after the fix, plus a new leak-detection test that forces `computeSummary` to fail 200 times in a row and asserts `pool.Stat().AcquiredConns()` returns to 0 afterward: `PASS`, all 12 cases.

Sizing (15/replica) is a recommendation based on today's `max_connections` and today's other consumers — if the reporting service's connection count changes, this number needs revisiting too, and there's no automated check tying them together. Worth a comment in the deploy config linking the two, or an alert on total `pg_stat_activity` count crossing 85, rather than relying on this math staying accurate by memory.
