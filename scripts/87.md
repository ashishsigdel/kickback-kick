# Script 87 — date-range filtering breaks across timezone boundaries

[[thinking]]
"Off by one day near midnight" almost always means a date got parsed or compared in the wrong timezone somewhere along the pipeline — either the browser's local time leaking into a UTC-assumed boundary, or a `Date` truncated to midnight in one zone and compared against midnight in another. I want the actual failing input before guessing which.

Reproduced with the report the user filed: filtering "Jul 21" from a browser in `America/Los_Angeles` (UTC-7) returns rows from Jul 20 21:00 UTC through Jul 21 21:00 UTC — a day shifted 3 hours early.

`src/components/ReportFilters.tsx:38`:

```ts
const start = new Date(`${dateStr}T00:00:00`);
const end = new Date(`${dateStr}T23:59:59`);
```

`new Date("2026-07-21T00:00:00")` (no `Z`, no offset) is parsed in the **browser's local timezone** per the spec. So "midnight Jul 21" becomes 07:00 UTC in LA, not 00:00 UTC. That gets sent to the API as an ISO string and compared against `created_at`, which is stored in UTC — hence the shift.

`src/lib/reporting/dateRange.ts:14` then re-derives the end boundary from `start` by adding 24h, which compounds the same local-time assumption rather than fixing it.

[[thinking]]
The fix has to pick one timezone semantics and be consistent everywhere the range is used — client construction, the API contract, and the SQL comparison. Mixing "user's local calendar day" with "UTC day" is the actual bug, not just this one call site, so I want to check what the API expects before changing only the client.

[[response]]
API contract (`api/reports.openapi.yaml:112`) documents `start`/`end` as UTC ISO-8601, but the intent of the filter is "the user's local calendar day" — a report for "Jul 21" should mean Jul 21 in the viewer's timezone, converted correctly to a UTC range, not Jul 21 UTC.

Fix: construct the boundaries explicitly in the user's zone using `Intl` offset math instead of relying on the ambiguous no-offset string:

```ts
function localDayToUtcRange(dateStr: string, tz: string) {
  const start = fromZonedTime(`${dateStr}T00:00:00`, tz); // date-fns-tz
  const end = fromZonedTime(`${dateStr}T23:59:59.999`, tz);
  return { start: start.toISOString(), end: end.toISOString() };
}
```

`fromZonedTime` treats the input as wall-clock time *in `tz`* and converts to a real UTC instant, instead of `new Date()` silently using whatever zone the JS runtime happens to be in.

```diff
-const start = new Date(`${dateStr}T00:00:00`);
-const end = new Date(`${dateStr}T23:59:59`);
+const { start, end } = localDayToUtcRange(dateStr, Intl.DateTimeFormat().resolvedOptions().timeZone);
```

[[response]]
Added a parametrized test across offsets:

```ts
test.each(["UTC", "America/Los_Angeles", "Asia/Kathmandu", "Pacific/Kiritimati"])
  ("Jul 21 local day maps to correct UTC range in %s", (tz) => {
    const { start, end } = localDayToUtcRange("2026-07-21", tz);
    expect(isWithinInterval(new Date(start), { start: parseISO(start), end: parseISO(end) })).toBe(true);
    expect(new Date(end).getTime() - new Date(start).getTime()).toBeCloseTo(86_399_999, -2);
  });
```

Included `Asia/Kathmandu` (UTC+5:45) and `Pacific/Kiritimati` (UTC+14) deliberately — non-hour offsets and the extreme edges are what a UTC-only CI run would never exercise.
