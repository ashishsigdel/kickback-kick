# Script 24 — investigating why the CI pipeline has gotten slow

[[thinking]]
"CI got slow" over time in a monorepo usually means one of: the dependency graph between jobs got worse (less parallelism than before), individual jobs got slower (cache misses, more code to build/test), or the number of jobs grew without the runners scaling to match, so queueing time dominates over actual execution time. Those look identical from "the pipeline takes 40 minutes now instead of 12" alone — I need the run history broken into queue time vs. execution time before I can tell which one this is.

name: Bash

  command: gh run list --workflow=ci.yml --limit 50 --json databaseId,createdAt,updatedAt,status | jq -r '.[] | "\(.createdAt) \(.updatedAt)"' > /tmp/runs.txt && gh api repos/:owner/:repo/actions/runs/$(gh run list --workflow=ci.yml --limit 1 --json databaseId -q '.[0].databaseId')/jobs | jq '.jobs[] | {name, started_at, completed_at}'

[[response]]
Total pipeline wall time went from ~12 min (90 days ago) to ~41 min today, but the per-job breakdown on the latest run shows most individual jobs are barely slower than before — `lint` (2m → 2m10s), `build-web` (4m → 4m40s). The blowup is elsewhere:

```
name                started_at            completed_at          duration
test-unit-shard-1   09:12:03              09:19:44              7m41s
test-unit-shard-2   09:12:04              09:19:58              7m54s
test-integration     09:20:12              09:41:03              20m51s
e2e-smoke            09:41:15              09:47:02              5m47s
```

`test-integration` is 20m51s and, critically, its `started_at` (09:20:12) is right after `test-unit-shard-2` finishes (09:19:58) — it's not running in parallel with the unit test shards even though nothing in the job itself looks like it needs to. That's a scheduling/dependency problem, not a job-got-slower problem.

[[thinking]]
If `test-integration` doesn't actually need the unit test shards to finish first, this is a `needs:` graph issue — someone added a dependency edge that isn't a real data dependency, probably to guarantee ordering for an unrelated reason (maybe a shared artifact or a flaky-avoidance workaround), and it's now serializing 8 minutes of work that could overlap. I should check the workflow YAML directly rather than infer intent from timing alone.

[[response]]
Confirmed in `.github/workflows/ci.yml:34`:

```yaml
jobs:
  test-unit:
    strategy:
      matrix: { shard: [1, 2] }
    runs-on: ubuntu-latest
    steps: [...]

  test-integration:
    needs: [test-unit]
    runs-on: ubuntu-latest
    steps: [...]
```

`git log -p --follow .github/workflows/ci.yml` shows `needs: [test-unit]` was added 6 weeks ago in `4f8a2c1`, commit message: `"ci: don't waste runners on integration tests if unit tests are already failing"`. That's a real, reasonable intent — fail fast, don't burn 20 minutes of integration test time on a build that's already broken. But it was implemented as a hard sequential dependency, which serializes the *happy path* (the common case, where unit tests pass) to save time only on the *unhappy path* (unit tests failing, presumably a minority of runs).

Checked how often that tradeoff actually pays off: `gh run list --workflow=ci.yml --limit 200 --json conclusion,name | jq` — of runs where `test-unit` matters, unit tests fail on their own (independent of integration) in about 4% of runs over the last 200. So the fail-fast behavior is saving ~20 min on ~4% of runs and costing ~8 min (the non-overlapping unit test duration) on the other 96%.

[[thinking]]
The original intent is legitimate, so I don't want to just delete the edge — I want to check whether Actions has a way to keep the fail-fast property without paying the serialization cost on the common path. `needs:` in Actions always waits for full completion, not just failure, so there's no native "start now, abort if the dependency fails" primitive. Given that, the real decision is whether the 96% case's wall-clock win is worth giving up the 4% case's saved runner-minutes.

[[response]]
Recommending removing the `needs: [test-unit]` edge and letting `test-integration` run in parallel, trading a small amount of wasted runner-minutes for a large wall-clock win:

```diff
 test-integration:
-  needs: [test-unit]
   runs-on: ubuntu-latest
   steps: [...]
```

Tradeoff, explicit: on the ~4% of runs where unit tests fail, integration tests now run to completion anyway instead of being skipped — costing roughly 20 extra runner-minutes on those runs. In exchange, the other 96% of runs save ~8 minutes of wall-clock time each, since integration no longer waits for unit tests to finish before starting. Net runner-minute cost is roughly a wash (200 runs: old ≈ 200×8min serialization = 1600 min tied up waiting + 8 failing runs skipping ~20min = -160min saved there; new ≈ 0 serialization + 8 failing runs × 20min wasted = 160min spent); the actual win is wall-clock time for every contributor waiting on green CI, which was the original complaint.

Tested on a throwaway branch by pushing a commit with the edge removed: pipeline wall time 41min → 32min (test-integration now starts at 09:12:05, essentially parallel with the unit shards, and finishes at 09:33:10 — the new critical path).

Did not touch the actual per-job durations (`test-integration` itself is still 20m51s, which is a lot) — that's a separate investigation into whether that suite itself has grown without the runner count matching, but it's independent of the scheduling issue I found here. Want me to look at why `test-integration` alone takes 21 minutes next, or is unblocking the parallelism enough for now?
