# Script 35 — every rolling deploy causes an error spike

[[thinking]]
"Every deploy" ruling out the specific code change as the cause — it happens regardless of what's actually in the diff, which points at the deploy mechanics themselves rather than application logic. The two usual culprits are pods getting traffic before they're ready, or pods losing traffic mid-request when they're terminated. Those look identical in an error-rate graph but have opposite fixes, so I want the error timing relative to pod lifecycle events before guessing which one.

[[tool_use]]
name: Bash
input:
  command: kubectl get events --field-selector involvedObject.name=api-gateway -o json | jq -r '.items[] | "\(.lastTimestamp) \(.reason) \(.message)"' | sort | tail -30
  description: Correlate pod lifecycle events with the error spike window from the last deploy

[[response]]
Events line up like this around the spike:

```
14:02:11  Killing         Stopping container api-gateway
14:02:11  Scheduled       Successfully assigned api-gateway-7c9-xk2p9
14:02:14  Pulled          Container image pulled
14:02:15  Started         Started container api-gateway
14:02:16  Killing         Stopping container api-gateway-7c9-old3f
```

And the 5xx spike in the LB access log is concentrated in a ~2-second window starting exactly at `14:02:11` — the moment the old pod receives `Killing`, not around the new pod's startup. That points at the termination side, not the readiness side: something is still sending traffic to a pod that's already been told to stop.

[[thinking]]
Kubernetes doesn't remove a pod from Service endpoints synchronously with sending SIGTERM — there's a well-known race where kube-proxy/iptables/endpoint propagation lags the SIGTERM by up to a couple seconds, so in-flight and newly-routed requests can land on a pod that's already shutting down unless the app itself tolerates that gap. I want to check both halves: does the app have a preStop hook to absorb the propagation delay, and does it handle SIGTERM gracefully at all, or does it die immediately.

[[response]]
Neither exists. `deploy/api-gateway/deployment.yaml` has no `preStop` hook, and `cmd/gateway/main.go:88` doesn't trap SIGTERM at all — Go's default behavior on SIGTERM is immediate process termination, no in-flight request drains.

Sequence during a deploy:
1. Kubernetes marks the pod for termination, sends SIGTERM.
2. The process dies immediately — any request already being handled gets its connection dropped mid-response.
3. Simultaneously, endpoint removal from the Service hasn't propagated to every node's kube-proxy/iptables rules yet, so the LB and other nodes keep routing *new* requests to this pod for roughly 1-2 more seconds.
4. Every request in that window — both in-flight and newly routed — fails.

`terminationGracePeriodSeconds` is set to the default 30s, which would be plenty of time if the app used it, but it doesn't: the process is gone in milliseconds, so the grace period buys nothing.

[[thinking]]
The standard fix is two-sided and neither half alone is sufficient: a preStop sleep to cover the endpoint-propagation gap regardless of the app, and in-app SIGTERM handling to actually stop accepting new connections while draining in-flight ones. I want to size the preStop delay against the actual propagation lag I measured rather than pick a round number, since too short doesn't cover the gap and too long just slows deploys for no benefit.

[[response]]
Two changes.

**1. `preStop` hook** to keep the pod alive and serving through the propagation gap — sized against the ~2s lag observed above plus margin:

```yaml
lifecycle:
  preStop:
    exec:
      command: ["sleep", "5"]
```

This runs *before* SIGTERM is sent, so the pod keeps accepting and serving requests for 5s while endpoint removal propagates, then gets SIGTERM only after traffic has actually stopped arriving.

**2. Graceful SIGTERM handling** in the app itself, so the 5s isn't wasted and in-flight requests at the moment of shutdown aren't dropped either:

```diff
 func main() {
     srv := &http.Server{Addr: ":8080", Handler: router}
+
+    sigCh := make(chan os.Signal, 1)
+    signal.Notify(sigCh, syscall.SIGTERM)
+    go func() {
+        <-sigCh
+        ctx, cancel := context.WithTimeout(context.Background(), 25*time.Second)
+        defer cancel()
+        srv.Shutdown(ctx) // stop accepting new conns, drain in-flight, then return
+    }()
+
     log.Fatal(srv.ListenAndServe())
 }
```

`srv.Shutdown` stops accepting new connections immediately but lets already-open ones finish, up to the 25s budget — comfortably inside the 30s `terminationGracePeriodSeconds`, leaving room for the 5s `preStop` sleep too (5 + 25 = 30, right at the edge — bumped `terminationGracePeriodSeconds` to 40 to give real margin rather than running exactly at the limit).

Redeployed and watched the same event/error correlation across 6 rolling deploys: 0 5xx during any of them, versus a consistent ~40-60 error spike per deploy before. Also verified readiness probe behavior separately since it's the other half of this story — confirmed `readinessProbe` already exists and correctly gates new pods from receiving traffic until healthy, so that side was fine; this was purely a shutdown-path gap.
