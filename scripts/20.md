[[thinking]]
The clean split is Pricing, Inventory, Notification, Audit as their own classes with `OrderManager` reduced to orchestration. But the three cross-cutting methods are the risk — if I split naively, `placeOrder` ends up calling four new services and coordinating a multi-step operation that used to be enforced by shared local state and a single method body. The real question before I write any code is whether `placeOrder` currently has any implicit ordering or rollback behavior on failure that depends on being one method — because that's exactly the kind of thing that breaks silently across a split.

[[response]]
Checked `placeOrder`, `OrderManager.java:412`, for implicit sequencing:

```java
public Order placeOrder(OrderRequest req) {
    inventoryClient.reserveStock(req.items());
    BigDecimal total = calculateTotal(req);          // pricing
    Order order = orderRepository.save(new Order(req, total));
    try {
        sendConfirmation(order);                       // notification
    } catch (NotificationException e) {
        logger.warn("confirmation failed, continuing", e);
    }
    auditLog.logOrderEvent(order.id(), "PLACED");       // audit
    return order;
}
```

Two things matter for the split: stock is reserved *before* pricing runs, and notification failures are swallowed while audit failures are not (no try/catch around `logOrderEvent` — an audit write failure currently propagates and fails the whole order). Both are deliberate-looking, not accidental, so the extracted services need an orchestrator that preserves this exact ordering and exact failure semantics — a naive split where each service is called independently, in whatever order looks nice, would silently change behavior (e.g., reserving stock after computing price would change nothing today but changes behavior under a future pricing rule that depends on availability).

Proposed structure:

```java
public class OrderPlacementOrchestrator {
    public Order placeOrder(OrderRequest req) {
        inventoryService.reserveStock(req.items());
        BigDecimal total = pricingService.calculateTotal(req);
        Order order = orderRepository.save(new Order(req, total));
        notificationService.sendConfirmationBestEffort(order); // name makes swallow-on-failure explicit
        auditService.logOrderEvent(order.id(), "PLACED");      // still propagates
        return order;
    }
}
```

Renaming `sendConfirmation` to `sendConfirmationBestEffort` in the new `NotificationService` so the failure-swallowing behavior is visible at the call site instead of hidden inside a try/catch someone has to go read.

[[thinking]]
This is a large, mechanical-but-risky change touching four new classes and one orchestrator, on a class that's presumably load-bearing for the whole billing flow. I should not do this as one big-bang PR — the standard way to de-risk a God object split is one extraction at a time, each independently shippable and tested, so a regression is traceable to one commit instead of a 1,800-line diff.

[[response]]
Recommending a four-step extraction, each its own PR, in this order:

1. **Audit first** — smallest, no cross-cluster calls except from `placeOrder`/`cancelOrder`, lowest risk. Good to validate the extraction pattern before touching anything with real business logic.
2. **Notification second** — same shape, and the `sendConfirmationBestEffort` rename happens here where it's easy to review in isolation.
3. **Inventory third** — now the risk shows up: `reserveStock`/`releaseReservation` need to move together, and existing tests in `OrderManagerTest.java:230-310` mock `inventoryClient` directly, so those tests move to `InventoryServiceTest` rather than being deleted.
4. **Pricing last** — most call sites (used by reporting and the admin console too, not just order placement), so it benefits from the other three already being proven out.

`OrderPlacementOrchestrator` gets assembled incrementally as each piece lands, rather than written once up front — that way `placeOrder`'s behavior is verified against the original at every step via the existing integration test (`OrderFlowIntegrationTest`, currently 19 passing, covers stock-reservation-then-pricing ordering and the notification-swallow behavior) instead of one big-bang cutover where a regression could hide anywhere in 1,800 lines.
