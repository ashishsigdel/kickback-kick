# Script 63 — reviewing a diff adding a circuit breaker around a flaky call

[[thinking]]
A circuit breaker is only as good as its trip and reset thresholds, and those are the part reviewers usually skim past to check "yes, it's wrapped in a CircuitBreaker." I want to read the config numbers first — failure rate threshold, wait duration, sliding window — because a breaker with the defaults left in place is often tuned for a different traffic profile than the endpoint it's protecting.

[[tool_use]]
name: Bash
input:
  command: git diff main -- src/main/java/com/acme/pricing/ExchangeRateClient.java src/main/resources/application.yml | rg -n "resilience4j|circuitbreaker|CircuitBreaker" -i
  description: Review the circuit breaker config and its application to the flaky call

[[response]]
The diff wraps `ExchangeRateClient.fetchRate()`, which calls a third-party FX rate API that's been timing out intermittently. Config, `application.yml:44`:

```yaml
resilience4j.circuitbreaker:
  instances:
    exchangeRate:
      failureRateThreshold: 50
      slidingWindowSize: 10
      waitDurationInOpenState: 5s
      permittedNumberOfCallsInHalfOpenState: 3
```

Two things stand out.

**1. `slidingWindowSize: 10` is too small for this call's actual volume.** `fetchRate()` is called from `PricingService.quote()`, invoked roughly 2x/second per `PricingService.java:29`'s usage in the checkout path. A window of 10 calls is ~5 seconds of traffic — the breaker's failure rate is being computed over a tiny, noisy sample. A single bad 3-call streak (30% of the window) doesn't trip it, but it also means the breaker's state can flap based on very few data points rather than a stable signal.

**2. No fallback registered.** `ExchangeRateClient.java:38`:

```java
@CircuitBreaker(name = "exchangeRate")
public BigDecimal fetchRate(String pair) {
    return restClient.get(...);
}
```

No `fallbackMethod`. When the breaker opens, calls to `fetchRate()` throw `CallNotPermittedException` instead of the original exception — and I need to check whether the caller handles that, because it's a new exception type this diff introduces into the call path.

[[thinking]]
This is the part that actually determines whether the diff is safe to merge — if `PricingService.quote()` has a catch block for the FX client's original timeout exception but not for `CallNotPermittedException`, then adding the breaker changes an already-handled failure into an unhandled one, which is a regression dressed up as a resilience improvement.

[[response]]
Checked `PricingService.java:52`:

```java
try {
    rate = exchangeRateClient.fetchRate(pair);
} catch (RestClientException e) {
    rate = lastKnownRate.get(pair);
}
```

Confirmed — this only catches `RestClientException`. `CallNotPermittedException` is a `resilience4j` exception, not a `RestClientException`, so it doesn't match and propagates up uncaught. Before this diff, a flaky FX API meant `quote()` degraded to `lastKnownRate`. After this diff, once the breaker opens, `quote()` throws a 500 instead — the exact scenario a circuit breaker is supposed to prevent, inverted.

This is a real regression, not a style nit: the failure mode gets worse specifically during the outage the breaker exists to handle.

[[thinking]]
The fix should keep the existing degrade-to-last-known-rate behavior as the single source of truth for "what happens when FX is unavailable," rather than duplicating that logic into a Resilience4j fallback method and a catch block that could drift apart later.

[[response]]
Two changes, both in the fallback so there's one place that owns "FX is unavailable" behavior:

```diff
 @CircuitBreaker(name = "exchangeRate", fallbackMethod = "fetchRateFallback")
 public BigDecimal fetchRate(String pair) {
     return restClient.get(...);
 }
+
+private BigDecimal fetchRateFallback(String pair, Throwable t) {
+    log.warn("exchangeRate breaker fallback for {}: {}", pair, t.toString());
+    return lastKnownRate.get(pair);
+}
```

```diff
 exchangeRate:
   failureRateThreshold: 50
-  slidingWindowSize: 10
+  slidingWindowSize: 50
+  minimumNumberOfCalls: 20
   waitDurationInOpenState: 5s
   permittedNumberOfCallsInHalfOpenState: 3
```

The Resilience4j fallback method catches both the original exception and `CallNotPermittedException` automatically — any `Throwable` matching the method signature — so I removed the now-redundant `RestClientException` catch in `PricingService.java:52` and let the breaker own the whole degrade path. `slidingWindowSize: 50` with `minimumNumberOfCalls: 20` means the breaker won't make a trip decision until it has a real sample, at this call's actual ~2/s rate that's a 10-second minimum window instead of 5.

`mvn test -Dtest=ExchangeRateClientTest,PricingServiceTest` → `Tests run: 21, Failures: 0`. Added `breakerOpen_fallsBackToLastKnownRate` which forces the breaker open via repeated simulated timeouts and asserts `quote()` still returns a price instead of throwing.
