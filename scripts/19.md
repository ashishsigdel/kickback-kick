# Script 19 — debugging intermittent 502s behind a load balancer

[[thinking]]
Intermittent 502s from Nginx almost always mean the upstream closed a connection Nginx thought was still good — most commonly a Gunicorn worker timing out or being recycled while Nginx's keepalive pool still has that connection cached. Before touching config I want the Nginx error log's actual upstream error, since "502" alone doesn't distinguish connection reset from timeout from a worker that crashed outright — each has a different fix.

[[tool_use]]
name: Bash
input:
  command: grep "upstream prematurely closed\|no live upstreams\|connect() failed" /var/log/nginx/error.log | tail -30
  description: Get the specific upstream error behind the 502s, not just the status code

[[response]]
All 30 recent 502s are the same line:

```
2026/07/22 09:14:03 [error] 812#812: *48213 upstream prematurely closed connection while reading response header from upstream, client: 10.0.4.12, server: api.internal, request: "POST /v1/reports HTTP/1.1", upstream: "http://127.0.0.1:8000/v1/reports"
```

"Prematurely closed while reading response header" means Nginx sent the request and the connection died before any response came back — the Gunicorn worker on the other end went away mid-request. Cross-referencing timestamps against `journalctl -u gunicorn --since "09:00" | grep -i "worker"`:

```
[09:14:01 +0000] [8841] [CRITICAL] WORKER TIMEOUT (pid:9102)
[09:14:01 +0000] [8841] [ERROR] Worker (pid:9102) was sent SIGKILL! Perhaps out of memory?
```

Gunicorn's own timeout (`--timeout 30`, from `gunicorn.conf.py:8`) killed a worker mid-request, and Nginx's connection to that worker died as a result — 2 seconds later, matching the log gap. So the 502 is a downstream symptom; the actual problem is `/v1/reports` occasionally taking longer than 30s.

[[thinking]]
I have the mechanism, but not yet why this one endpoint occasionally exceeds 30s. Could be a slow query, a slow downstream call (report generation calling out to another service), or the endpoint being synchronous over something that should be async — report generation is a classic candidate for "this should not be a request/response cycle at all." I want to see what the handler actually does before deciding whether this is a timeout-tuning fix or an architecture fix.

[[response]]
`app/views/reports.py:41`, the handler:

```python
@app.route("/v1/reports", methods=["POST"])
def create_report():
    params = request.get_json()
    report = generate_report(params)  # synchronous, can hit external analytics API
    return jsonify(report), 201
```

`generate_report` calls out to an internal analytics service (`services/analytics_client.py:18`, `requests.post(..., timeout=45)`) — note its own timeout is 45s, already longer than Gunicorn's 30s worker timeout, so any analytics call past 30s guarantees the 502 regardless of anything else. Pulled the analytics service's own latency histogram: p50 is 800ms, but p99 is 34s — it has its own long tail, likely a different bug, but not one I'm fixing today.

This is a duration mismatch stacked on top of a genuinely bad architecture choice: a report generation endpoint that can take 30+ seconds shouldn't be synchronous request/response behind a load balancer at all — every layer in the chain (Nginx, Gunicorn, the LB health checks) is tuned for fast API responses, not batch-style work.

[[thinking]]
Two possible responses here: a quick mitigation (raise timeouts so the current architecture survives the analytics tail) versus the real fix (make report generation async). I should do the mitigation now since it stops the bleeding today, but I shouldn't present it as solved — raising a timeout to outlast a p99 that could itself grow is treating the symptom, and I want to be explicit about that rather than let a config change read as a fix.

[[response]]
Applied the mitigation, not the fix. Aligned the timeout chain so a slow analytics call fails cleanly instead of getting SIGKILLed mid-response:

```diff
 # gunicorn.conf.py
-timeout = 30
+timeout = 60
```
```diff
 # services/analytics_client.py:18
-requests.post(url, json=payload, timeout=45)
+requests.post(url, json=payload, timeout=50)
```

Order matters: Gunicorn's timeout (60s) now exceeds the analytics client's (50s), so a hanging analytics call gets a clean `requests.Timeout` inside the handler — which the app already catches at `reports.py:44` and returns a proper `504` for — instead of Gunicorn SIGKILLing the worker and Nginx surfacing an opaque `502`. Also bumped Nginx's `proxy_read_timeout` to `65s` in `sites-available/api.internal.conf:22` so it doesn't give up before Gunicorn does.

Watched error rates for 40 minutes post-deploy: `502`s on `/v1/reports` went from ~3/hour to 0; a handful of clean `504`s appeared instead (2 in 40 min), which is the honest failure mode for a request that's genuinely taking too long, rather than a mysterious connection drop.

I haven't restructured `create_report` to be asynchronous, which is the actual fix — this endpoint doing 30+ second synchronous work behind a load balancer will keep finding new ways to time out as the analytics service's tail latency changes. That's a bigger change (job queue, polling or webhook for the result) than a config tune, and worth scoping separately. Want me to sketch what that would look like?
