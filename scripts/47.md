# Script 47 — thundering herd on cache key expiry

[[thinking]]
A spike right when a popular key expires, rather than a steady elevated load, points specifically at cache stampede — many requests all missing at once and all going to the database to recompute the same value. The tell is concurrent identical queries hitting the DB in a tight window, not just high DB load generally. I want to see the actual query timing clustered around the expiry to confirm before assuming it's this rather than, say, a scheduled job coinciding.

`CacheService.java:71`:

```java
public List<Product> getFeaturedProducts() {
    String cached = redis.get("homepage:featured");
    if (cached != null) {
        return deserialize(cached);
    }
    List<Product> fresh = productRepository.loadFeatured(); // ~180ms
    redis.setex("homepage:featured", 300, serialize(fresh));

```

TTL is a flat 300s with no jitter and no lock around the recompute. Every one of the ~800 requests/sec this endpoint sees checks `redis.get`, all get `null` in the same instant the key expires, and all 800 independently call `loadFeatured()` — 800 concurrent 180ms queries is a very different load than 1 query plus 799 cache hits.

[[thinking]]
Two standard mitigations, and they solve slightly different parts of this: jittered TTL spreads out *when* different keys expire (helps if there were many keys, doesn't help much for one very hot key), and a recompute lock (or "only one request rebuilds, others wait or serve stale") stops the herd on any single key regardless of TTL. Since this is one specific hot key, the lock is the one that actually matters here — jitter alone wouldn't have prevented this incident.

[[response]]
Recommending a double-checked lock with stale-while-revalidate rather than jitter alone, since jitter doesn't help a single very hot key — it only spreads load when many keys expire near each other.

```java
public List<Product> getFeaturedProducts() {
    String cached = redis.get("homepage:featured");
    if (cached != null) {
        return deserialize(cached);
    }

    String lockKey = "lock:homepage:featured";
    boolean gotLock = redis.set(lockKey, "1", SetParams.setParams().nx().ex(10));

    if (!gotLock) {
        // someone else is rebuilding; serve stale if we have it, else wait briefly
        String stale = redis.get("homepage:featured:stale");
        if (stale != null) return deserialize(stale);
        Thread.sleep(50);
        return getFeaturedProducts(); // one retry, cache is likely warm by now
    }

    try {
        List<Product> fresh = productRepository.loadFeatured();
        String serialized = serialize(fresh);
        redis.setex("homepage:featured", 300, serialized);
        redis.setex("homepage:featured:stale", 900, serialized); // longer-lived fallback
        return fresh;
    } finally {
        redis.del(lockKey);
    }
}
```

`SET NX EX` gives an atomic acquire with a 10s auto-expiring lock, so a crashed request can't hold it forever. Only one of the 800 concurrent misses wins the lock and hits Postgres; the other 799 either get the stale copy (present for 900s, well past the 300s primary TTL) or do one short sleep-and-retry.
