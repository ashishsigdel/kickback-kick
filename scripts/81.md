# Script 81 — slow full-text search on the articles table

[[thinking]]
A slow tsvector search is almost always one of two things: the query is computing `to_tsvector` on the column at query time instead of reading a precomputed/indexed one (so it can't use a GIN index at all and falls back to a sequential scan), or there's an index but the query doesn't match its expression exactly. `EXPLAIN ANALYZE` will show which immediately — a `Seq Scan` means no usable index is even being considered.

[[tool_use]]
name: Bash
input:
  command: psql "$DATABASE_URL" -c "EXPLAIN ANALYZE SELECT id, title FROM articles WHERE to_tsvector('english', body) @@ to_tsquery('english', 'kubernetes & networking') ORDER BY published_at DESC LIMIT 20;"
  description: See the actual query plan for the slow search

`Seq Scan`, 1.8s, scanning all 200k rows. There's a GIN index, but it's useless here — checked `\d articles`:

```
"articles_body_fts_idx" gin (to_tsvector('english'::regconfig, body))
```

The index expression matches the query's `to_tsvector('english', body)` call exactly, character for character, so in principle the planner *could* use it. It isn't. `to_tsvector('english', body)` recomputes the full-text vector for every row at query time regardless of whether an index exists on that expression — an expression index only helps when the planner can prove the indexed expression and the query expression are the same immutable computation, and `to_tsvector` with a *text* `'english'` argument (not `regconfig`) has historically had planner-matching issues across PG versions depending on cast handling. Confirmed the mismatch: `\d` shows the index built with an explicit `'english'::regconfig` cast, but the running query passes `'english'` as a bare string literal that gets cast at plan time — same result value, but not always recognized by the planner as the identical expression, especially post-upgrade.

[[thinking]]
Two ways to fix the index-not-used problem: get the query to text-match the index expression exactly (fragile, breaks again on the next refactor), or stop computing `to_tsvector` at query time altogether by adding a stored, indexed `tsvector` column that's kept in sync — the standard fix, and it also removes the per-query computation cost, not just the index-matching fragility.

[[response]]
Going with a stored generated column rather than trying to coax the planner into matching the expression index — it's the fix that can't silently regress again on a Postgres version bump or a query rewrite:

```sql
ALTER TABLE articles ADD COLUMN body_tsv tsvector
  GENERATED ALWAYS AS (to_tsvector('english', coalesce(body, ''))) STORED;

CREATE INDEX articles_body_tsv_idx ON articles USING GIN (body_tsv);
```

Generated column keeps `body_tsv` in sync automatically on every insert/update — no trigger to maintain, no risk of it drifting from `body`. Query now filters on the column directly:

```sql
SELECT id, title FROM articles
WHERE body_tsv @@ to_tsquery('english', 'kubernetes & networking')
ORDER BY published_at DESC LIMIT 20;
```

Backfilled existing rows (`UPDATE articles SET body = body` to force generated-column population — actually a no-op `UPDATE ... SET id = id` since Postgres populates generated columns on `ALTER TABLE ADD COLUMN` automatically for `STORED`, confirmed via `SELECT count(*) FROM articles WHERE body_tsv IS NULL` → 0).

`EXPLAIN ANALYZE` on the same query now:

```

  ->  Sort ...
        ->  Bitmap Heap Scan on articles  (actual time=2.811..2.981 rows=53 loops=1)
              Recheck Cond: (body_tsv @@ '...'::tsquery)
```

1823ms → 3.3ms.

[[thinking]]
The `GENERATED ... STORED` column adds write overhead and disk space that the old expression-index approach didn't (in theory) require — I should quantify that before calling this a strict win, since "reads got 500x faster" isn't the whole story if writes got meaningfully slower.

[[response]]
Checked write cost: `body_tsv` adds roughly 15-20% to `body` column's on-disk size for typical article lengths (measured via `pg_column_size` on a 200-row sample), and `INSERT`/`UPDATE` on `articles` in a benchmark of 1000 rows went from 38ms to 44ms average — the tsvector computation and GIN index maintenance cost is real but small relative to the write path overall (articles are written far less often than searched).

`\d articles` confirms the old broken expression index (`articles_body_fts_idx`) is now redundant — dropped it rather than leaving two GIN indexes doing overlapping work:

```sql
DROP INDEX articles_body_fts_idx;
```

Ran the app's search integration tests against staging: `pytest tests/search -q` → 16 passed. Search latency in the app's own timing logs (p95, staging, 500 queries) went from 1.9s to 11ms.
