# Script 50 — a queue consumer that stops processing without crashing

[[thinking]]
A consumer that's alive but idle is a different bug class than one that died — process up, no exception logged, but messages stop being acked. That means either the consumer is blocked on something that never returns (a lock, an unbounded call), or the broker itself stopped delivering. Check the broker's view of the queue before the consumer's code, since that tells me which side owns the problem.

48,213 messages ready, only 1 unacked, 2 consumers attached. If the process had crashed mid-message, unacked would sit at `prefetch_count` per dead consumer, not 1 — one unacked message with two live consumers means one worker is stuck holding a single message and never acking or nacking it.

`rabbitmqctl list_consumers` confirms: one consumer's `ack_timestamp` is 6 hours old, the other's is seconds old. Not "the consumer died" — "one worker thread grabbed a message and never returned."

[[thinking]]
With a specific message pinned as the culprit, the fastest way to find the bug is to find that exact message in application logs by its correlation ID or delivery tag around the time it was picked up, and see what the consumer does with it that could block forever — most likely candidates are a downstream HTTP call with no timeout, or a lock/mutex acquired without a timeout.

Nothing after that, ever, on this worker thread. `consumers/order_events.py:88`:

```python
def handle(self, message):
    event = json.loads(message.body)
    logger.info("processing order_event", id=event["id"], type=event["type"])
    if event["type"] == "inventory_reserve":
        with self.inventory_lock:  # threading.Lock(), no timeout
            reserve_inventory(event)
    channel.basic_ack(delivery_tag=message.delivery_tag)
```

`self.inventory_lock` is a plain `threading.Lock()` shared across all consumer threads, acquired with no timeout (`with lock:` blocks indefinitely). If `reserve_inventory` hangs, every thread needing `inventory_reserve` blocks forever on `acquire()`, and since `basic_ack` is after the `with` block, the message never gets acked, nacked, or requeued. RabbitMQ has no idea anything is wrong; from the broker's view the consumer is alive and simply slow.

Two bugs stacked: the missing HTTP timeout is the root cause of the hang, and the untimed lock is what turns one hung request into total consumer starvation for this event type.

[[thinking]]
Fixing both, in order of what actually stops the immediate bleeding: the HTTP timeout prevents this specific hang from recurring, and switching to a timed lock acquisition (or per-item locking instead of one shared lock) prevents any future hang — from any cause — from taking out the whole consumer pool the same way.

A nack-and-requeue on lock timeout means a genuinely stuck message goes back to the queue and can be retried or dead-lettered by policy, rather than silently parking one worker forever.
