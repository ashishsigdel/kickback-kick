# Script 64 — duplicate events being processed twice

[[thinking]]
Duplicate processing in a Kafka consumer almost always traces to the commit/process ordering — either the offset commits before processing finishes (rare, usually safe-ish direction) or processing happens and then a rebalance/crash occurs before the commit lands, causing reprocessing on restart. I want the consumer's commit strategy and rebalance behavior before looking at the handler logic itself, since that's where this class of bug actually lives.

[[response]]
`internal/consumer/orders.go:34`:

```go
config := sarama.NewConfig()
config.Consumer.Offsets.AutoCommit.Enable = true
config.Consumer.Offsets.AutoCommit.Interval = 5 * time.Second
```

Auto-commit, every 5 seconds, independent of whether the handler actually finished processing the message. That's the mechanism: `sarama`'s auto-commit marks the *last consumed* offset on a timer, not the last *successfully processed* one. If a message is consumed, handed to the handler, and the handler takes longer than 5s (or the process crashes/rebalances) before it finishes, the offset can commit while processing is still in flight — or the reverse: processing finishes but the timer hasn't fired yet and a rebalance moves the partition to another consumer that starts from the last committed offset, reprocessing everything since.

Checked the incident window: `internal/consumer/orders.go:52` shows a rebalance log line at 14:32:07, and the duplicate `OrderPaidEvent` processing (evidenced by two `payment_captured` rows for the same `order_id` in the audit log) is for events consumed between 14:32:02 and 14:32:07 — right before the rebalance, in a window where auto-commit hadn't fired yet.

[[thinking]]
Eager rebalancing revokes all partitions from all consumers before reassigning, which maximizes the reprocessing window during a rebalance — cooperative-sticky rebalancing only revokes the specific partitions actually moving. If this group is still on eager assignment, that's a second, independent lever that shrinks the duplicate window, worth checking alongside the commit fix.

The standard fix is manual commit after successful processing, but that alone doesn't make the handler idempotent against the case where a duplicate does get through anyway — at-least-once delivery with a rebalance mid-batch can still redeliver even with correct commit ordering, since the commit and the rebalance aren't atomic with each other. I should check whether the handler is idempotent before treating manual commit as sufficient.

Checked the handler: `internal/consumer/orders.go:71`:

```go
func (h *OrderHandler) HandleOrderPaid(event OrderPaidEvent) error {
    return h.db.Exec(
        "UPDATE orders SET status = 'paid', paid_at = $1 WHERE id = $2",

    )
}
```

This handler is idempotent by accident — re-running the same `UPDATE` is a no-op the second time. But `internal/consumer/notifications.go:28`, sending the confirmation email off the same event, is not:

```go
func (h *NotificationHandler) HandleOrderPaid(event OrderPaidEvent) error {
    return h.mailer.Send(event.CustomerEmail, "payment_confirmed", event)
}
```

That's the actual symptom being reported — duplicate confirmation emails, not duplicate DB writes. Manual commit reduces the *frequency* of redelivery, but Kafka is at-least-once by design; a crash between "handler ran" and "commit sent" can still redeliver no matter how tight the window. The handler has to be idempotent regardless.

[[thinking]]
Fixing commit strategy and fixing idempotency are both needed but serve different purposes — commit strategy shrinks the duplicate window, idempotency makes duplicates harmless. I should do both rather than picking one, and the idempotency key needs to be something already on the event so this doesn't require a schema migration on the producer side.
