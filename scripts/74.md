# Script 74 — reviewing the soft-delete rollout for missed query paths

[[thinking]]
Soft delete is a good pattern with one systemic risk: every place that queries the affected model directly, via associations, via raw SQL, or via a background job has to respect the new `deleted_at` scope, and it's very easy to add the default scope while missing the raw SQL and the join-table associations. I want a full inventory of touchpoints before I trust the diff, not just a read of the migration.

  command: git diff main --stat && rg -n "Project\.\w|has_many :projects|joins\(:projects\)" -g "*.rb" app/ lib/
  description: See the diff's footprint and every place Project is queried outside the model itself

[[response]]
The diff (`app/models/project.rb`) adds the standard pieces:

```ruby
default_scope { where(deleted_at: nil) }
scope :with_deleted, -> { unscope(where: :deleted_at) }

def soft_delete!
  update!(deleted_at: Time.current)
end
```

That covers `Project.all`, `Project.find`, and association calls like `account.projects` — `default_scope` applies there too since it's the same underlying relation. Good so far.

Three places don't go through the default scope and are untouched by this diff:

1. `app/models/account.rb:34` — `has_many :projects, -> { unscope(where: :deleted_at) }` on a *different* association, `all_projects`, added two years ago for an admin report. That one's supposed to see deleted rows, so it's fine as-is, but worth confirming it's meant to.
2. `lib/reports/monthly_usage.rb:19` — raw SQL: `ActiveRecord::Base.connection.execute("SELECT count(*) FROM projects WHERE account_id = #{id}")`. Raw SQL bypasses `default_scope` entirely — this will keep counting soft-deleted projects.
3. `app/jobs/project_archiver_job.rb:11` — `Project.unscoped.where(...)`. Already explicitly unscoped, predates this diff, and its whole job is to sweep old projects — also fine, but it's now ambiguous whether "old" should include already-soft-deleted ones or not.

[[thinking]]
#1 and #3 look intentional but I can't be sure without checking what they're for — an admin report and an archiver job both plausibly want to see deleted rows, or both could be latent bugs that just never mattered until soft delete existed to matter about. #2 is unambiguous: a usage count report has no reason to include deleted projects, and I can fix that one without needing to ask anyone.

[[response]]
Fixed #2 — it's not a judgment call, a usage report counting deleted projects is just wrong:

```diff
-ActiveRecord::Base.connection.execute("SELECT count(*) FROM projects WHERE account_id = #{id}")
+ActiveRecord::Base.connection.execute("SELECT count(*) FROM projects WHERE account_id = #{id} AND deleted_at IS NULL")
```

Also flagging a related gap while I was in there: `lib/reports/monthly_usage.rb:19` builds SQL via string interpolation on `id`. `id` comes from an internal loop over `Account.pluck(:id)` today so it's not attacker-controlled right now, but it's one careless call site change away from being one. Separate from the soft-delete review — noting it, not fixing it here since it's out of scope for this diff.

[[thinking]]
Before wrapping up I should check one more thing the diff doesn't mention at all: uniqueness validations. If `Project` has a `validates_uniqueness_of` or a DB unique index on something like `slug` scoped to `account_id`, soft-deleted rows still occupy that uniqueness slot unless the index itself also excludes `deleted_at IS NULL` rows — otherwise a user can never reuse the name/slug of something they just "deleted."

[[response]]
Checked — and this one's a real bug the diff introduces, not just a gap. `db/schema.rb:412`:

```ruby
add_index :projects, [:account_id, :slug], unique: true
```

Plain unique index, no partial-index condition. Once a project is soft-deleted, its `slug` is still consuming that uniqueness constraint — creating a new project with the same slug in the same account now fails with a DB-level uniqueness violation, even though the deleted one is invisible everywhere else via `default_scope`. From a user's perspective: delete a project called "staging", try to create a new one called "staging", get a 500.

Fix is a partial unique index rather than a Rails-level validation, since the constraint needs to hold at the DB level regardless of which scope touches the table:

```ruby
# migration
remove_index :projects, [:account_id, :slug]
add_index :projects, [:account_id, :slug], unique: true, where: "deleted_at IS NULL"
```
