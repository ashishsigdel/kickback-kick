# Script 99 — retry logic double-charges customers on transient errors

[[thinking]]
Double charges from retry logic almost always come from retrying an operation that isn't idempotent at the point of failure — specifically, retrying *after* the charge succeeded but before the caller received confirmation of that success, so the client can't tell "definitely failed, safe to retry" from "maybe succeeded, unsafe to retry" and retries anyway. I want to see exactly what triggers the retry and what it's retrying before assuming where the gap is.

[[response]]
`billing/payment_processor.py:44`:

```python
def charge_customer(customer_id, amount_cents, idempotency_key=None):
    for attempt in range(3):
        try:
            return gateway.charge(customer_id, amount_cents)
        except (requests.Timeout, requests.ConnectionError):
            continue
    raise ChargeFailed(customer_id)
```

`idempotency_key` is a parameter — accepted, never used. `gateway.charge()` is called without one. On a `Timeout`, the request may have already been received and processed by the payment gateway; the client just didn't get the response back in time. The retry loop then calls `gateway.charge()` again with the same amount and no idempotency key, which the gateway has no way to recognize as "the same attempt" — it processes it as a brand-new charge. Three attempts means up to three real charges for a single logical checkout if the gateway is slow rather than actually down.

Confirmed against the gateway's own semantics (`docs/gateway-api.md` in this repo, vendored): the gateway explicitly supports an `Idempotency-Key` header and documents this exact scenario — "on timeout, retry the request with the same idempotency key rather than assuming failure." The client-side plumbing for it was never wired up.

[[thinking]]
The fix needs a key that's stable across retries of the *same logical charge* but unique per actual checkout attempt — reusing a key across genuinely different charges would cause the opposite bug (a legitimate second charge gets silently deduped as a repeat). I want to make sure the key is generated once per checkout, not once per gateway call, before writing this.

[[response]]
Generated the idempotency key once per logical charge attempt, at the call site, and threaded it through the retry loop so every attempt within one `charge_customer` call reuses the same key:

```diff

+def charge_customer(customer_id, amount_cents, idempotency_key=None):
+    idempotency_key = idempotency_key or f"charge:{customer_id}:{uuid4()}"
     for attempt in range(3):
         try:
-            return gateway.charge(customer_id, amount_cents)
+            return gateway.charge(customer_id, amount_cents, idempotency_key=idempotency_key)
         except (requests.Timeout, requests.ConnectionError):
             continue
     raise ChargeFailed(customer_id)
```

Key is generated once, before the loop, so attempt 1, 2, and 3 all send the identical key. The gateway's documented behavior: same key within its 24h idempotency window returns the original charge's result without processing a new one, whether that original attempt succeeded, failed, or is still in flight. Callers that already have a natural idempotency key (e.g. an order ID) can pass it in explicitly instead of the generated UUID — the checkout flow at `api/checkout.py:88` now passes `idempotency_key=f"order:{order.id}"` so a customer double-clicking "pay" also can't cause a duplicate charge, not just retries at this layer.

Verified the fix with a test that simulates a timeout on the first two gateway calls and a success on the third, using a fake gateway that tracks calls by idempotency key:
