# Script 16 — memory spike that started right after a deploy

[[thinking]]
"Started right after a deploy" is the important clue — it means I should look at the diff in that deploy before doing anything else, since a memory investigation that starts from a cold heap snapshot with no hypothesis can take hours. But I also want the actual growth shape (linear vs. step) from monitoring, because a step-change at deploy time with flat memory after points at a changed default (bigger cache, bigger buffer) rather than a leak, while continuous linear growth points at something never getting freed.

[[response]]
Memory graph (Grafana, `rss_bytes{service="ingest-api"}`) shows a step up from ~180MB to ~340MB at deploy time, then a slow, steady climb afterward — roughly +6MB/hour, still climbing 18 hours later. So it's both: a changed baseline *and* an actual leak layered on top.

The step is explained by `git log` — `a91f3c2 feat: switch response cache from LRU(500) to LRU(5000)` in `src/cache/responseCache.js:12`. That's a deliberate 10x cache size increase, consistent with the ~160MB step (average cached response ~32KB × 4500 extra entries checks out).

The ongoing climb is the actual bug. Pulled two heap snapshots 30 minutes apart via `node --inspect` and compared in Chrome DevTools:

```
node -e "require('v8').writeHeapSnapshot('/tmp/heap-t0.heapsnapshot')" # via inspector, attached to live process
# 30 min later
node -e "require('v8').writeHeapSnapshot('/tmp/heap-t1.heapsnapshot')"
```

Comparison view, sorted by delta: `+Retained Size`, top entry is 41,204 new `ResponseCacheEntry` objects, +38MB, all retained by a single `Map` — `responseCache._store` — which never shrinks even though the LRU is capped at 5000.

[[thinking]]
An LRU capped at 5000 entries that's holding 41,000+ retained objects means the eviction logic isn't actually running, or isn't removing what it thinks it is. I want to read the eviction path directly rather than guess — the deploy that bumped the cap is the obvious place a bug would've been introduced, since resizing an LRU sometimes means someone touched the eviction comparison too.

[[response]]
Found it — the eviction path breaks silently when the cache key includes a timestamp. `src/cache/responseCache.js:34`:

```js
set(key, value) {
  const fullKey = `${key}:${Date.now()}`; // added in a91f3c2 for cache-busting on TTL changes
  this._store.set(fullKey, value);
  if (this._store.size > this.maxSize) {
    const oldestKey = this._store.keys().next().value;
    this._store.delete(oldestKey);
  }
}
```

The eviction logic is correct — `Map` preserves insertion order, so `.keys().next().value` really is the oldest entry, and it does delete one entry per `set()`. But `a91f3c2` introduced the `Date.now()` suffix to invalidate stale entries on TTL boundaries, without noticing that it changed `get(key)` too:

```js
get(key) {
  for (const [k, v] of this._store) {
    if (k.startsWith(key + ':')) return v; // linear scan, O(n) — separate problem
  }
}
```

`get` still works. The actual leak: `set()` is called far more often than the eviction removes, because a separate code path — `src/cache/warmup.js:19` — pre-populates the cache on a cron every 5 minutes by calling `set()` for ~200 static keys, and each call inserts a *new* timestamped key rather than overwriting the old one (they're never equal since `Date.now()` differs). One eviction per `set()` can't keep up with warmup adding 200 entries every 5 minutes on top of normal traffic — net growth, forever, capped at nothing because the "same" logical key never collides.

[[thinking]]
The timestamp suffix served a real purpose (invalidating stale TTL entries) so I shouldn't just rip it out — I need a fix that keeps that intent but stops it from defeating the size cap. The cleanest fix is separating "cache key" from "freshness," rather than encoding freshness into the key at all.

[[response]]
Fixed by storing the TTL boundary as metadata on the value instead of baking it into the key, so `set(key, ...)` on the same logical key overwrites rather than accumulating:

```diff
 set(key, value) {
-  const fullKey = `${key}:${Date.now()}`;
-  this._store.set(fullKey, value);
+  this._store.set(key, { value, cachedAt: Date.now() });
   if (this._store.size > this.maxSize) {
     const oldestKey = this._store.keys().next().value;
     this._store.delete(oldestKey);
   }
 }

 get(key) {
-  for (const [k, v] of this._store) {
-    if (k.startsWith(key + ':')) return v;
-  }
+  const entry = this._store.get(key);
+  if (!entry) return undefined;
+  if (Date.now() - entry.cachedAt > this.ttlMs) {
+    this._store.delete(key);
+    return undefined;
+  }
+  return entry.value;
 }
```

`get` is now O(1) instead of a linear scan, as a side benefit.

Deployed to canary, watched RSS for 2 hours: flat at ~342MB, no climb. `npm test -- cache` → 14 passed, added `evicts and does not duplicate on repeated set() with same key` to cover this directly. Full rollout pending your sign-off on the canary window.
