# Script 29 — a React app that grows heap on every navigation

[[thinking]]
"Detached DOM nodes" in a heap snapshot almost always means something outside React's own lifecycle is holding a reference — React itself unmounts and dereferences fine. The two usual suspects are a subscription/listener that outlives the component, or a closure captured in a ref/global that keeps the node alive. I want the heap snapshot's retainer chain before guessing which component, since that tells me exactly what's holding the reference rather than me hunting through components.

[[tool_use]]
name: Bash
input:
  command: |-
    node scripts/heap-diff.js --route=/dashboard --navigations=10 --out=/tmp/heap-diff.json && jq '.detachedNodes[] | {tag, retainedSize, retainers: .retainerChain[0:3]}' /tmp/heap-diff.json | head -40
  description: Diff heap snapshots across 10 mount/unmount cycles of the dashboard route and inspect top retainers

[[response]]
10 navigations in and out of `/dashboard` leave 214 detached DOM nodes behind, ~2.1MB retained. Every one traces through the same retainer chain:

```json
{
  "tag": "div.chart-tooltip",
  "retainedSize": 9840,
  "retainerChain": [
    "(closure) handleResize",
    "window.addEventListener('resize', ...)",
    "LiveChart"
  ]
}
```

`LiveChart` is the component. It's registering a `resize` listener on `window` — which is document-lifetime, not component-lifetime — and that listener's closure captures the chart's tooltip DOM node. Every mount adds a new listener; nothing removes the old ones.

[[thinking]]
Before I fix it I want to see the actual useEffect, because there are two different bugs that look identical from a heap snapshot: no cleanup function at all, or a cleanup function that removes the wrong reference (e.g. re-creating the handler on every render so the removeEventListener call never matches the one that's attached). The fix is different depending on which one it is.

[[response]]
It's the second, subtler variant — `src/components/LiveChart.tsx:38`:

```tsx
useEffect(() => {
  function handleResize() {
    setDimensions(getContainerSize(containerRef.current));
  }
  window.addEventListener("resize", handleResize);
  return () => window.removeEventListener("resize", handleResize);
}, [containerRef.current]); // <-- re-runs on every ref change
```

The dependency array has `containerRef.current`, a ref value — React doesn't track ref mutations as reactive, so this dependency is stale/meaningless for triggering re-runs in the way the author intended, but worse: this component also re-renders on every websocket tick (`LiveChart` re-renders on each `dashboard:metrics` push, ~1/sec), and *that's* what's actually driving effect re-runs indirectly via a parent key change I found at `Dashboard.tsx:71` — the chart is remounted with a new `key` on every metrics tick rather than updated in place. Each remount runs the effect fresh, attaches a new listener, and the cleanup from the previous mount does fire — but by the time it does, `containerRef.current` on the old instance is already null, so `getContainerSize(null)` inside the stale closure was already a silent bug too. The net effect: listeners accumulate roughly 1/sec while the dashboard is open, each retaining its own tooltip node.

[[thinking]]
Two independent bugs stacked: the remount-on-every-tick in the parent, and the ref-based dependency array in the child. Fixing only one still leaves a leak, just a slower one. I should fix both and verify with the same heap-diff tool rather than trust that the diff looks "better" — I want it back to zero detached nodes, not just fewer.

[[response]]
Fixed both.

`Dashboard.tsx:71` — stop remounting on metrics ticks, key was leftover from an earlier implementation:

```diff
-<LiveChart key={metrics.timestamp} data={metrics} />
+<LiveChart data={metrics} />
```

`LiveChart.tsx:38` — depend on nothing (attach once) and read the ref inside the handler instead of capturing size at attach time:

```diff
 useEffect(() => {
   function handleResize() {
     setDimensions(getContainerSize(containerRef.current));
   }
   window.addEventListener("resize", handleResize);
   return () => window.removeEventListener("resize", handleResize);
-}, [containerRef.current]);
+}, []);
```

Re-ran the same 10-navigation heap diff: 0 detached nodes, 0 retained bytes attributable to `LiveChart`. Also profiled 5 minutes of the dashboard sitting open with live metrics flowing (previously the leak scenario didn't even need navigation, just time) — heap holds flat at ~34MB instead of climbing to 61MB it hit before over the same window.

`npm test -- LiveChart` → 9 passed.
