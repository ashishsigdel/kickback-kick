# Script 68 — a leaking HikariCP connection pool exhausts under load

[[thinking]]
"Pool exhausts under load" with HikariCP is nearly always a connection acquired and never returned on some code path — usually an exception thrown between acquiring a connection and the try-with-resources or `close()` that would release it. Rather than reading every DAO, I want HikariCP's own leak-detection output first, since it's built to point at exactly this.

[[tool_use]]
name: Bash
input:
  command: rg -n "leakDetectionThreshold|maximumPoolSize" src/main/resources/application.yml && grep -A3 "Connection leak detection" /var/log/app/service.log | tail -40
  description: Check leak-detection config and recent leak warnings in production logs

[[response]]
`leakDetectionThreshold` wasn't set at all, so HikariCP wasn't even watching for this — that's the first fix regardless of what else I find. Enabled it locally (`leakDetectionThreshold: 30000`) and reproduced under a load test matching production burst shape (200 concurrent report-generation requests):

```

  Connection leak detection triggered for connection acquired at
  com.acme.reports.ReportDao.streamLargeExport(ReportDao.java:88), stack trace follows...
```

`src/main/java/com/acme/reports/ReportDao.java:82`:

```java
public void streamLargeExport(String reportId, OutputStream out) {
    Connection conn = dataSource.getConnection();
    Statement stmt = conn.createStatement();
    ResultSet rs = stmt.executeQuery("SELECT * FROM export_rows WHERE report_id = '" + reportId + "'");
    while (rs.next()) {
        writeRow(out, rs);   // throws IOException if client disconnects mid-stream
    }
    conn.close();
}
```

`conn.close()` is only reached if the loop completes normally. If `writeRow` throws — which it does whenever a client disconnects mid-download, common for large exports — the method exits via exception and the connection is never closed. Under normal load this is rare enough not to matter; under a burst of large-export requests with a proportionally higher disconnect rate, it exhausts the pool (`maximumPoolSize: 20`) within minutes.

[[thinking]]
try-with-resources is the obvious fix, but I should check whether `Connection`/`Statement`/`ResultSet` here are actually used in a way that's safe to wrap that way — e.g. if `conn` is reused elsewhere in a wider scope, or if there's custom cleanup logic beyond `close()` that a bare try-with-resources would skip. Worth a quick check of the whole method before rewriting it.

[[response]]
Confirmed `conn`, `stmt`, `rs` are all local to this method with no reuse elsewhere and no custom cleanup beyond `close()` — safe to wrap directly. Also noticed the string-concatenated SQL is a separate SQL injection risk (`reportId` is user-supplied via the export URL), worth fixing in the same pass since it's touching this exact line.

[[thinking]]
try-with-resources fixes this call site, but if this pattern — manual `getConnection()`/`close()` without try-with-resources — exists elsewhere in the DAO layer, those are the same bug waiting for their own burst scenario. Worth checking before calling this closed, since the incident report only mentioned this endpoint but the root cause could be systemic.

[[response]]
`rg -n "dataSource.getConnection\(\)" src/main/java -A1 | rg -B1 "createStatement|prepareStatement"` turned up two more manual-lifecycle call sites: `AuditDao.java:41` and `LegacyBillingDao.java:19`, both missing try-with-resources the same way. Fixed both identically.

Also set `leakDetectionThreshold: 30000` in `application.yml` for all environments going forward — 30s is well above any legitimate query in this service (the p99 is ~800ms per the DB dashboard), so it'll only fire on genuine leaks, not slow queries.

Re-ran the load test (200 concurrent report requests, 30% forced client-disconnect mid-stream to simulate the production pattern): pool utilization peaked at 14/20 connections and returned to baseline after the burst, versus climbing to 20/20 and staying there before. `mvn test -Dtest=ReportDaoTest,AuditDaoTest,LegacyBillingDaoTest` → `Tests run: 34, Failures: 0`.
