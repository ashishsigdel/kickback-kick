# Script 53 — CORS blocking a legitimate frontend origin

[[thinking]]
A CORS failure that only affects one origin, while others work, means the allowlist logic is doing something more specific than "CORS is broken" — a total misconfiguration would block everyone. I want the actual preflight response headers for the failing origin versus a working one, side by side, since CORS bugs are almost always visible directly in the response headers rather than needing code inspection first.


No `Access-Control-Allow-Origin` header at all in the response — the preflight succeeds at the transport level (204) but carries none of the CORS headers, so the browser correctly refuses to let the actual request through. Compared against a known-working origin (`https://admin.example.com`), which gets the full set:

```
Access-Control-Allow-Origin: https://admin.example.com
Access-Control-Allow-Methods: GET,POST,PUT,DELETE
Access-Control-Allow-Headers: Content-Type,Authorization
Access-Control-Allow-Credentials: true
```

`src/middleware/cors.js:12`:

```js
const allowedOrigins = [
  "https://admin.example.com",
  "https://app.example.com/",   // <- trailing slash
];

app.use(cors({
  origin: (origin, callback) => {
    if (!origin || allowedOrigins.includes(origin)) {
      callback(null, true);
    } else {
      callback(new Error("Not allowed by CORS"));
    }
  },
  credentials: true,
}));
```

`https://app.example.com/` (with trailing slash) is in the allowlist; the `Origin` header the browser actually sends is `https://app.example.com` (no trailing slash — the `Origin` header is never sent with a path or trailing slash per spec). `allowedOrigins.includes(origin)` does exact string comparison, so it never matches, `callback` fires with an error, and the `cors` middleware responds with no CORS headers at all rather than a rejection message — which is why this shows up as a silent block in the browser console instead of an explicit error.

[[thinking]]
A hardcoded exact-match array is fragile for exactly this reason — one stray character breaks an origin silently, and there's no test that would have caught it since it's a data entry mistake, not a logic bug. Before just fixing the trailing slash I should check whether this list is meant to eventually support wildcards or multiple environments (staging subdomains, preview deploys), since if so, the better fix is a more robust comparison, not just correcting one string.

[[response]]
Checked `docs/environments.md` and the deploy config — there are exactly two production frontend origins today, no wildcard subdomains, no preview-deploy pattern to support. So I'm not over-engineering this into a wildcard matcher; the fix is correcting the entry and normalizing on the way in so this specific mistake can't recur silently:

```diff
 const allowedOrigins = [
   "https://admin.example.com",
-  "https://app.example.com/",
+  "https://app.example.com",
 ];

 app.use(cors({
   origin: (origin, callback) => {
-    if (!origin || allowedOrigins.includes(origin)) {
+    const normalized = origin?.replace(/\/$/, "");
+    if (!origin || allowedOrigins.includes(normalized)) {
       callback(null, true);
     } else {
+      console.warn(`CORS rejected origin: ${origin}`);
       callback(new Error("Not allowed by CORS"));
     }
   },
   credentials: true,
 }));
```

The normalization is defense-in-depth for the next entry someone adds by hand, not a substitute for fixing the actual wrong string — both changes together mean this specific class of typo can't silently reproduce. Added the `console.warn` on rejection too, since the original failure mode (silent 204 with no headers, no server-side log) made this take longer to diagnose than it should have — anyone hitting this again will see it in the server logs immediately rather than needing a manual `curl` comparison.
