# Script 52 — logs missing for the exact window of an incident

[[thinking]]
"Missing for exactly one window" rather than "missing everywhere" is the important detail — a systemic logging failure would show gaps all over, random drops would look scattered. A clean, bounded gap points at something that happened once: a shipper restart, a disk-full event, or a rate limiter kicking in under the exact load spike the incident caused. Check the host in that window before assuming application code is at fault.

[[tool_use]]
name: Bash
input:
  command: journalctl -u vector --since "2026-07-19 14:00" --until "2026-07-19 14:45" | grep -iE "error|restart|buffer|drop"
  description: Check the log shipper's own logs for the incident window

[[response]]
Found it in the shipper's own log, not the application:

```
14:07:52 vector[1]: WARN sink{component_id=datadog_logs}: vector::sinks::util::buffer: Buffer full; dropping events. dropped_events_total=48219
14:07:52 vector[1]: WARN sink{component_id=datadog_logs}: request timed out, retrying
14:08:41 vector[1]: INFO vector: buffer drained, resuming normal operation
```

`vector` (the log shipping agent) hit a full buffer at 14:07:52 and started dropping events on the floor rather than blocking the application or spilling to disk. That's a 49-second window — 14:07:52 to 14:08:41 — and it lines up exactly with the reported gap. This isn't the application failing to log; the application logged normally, and the shipper discarded 48,219 events before they ever left the host.

`vector.toml:38`:

```toml
[sinks.datadog_logs]
type = "datadog_logs"
buffer.type = "memory"
buffer.max_events = 5000
buffer.when_full = "drop_newest"
```

`buffer.type = "memory"` with `drop_newest` means: if the downstream (Datadog's ingest endpoint) can't keep up, once the in-memory buffer of 5,000 events fills, every new event past that is silently discarded rather than queued to disk or blocking upstream.

[[thinking]]
Why did the buffer fill at exactly that moment — was ingest degraded on the receiving side, or did the incident itself generate a burst large enough to overwhelm a normally-sufficient buffer? These point at different fixes: a degraded-ingest cause is a one-off; incident-driven volume means any future incident of similar severity reproduces the exact same gap — logging disappears precisely when it's needed most.

[[response]]
Event volume in the minutes before the drop climbed from a baseline ~800 events/sec to over 14,000 events/sec starting at 14:07:40, twelve seconds before the buffer filled.

That's when the upstream payment service started throwing errors, and every error logs a full stack trace plus request/response bodies at `WARN` — the incident itself generated the burst that exhausted the buffer. Worse case: the pipeline's failure mode is coupled to error volume, so any incident severe enough to matter is also most likely to erase its own evidence. Same pattern happened in a smaller incident on 2026-06-03 — 6,140 dropped, under the threshold anyone noticed at the time.

[[thinking]]
The fix isn't "make the buffer bigger" — that just moves the threshold, and the next incident that's 3x worse reproduces the exact same failure. What actually breaks the coupling is giving the buffer somewhere to overflow to that isn't bounded by incident severity: disk-backed buffering, so a burst spills to local disk instead of getting dropped, plus alerting on `dropped_events_total` itself so this is never silently discovered after the fact again.

[[response]]
Two changes, aimed at breaking the coupling between incident severity and log loss rather than just raising a number:

```diff
 [sinks.datadog_logs]
 type = "datadog_logs"
-buffer.type = "memory"
-buffer.max_events = 5000
-buffer.when_full = "drop_newest"
+buffer.type = "disk"
+buffer.max_size = 268435488  # 256MB on-disk, holds ~15min at incident-level burst rate
+buffer.when_full = "block"
```

Disk-backed buffering means a burst spills to local disk rather than memory, and `when_full = "block"` means if the disk buffer itself somehow fills, the shipper applies backpressure rather than silently discarding — worth noting this could in theory block the application's log writes under a truly extreme burst, which is a deliberate trade: backpressure is visible (things slow down, someone notices) where silent dropping is not.

Also added a Vector-internal metric alert — `dropped_events_total` increasing at all now pages, at any volume, since 6,140 silently dropped in the June incident should have been caught the first time:

```yaml
# monitors/vector_dropped_events.yaml
alert: LogShipperDroppingEvents
expr: increase(vector_component_discarded_events_total[5m]) > 0
severity: warning
```

Replayed the 14:07–14:08 burst against a staging shipper with the new config (synthetic load generator producing the same 14K events/sec for 90s): 0 dropped, buffer peaked at 41MB of the 256MB disk allowance, drained within 12s of load returning to baseline.

The 49-second gap in the July 19 incident and the smaller June 3 gap are both permanently unrecoverable — nothing was actually persisted for either window. Flagging that rather than trying to reconstruct anything: if root-causing those specific incidents still needs that window's detail, it isn't recoverable from logs, only from whatever the application's own error tracker (separate pipeline, unaffected) captured for the same period.
