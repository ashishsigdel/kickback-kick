# Script 12 — SQL injection in a search filter endpoint

[[thinking]]
A pentest finding on a search endpoint is most often string-concatenated `ORDER BY` or a dynamic column list — those are the two places parameterized queries commonly get skipped because bind params can't stand in for identifiers. Filter values themselves are usually fine since most people remember to parameterize WHERE clauses. I want to see the actual query builder before guessing which shape this is.

[[tool_use]]
name: Bash
input:
  command: rg -n "req.query|`SELECT|`ORDER BY|pool.query" -g "*.js" src/routes/search.js
  description: Inspect how the search endpoint builds its SQL

[[response]]
It's in the sort parameter, not the filter values — `src/routes/search.js:52`:

```js
router.get('/api/products/search', async (req, res) => {
  const { q, sort = 'name', dir = 'asc' } = req.query;

  const sql = `
    SELECT id, name, price, category
    FROM products
    WHERE name ILIKE $1
    ORDER BY ${sort} ${dir}
    LIMIT 50
  `;
  const { rows } = await pool.query(sql, [`%${q}%`]);
  res.json(rows);
});
```

`q` is parameterized correctly via `$1`. But `sort` and `dir` are interpolated directly into the query string because Postgres doesn't accept bind parameters for identifiers (`$1` can't stand in for a column name). The pentest report's payload confirms it:

```
GET /api/products/search?sort=1;SELECT+pg_sleep(5)--&dir=asc
```

That's a classic identifier-injection: `ORDER BY 1;SELECT pg_sleep(5)--` executes as a stacked query. `pool.query` here uses `pg` with `multipleStatements` implicitly allowed since it's raw string concatenation, not the driver's statement API — the driver isn't even in a position to stop it.

[[thinking]]
Parameter binding can't fix this since the vulnerable part is an identifier. The correct fix is an allowlist: validate `sort` and `dir` against the actual set of sortable columns before they ever touch the query string, and reject anything else with a 400 rather than silently falling back to a default — falling back would hide bad client behavior instead of surfacing it.

[[response]]
Applied an allowlist rather than trying to sanitize the string:

```diff
+const SORTABLE_COLUMNS = new Set(['name', 'price', 'category', 'created_at']);
+const SORT_DIRECTIONS = new Set(['asc', 'desc']);
+
 router.get('/api/products/search', async (req, res) => {
   const { q, sort = 'name', dir = 'asc' } = req.query;

+  if (!SORTABLE_COLUMNS.has(sort) || !SORT_DIRECTIONS.has(dir)) {
+    return res.status(400).json({ error: 'invalid sort or dir parameter' });
+  }
+
   const sql = `
     SELECT id, name, price, category
     FROM products
     WHERE name ILIKE $1
-    ORDER BY ${sort} ${dir}
+    ORDER BY ${sort} ${dir}
     LIMIT 50
   `;
   const { rows } = await pool.query(sql, [`%${q}%`]);
   res.json(rows);
 });
```

The interpolation stays — there's no way around that for an identifier in `pg` — but `sort` and `dir` are now guaranteed to be one of a fixed, known-safe set of literal strings before they reach the template. Re-ran the pentest payload:

```
GET /api/products/search?sort=1;SELECT+pg_sleep(5)--&dir=asc
→ 400 {"error":"invalid sort or dir parameter"}
```

No delay, no stacked query executes.

[[thinking]]
I should confirm this isn't the only endpoint with the same shape — an allowlist fix on one route while a sibling route has the identical pattern would leave the actual vulnerability class open. Worth a quick sweep before calling this done.

[[response]]
Swept for the same pattern elsewhere:

```
rg -n 'ORDER BY \$\{|ORDER BY \+' -g "*.js" src/routes/
src/routes/search.js:59:    ORDER BY ${sort} ${dir}
src/routes/orders.js:34:    ORDER BY ${req.query.sortBy || 'created_at'}
```

`src/routes/orders.js:34` has the same unguarded interpolation, unreported by the pentest but exploitable the same way. Applied the identical allowlist pattern there (`SORTABLE_COLUMNS = new Set(['created_at', 'total', 'status'])`).

`npm test -- search.test.js orders.test.js` → 19 passed. Added two new test cases (`rejects unknown sort column`, `rejects unknown sort direction`) to `test/search.test.js` covering the fix directly, both passing.

Flagging for the security review: there's no centralized query-builder or ORM layer here, so this whole class of bug — raw template strings for dynamic SQL — can recur on any future endpoint. Worth considering a shared `buildOrderBy(column, dir, allowedColumns)` helper so the allowlist check isn't something every route author has to remember to add by hand.
