# Script 89 — cross-account data access in a multi-tenant API

[[thinking]]
A user seeing another account's data is either a missing `WHERE account_id = ?` on one query, or an authorization check that validates the JWT but never compares its tenant claim against the resource being fetched. Those look identical from the outside but the fix location is completely different, so I need the actual request path, not just the report of "user A saw user B's invoices."

[[tool_use]]
name: Bash
input:
  command: rg -n "router\.(get|post|put|delete)\('/invoices" src/routes/invoices.js src/middleware/auth.js
  description: Find the invoice route and whatever tenant-scoping middleware sits in front of it

[[response]]
Reproduced with two test accounts. `GET /api/invoices/:id` for an invoice belonging to tenant B, called with tenant A's JWT, returns 200 with tenant B's data.

`src/middleware/auth.js:22` verifies the JWT and attaches `req.tenantId` from the token claim — that part's fine. The route itself, `src/routes/invoices.js:41`:

```js
router.get('/invoices/:id', requireAuth, async (req, res) => {
  const invoice = await Invoice.findByPk(req.params.id);
  if (!invoice) return res.status(404).end();
  res.json(invoice);
});
```

`requireAuth` only checks that the token is valid — it never compares `req.tenantId` against the row it fetched. `Invoice.findByPk` has no tenant filter at all, so any authenticated user can pull any invoice by guessing or incrementing sequential IDs.

[[thinking]]
Before writing the one-line fix for this route, I want to know whether this is one route with a missing check or a pattern repeated across the other resource routes — invoices, payments, customers. If it's systemic, patching this route and calling it done leaves the same hole open in six other places.

[[response]]
It's systemic. Grepped every `findByPk`/`findOne` in `src/routes/`:

```
src/routes/invoices.js:41    Invoice.findByPk(req.params.id)         — no tenant filter
src/routes/payments.js:29    Payment.findByPk(req.params.id)         — no tenant filter
src/routes/customers.js:18   Customer.findByPk(req.params.id)        — no tenant filter
src/routes/subscriptions.js:33  Subscription.findOne({ where: { id } }) — no tenant filter
```

Four routes, same shape of bug: `requireAuth` authenticates but nothing authorizes at the row level. Only `orders.js:52` does it correctly (`where: { id, tenantId: req.tenantId }`), which is presumably why this wasn't caught earlier — orders was the reference implementation and the other resources didn't copy the tenant filter when they were added.

[[thinking]]
The safest fix here isn't four individual patches — that's exactly how the fourth route ends up missed again next time someone adds a resource. A Sequelize default scope or a shared query helper that makes tenant filtering the path of least resistance is worth more than fixing today's four instances by hand. I'll patch the four now and propose the guardrail separately.

[[response]]
Patched all four to scope by tenant:

```diff
-const invoice = await Invoice.findByPk(req.params.id);
+const invoice = await Invoice.findOne({ where: { id: req.params.id, tenantId: req.tenantId } });
 if (!invoice) return res.status(404).end();
```

Same change in `payments.js`, `customers.js`, `subscriptions.js`. Returning 404 rather than 403 on tenant mismatch deliberately — a 403 would confirm to an attacker that the ID exists in *some* tenant, a 404 doesn't.

`npm test -- routes/invoices routes/payments routes/customers routes/subscriptions` → 34 passed. Added a cross-tenant regression test per resource (tenant A token, tenant B's row ID → 404) so this can't regress silently.

I haven't added the guardrail yet — a Sequelize scope like `Model.scope({ method: ['forTenant', tenantId] })`, or wrapping every model in a base repository that requires a tenant ID to query — because that's a structural change touching every model, not a bug fix, and it's worth a design decision rather than me picking one under a security-incident timeline. Want me to draft that as a separate follow-up?
