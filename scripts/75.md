# Script 75 — partial writes when a bulk insert fails midway

[[thinking]]
Partial writes on failure means either the transaction boundary is wrong (not actually wrapping the whole batch) or the driver is doing something like autocommit per statement that defeats an otherwise-correct `BEGIN`. Since psycopg2/psycopg3 both default to non-autocommit unless explicitly configured, I should check the connection setup before assuming the application code is at fault.

Found the bug, and it's not autocommit — it's a transaction that's closed too early. `ingest/loader.py:52`:

```python
def bulk_insert_events(conn, rows):
    with conn.cursor() as cur:
        for chunk in batched(rows, 5000):
            cur.executemany(
                "INSERT INTO events (id, payload, ts) VALUES (%s, %s, %s)",
                chunk,
            )
            conn.commit()
    conn.close()
```

`conn.commit()` runs after every 5,000-row chunk, inside the loop. There's no outer transaction spanning the whole call — each chunk is its own committed transaction. If chunk 4 of 10 raises (bad payload, constraint violation, connection blip), chunks 1–3 are already durably committed and chunks 5–10 never ran. The caller sees an exception and assumes nothing was written, but 15,000 rows are sitting in the table.

[[thinking]]
The fix is to make the whole batch one transaction, but that has a real tradeoff — a 50k-row batch failing on row 49,999 now rolls back all 50k instead of losing only the last chunk, and it holds one long-running transaction against the table instead of many short ones. I want to check whether anything downstream depends on the current per-chunk commit behavior (e.g. a progress indicator reading partial counts) before changing it.

[[response]]
Checked callers — `ingest/cli.py:28` only checks the return value/exception after `bulk_insert_events` returns, nothing reads intermediate row counts mid-call. No dependency on partial visibility. Single transaction is safe to adopt.

`executemany` chunking is still useful for memory/protocol reasons, it just no longer commits per chunk — one `BEGIN`/`COMMIT` around the whole batch, explicit `rollback()` on any exception before the connection closes (previously a raised exception skipped straight to `conn.close()`, which on some pool configs implicitly rolls back but shouldn't be relied on).

A long-held write lock on a hot table can back up other writers even if the transaction itself succeeds.

Not a concern at that scale; would reconsider if batch sizes grow by an order of magnitude.
