# Script 43 — a slow GraphQL query, missing index and over-fetching

[[thinking]]
"Slow GraphQL query" can mean the resolver is doing something dumb (N+1 across fields), the underlying SQL is unindexed, or the query itself is asking for more than the client uses. Those look identical from the client's timing. The Prisma query log will separate them — if I see one query per field per row, that's resolver-level; if it's one query but it's slow, that's the database.

[[tool_use]]
name: Bash
input:
  command: DEBUG="prisma:query" node -e "require('./dist/index.js')" 2>&1 | grep -A2 "storefront { products" | head -60
  description: Capture the actual Prisma queries generated for the slow storefront query

[[response]]
It's both, layered.

One query, not N+1 — good. But it's this:

```sql
SELECT "id","name","description","sku","weightGrams","dimensions",
       "supplierCost","internalNotes","createdAt","updatedAt",
       "vendorId","categoryId","searchVector"
FROM "Product"
WHERE "categoryId" = $1
ORDER BY "createdAt" DESC
LIMIT 24 OFFSET $2;
```

`EXPLAIN ANALYZE` on it:

```
Limit  (actual time=612.331..612.340 rows=24 loops=1)
  ->  Sort  (actual time=612.318..612.322 rows=24 loops=1)
        Sort Key: "createdAt" DESC
        ->  Seq Scan on "Product"  (actual time=0.018..588.204 rows=41,208 loops=1)
              Filter: ("categoryId" = 'cat_9f2...')
              Rows Removed by Filter: 118,904
```

612ms, sequential scan across 160K rows, no index on `categoryId`. And the GraphQL query only asks for `id, name, price, thumbnailUrl` in the schema — the resolver is selecting all 14 columns including `searchVector` (a tsvector, expensive to serialize) and `internalNotes`, which shouldn't be reachable from this resolver at all.

[[thinking]]
Two separate fixes with two separate risk profiles. The index is safe and mechanical. The over-fetch fix touches the resolver's field selection, and Prisma's `select` needs to be built from the requested GraphQL fields (using `graphql-fields` or similar) rather than hand-listing columns, otherwise this same bug reappears the next time someone adds a field to the query. Worth checking how the resolver currently decides what to select before I change it.

[[response]]
`resolvers/product.ts:34` — the resolver doesn't look at the GraphQL selection set at all, it just calls `prisma.product.findMany()` with no `select`, so Prisma defaults to `SELECT *`:

```ts
async function products(_, { categoryId, first, after }) {
  return prisma.product.findMany({
    where: { categoryId },
    orderBy: { createdAt: "desc" },
    take: first,
    skip: after,
  });
}
```

Fixed by projecting from the actual requested fields using `graphql-fields`, so the `select` clause tracks whatever the query asks for instead of drifting from the schema:

```diff
+import graphqlFields from "graphql-fields";
+
-async function products(_, { categoryId, first, after }) {
+async function products(_, { categoryId, first, after }, ctx, info) {
+  const requested = Object.keys(graphqlFields(info));
   return prisma.product.findMany({
     where: { categoryId },
     orderBy: { createdAt: "desc" },
     take: first,
     skip: after,
+    select: Object.fromEntries(requested.map((f) => [f, true])),
   });
}
```

`internalNotes` and `supplierCost` are still on the Prisma model — they're just no longer reachable unless a resolver explicitly asks, which is the right default for fields that shouldn't be client-visible in the first place. Worth a separate pass to check no other resolver leans on the `SELECT *` default the same way.

[[thinking]]
Index next. `categoryId` alone matches the WHERE clause, but the query also sorts by `createdAt` and paginates with `LIMIT/OFFSET` — a composite index in `(categoryId, createdAt)` lets Postgres satisfy the filter and the sort from the same index without a separate sort step, which matters once row counts grow.

[[response]]
Added the index via Prisma migration:

```prisma
// schema.prisma
model Product {
  // ...
  @@index([categoryId, createdAt])
}
```

```
$ npx prisma migrate dev --name add_product_category_created_index
```

Re-ran `EXPLAIN ANALYZE`:

```
Limit  (actual time=1.204..1.298 rows=24 loops=1)
  ->  Index Scan using Product_categoryId_createdAt_idx on "Product"
        (actual time=1.198..1.284 rows=24 loops=1)
```

612ms → 1.3ms for the query; combined with the select-projection fix, the GraphQL response payload for this field dropped from ~9KB/row to ~380B/row since `searchVector` alone was most of it.

`npm run test:resolvers` → 44 passed. Client-side `first`/`after` pagination behavior unchanged — verified with the existing Cypress spec for the storefront grid, still green.

One thing I didn't touch: pagination here is offset-based (`skip: after`), which will show the same drift-under-inserts problem as any offset pagination once the catalog is being edited concurrently with someone paging through it. Not urgent for a storefront grid, but worth knowing if `after` is ever exposed as a stable cursor to an external API.
