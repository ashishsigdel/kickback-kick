Green on every PR but red on main is a strong signal: the difference isn't the code, it's the environment or the event that triggers the run. Pull request builds and push builds get different refs, different concurrency, and often different secrets. Let me read the workflow before I read any test.

[[response]]
The workflow is the problem, not the tests.

`.github/workflows/ci.yml:8`

```yaml
on:
  pull_request:
  push:
    branches: [main]
```

Both events run the same job — but `actions/checkout` at line 21 uses the default `fetch-depth: 1`. On a pull request GitHub gives you a merge ref with both parents reachable, so `git diff origin/main...HEAD` resolves. On a push to main you get a single commit with no history, and that same diff command has nothing to compare against.

`scripts/check-changed.sh:12` runs exactly that diff. On main it returns everything as "changed", so the changed-files lint runs across the whole repo and trips on files nobody touched.

Nothing is wrong with the tests — the failure is the shallow clone plus a diff that assumes history.

[[thinking]]
Two candidate fixes and they're not equivalent. Fetching full history is one line and always correct but slows every run on a big repo. Making the script handle the no-history case is more code but keeps clones shallow. The right answer depends on repo size, so I should measure rather than assume.

[[response]]
Checked the repo size before choosing: 4,200 commits, ~38MB of history. A full fetch adds roughly 6s to a job that currently takes 3m40s.

That makes the decision easy — 6s for correctness is worth it, and `fetch-depth: 0` is far harder to get subtly wrong than branch-point logic in bash:

```yaml
- uses: actions/checkout@v4
  with:
    fetch-depth: 0
```

I'd have argued the other way on a repo with 200k commits, where the fetch would dominate the run. That isn't this repo.

What I verified locally: with a full clone, `git diff --name-only origin/main...HEAD` returns 3 files on a branch and empty on main itself — which is the correct behaviour in both cases. Under `--depth=1` the same command errors with `fatal: no merge base`, reproducing your CI failure exactly.

What I can't verify from here: the actual CI run. The reproduction matches the symptom precisely, so I'm fairly confident, but the confirming evidence is a green build on main — worth watching the next push rather than taking my word for it.

If it's still red after this, the next thing I'd look at is the `concurrency` block at line 14. It cancels in-progress runs by ref, and pushes to main share a ref in a way PR runs don't — a cancelled job can report as a failure. That would be a second, independent cause rather than a sign this fix was wrong.
