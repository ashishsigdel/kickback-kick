# Script 62 — expired JWTs are being accepted

[[thinking]]
"Expired tokens let through" in a JWT setup is usually one of a small number of causes: `exp` not being checked at all, a clock skew allowance that's too generous, or a verify call that's silently falling back to decode-without-verify somewhere. Rather than reading the whole auth module, I want to reproduce it directly with a token I mint and expire myself, so I know exactly what "through" means before hunting for why.

[[tool_use]]
name: Bash
input:
  command: |-
    node -e "
    const jwt = require('jsonwebtoken');
    const token = jwt.sign({sub: 'user_1'}, process.env.JWT_SECRET, {expiresIn: -60});
    fetch('http://localhost:4000/api/me', {headers: {authorization: 'Bearer ' + token}})
      .then(r => r.json()).then(console.log);
    "
  description: Mint a token expired 60s ago and hit the protected endpoint directly

[[response]]
Confirmed — `{ sub: 'user_1', iat: ..., exp: ... }` comes back as a 200 with the user's profile. A token that expired a minute ago is being treated as valid.

`src/middleware/auth.js:18`:

```javascript
function verifyToken(token) {
  const decoded = jwt.decode(token);
  if (!decoded || !decoded.sub) return null;
  return decoded;
}
```

That's `jwt.decode`, not `jwt.verify`. `decode()` in `jsonwebtoken` just base64-parses the payload — it doesn't check the signature and doesn't check `exp`. Any well-formed JWT, expired or not, forged or not, passes this function as long as it has a `sub` claim.

[[thinking]]
Before assuming this is the only call site, I should check whether `jwt.verify` is used anywhere else in the codebase correctly — if so, this might be one code path that regressed (e.g. a refactor that swapped verify for decode by mistake) rather than the auth system never having verified anything, which would change how worried I am about the blast radius.

[[response]]
`rg -n "jwt\.(verify|decode)" src/` shows three call sites: `src/middleware/auth.js:18` (the broken one, `decode`), `src/routes/refresh.js:31` (`verify`, correct), and `src/services/websocket-auth.js:14` (`decode` again — same bug, second call site).

Checked git blame on `auth.js:18` — this was `jwt.verify(token, secret)` until `a3f9c02`, four months ago, a refactor titled "extract shared token parsing." The refactor pulled the parsing logic into a shared helper and swapped to `decode` because `verify` throws on failure and the new helper wanted to return `null` instead of throwing — but nobody added the `exp`/signature checks back in.

[[thinking]]
Before fixing just these two call sites, I should check whether the refresh-token flow has the same exposure — if refresh tokens go through a similarly broken decode, an attacker could mint fresh access tokens indefinitely off an expired or revoked refresh token, which is a bigger problem than one expired access token slipping through.

[[response]]
Checked `src/routes/refresh.js:31` — already uses `jwt.verify` correctly, and additionally checks a `tokenVersion` claim against `users.token_version` in the DB, so a validly-signed refresh token is still rejected after a password reset or explicit revocation. That path was never affected — the bug is contained to the two `decode()` call sites already found.

[[thinking]]
The fix needs to preserve the "return null instead of throw" behavior the refactor wanted, not just swap back to a throwing `verify` and break every call site expecting null-on-failure. Wrapping verify in a try/catch gets both properties without reintroducing the original exception-based control flow this was refactored away from.

[[response]]
Fixed both call sites with a shared wrapper that keeps the null-return contract:

```diff
+function safeVerify(token) {
+  try {
+    return jwt.verify(token, process.env.JWT_SECRET, { algorithms: ['HS256'] });
+  } catch (err) {
+    return null;
+  }
+}
+
 function verifyToken(token) {
-  const decoded = jwt.decode(token);
-  if (!decoded || !decoded.sub) return null;
+  const decoded = safeVerify(token);
+  if (!decoded || !decoded.sub) return null;
   return decoded;
 }
```

Same change applied at `src/services/websocket-auth.js:14`. Pinned `algorithms: ['HS256']` explicitly rather than leaving it to autodetect — `jsonwebtoken` will honor an `alg: none` or `alg: RS256` header from the token itself if you don't restrict it, which is a separate known bypass class for libraries in this family.

Re-ran the repro: expired token now gets `401 { error: "jwt expired" }`. Ran the full auth suite plus added a regression test for both call sites: `npm test -- auth` → `47 passed, 0 failed`. Also confirmed a token signed with a different secret is rejected, and one with `alg: none` and a stripped signature is rejected — both were silently accepted before this fix, not just expired ones.
