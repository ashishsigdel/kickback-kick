I want to look specifically at context usage and status codes rather than business logic, since those are the parts that are expensive to retrofit later and easy to get subtly wrong even in code that "works" in the happy path.

[[response]]
Read the diff for `services/pricing`. It's a clean service otherwise — proto is well-shaped, request validation is thorough — but three things around error handling and timeouts need to change before this joins the mesh:

**1. Dropped deadline on the downstream call** — `services/pricing/internal/fx/client.go:41`

```go
func (c *Client) ConvertRate(ctx context.Context, from, to string, amount decimal.Decimal) (decimal.Decimal, error) {
    ctx = context.Background() // <-- overwrites the caller's ctx entirely
    resp, err := c.fxClient.Convert(ctx, &fxpb.ConvertRequest{...})
    ...
}
```

The incoming `ctx` — which carries the caller's deadline and any tracing metadata — is discarded and replaced with `context.Background()`, which never times out and carries nothing. If `fx-service` hangs, this call hangs forever regardless of what deadline the original caller set, and the hang is invisible in traces because the span parent is gone.

**2. Errors collapsed to `Unknown`** — `services/pricing/internal/server.go:88`

```go
rate, err := c.fxClient.ConvertRate(ctx, req.From, req.To, req.Amount)
if err != nil {
    return nil, status.Error(codes.Internal, "pricing failed")
}
```

Every failure — invalid currency code, fx-service down, deadline exceeded, a real internal bug — comes back to the caller as the same opaque `Internal`. A caller can't distinguish "retry me, I was down for a second" from "this currency pair doesn't exist, retrying won't help," so any retry policy built against this service either retries everything (including permanent failures) or nothing.

[[response]]
Checked the dial site — `services/pricing/internal/fx/client.go:18`:

```go
conn, err := grpc.Dial(cfg.FXServiceAddr, grpc.WithInsecure())
```

No `grpc.WithTimeout`, no keepalive params, no `WithBlock`. That's actually *correct* for `Dial` itself in modern grpc-go (connections are meant to be long-lived and non-blocking at dial time), so the bound has to come from the per-call context, not the dial — which makes fix #1 above the actual load-bearing one, not a nice-to-have alongside a dial-level timeout. Confirmed there's no other layer quietly saving this.

**3. Status codes**, mapping recommendation:

| Failure | Current | Should be |
| --- | --- | --- |
| Unknown currency pair | `Internal` | `InvalidArgument` — caller's fault, don't retry |
| fx-service unreachable | `Internal` | `Unavailable` — transient, safe to retry |
| Deadline exceeded on downstream | `Internal` | `DeadlineExceeded` — propagate, don't retry blindly |
| Actual panic/bug | `Internal` | `Internal` — this one's correctly mapped already |

**Summary of required changes before merge:**

```diff
-func (c *Client) ConvertRate(ctx context.Context, from, to string, amount decimal.Decimal) (decimal.Decimal, error) {
-    ctx = context.Background()
+func (c *Client) ConvertRate(ctx context.Context, from, to string, amount decimal.Decimal) (decimal.Decimal, error) {
+    ctx, cancel := context.WithTimeout(ctx, 2*time.Second) // bounded, and inherits caller's deadline if shorter
+    defer cancel()
     resp, err := c.fxClient.Convert(ctx, &fxpb.ConvertRequest{...})
+    if err != nil {
+        return decimal.Zero, mapFXError(err) // preserves status code instead of collapsing it
+    }
```
