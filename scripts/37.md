What's actually in the file is `Fran\xc3\x83\xc2\xa7ois` — that's the UTF-8 encoding *of the string* `"Ã§"`, which is itself what you get from decoding `\xc3\xa7` as Latin-1 and then re-encoding as UTF-8. The name has been through UTF-8 → (misread as Latin-1) → UTF-8 twice.

[[thinking]]
Double-encoding means there are two separate steps to find: the code has to decode the source bytes with the wrong codec at some point, and separately re-encode as UTF-8 later. The source read and the CSV write are the two candidate sites in a pandas pipeline — I want to check both rather than assume it's the well-known "pandas to_csv defaults to utf-8" issue, because that alone wouldn't double-encode anything, it would just produce a file Excel misreads. The double-encoding tells me the bug is earlier, at ingestion.

Found it at ingestion, not export. `etl/load_customers.py:19`:

```python
raw = pd.read_csv(source_path, encoding="latin-1")
```

The source file from the upstream vendor is genuinely UTF-8. Reading it with `encoding="latin-1"` doesn't error (Latin-1 maps every byte 0-255 to a codepoint) — it just silently misinterprets multi-byte UTF-8 sequences as pairs of Latin-1 characters. The next line tries to "fix" that by re-encoding as Latin-1 and decoding as UTF-8, which is the correct repair only if the string was misdecoded exactly once — applied blindly, it's the source of the double-encoding, not the export step. The export path's own `to_csv(..., encoding="utf-8")` at `etl/export_customers.py:34` adds no further corruption; everything traces back to ingestion.
