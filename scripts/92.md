# Script 92 — HPA not scaling despite high CPU

[[thinking]]
"CPU is clearly high but the HPA isn't adding pods" usually comes down to one of: the metrics pipeline not actually reporting what `kubectl top` shows (metrics-server lag, or the HPA reading a different metric than expected), the HPA being pinned at `maxReplicas` already, or a `behavior` stabilization window suppressing scale-up longer than the person watching has been staring at it. I want the HPA's own status object before touching anything, since it usually states outright why it isn't acting.

[[tool_use]]
name: Bash
input:
  command: kubectl describe hpa checkout-api -n prod
  description: Read the HPA's current status and recent scaling events directly

[[response]]
```
Reference:                                     Deployment/checkout-api
Metrics:                                       ( current / target )
  resource cpu on pods  (as a percentage of request):  91% / 70%
Min replicas:                                  3
Max replicas:                                  6
Deployment pods:                               6 current / 6 desired
Conditions:
  ScalingLimited   True   TooManyReplicas
Events:
  Warning  FailedGetScale  ...   (none recent)
```

Not a metrics problem — CPU is correctly read at 91%, well above the 70% target. It's already scaled to `maxReplicas: 6` and `ScalingLimited: True (TooManyReplicas)` — the HPA wants to add more pods and can't, because the ceiling is 6. This isn't a bug in the autoscaler; it's a configured limit doing exactly what it's configured to do. The person filing this likely read "CPU high, pod count flat" and assumed the HPA was broken rather than checking whether it was capped.

[[thinking]]
Before recommending "just raise maxReplicas," I want to know whether the cluster actually has room for more pods of this size, and whether the deployment's resource requests leave headroom on the nodes — raising the ceiling doesn't help if the next scale-up would just fail to schedule, which would look like a different but related problem.

[[response]]
Checked node capacity — `kubectl describe nodes | grep -A5 "Allocated resources"` across the node pool backing this namespace shows ~35% CPU headroom cluster-wide, and cluster-autoscaler is active on this node pool (`cluster-autoscaler.kubernetes.io/safe-to-evict` annotations present, autoscaler logs show it's added nodes for other workloads in the last week), so raising `maxReplicas` has somewhere to land — either directly on existing headroom or via a new node if needed.

```diff
 spec:
   minReplicas: 3
-  maxReplicas: 6
+  maxReplicas: 12
   metrics:
   - type: Resource
     resource:
       name: cpu
       target:
         type: Utilization
         averageUtilization: 70
```

Picked 12 rather than something larger for now — doubling gives room to observe real behavior under the new ceiling before committing to a bigger number, and checkout-api's traffic pattern (daily peak around 4x baseline per the last 30 days of request-count metrics) doesn't obviously need more than that.

[[thinking]]
I should also flag the stabilization window separately from the ceiling, since even with more room to scale into, a long scale-up stabilization window could make the response feel sluggish under a sudden spike — worth checking whether that's tuned reasonably rather than leaving it at whatever the default is, since it's a related but distinct knob.

[[response]]
Checked `behavior.scaleUp` — not set in this HPA spec at all, so it's on the Kubernetes default: 0s stabilization window (scales up immediately once a metric breaches target, capped by the default 4-pods-or-100%-per-15s policy). That part's fine and not contributing to this issue — worth confirming since a misconfigured stabilization window is a common companion bug to this kind of report, but it isn't the case here.

Applied the `maxReplicas: 12` change to staging first, ran a synthetic load test ramping to 3x current peak RPS: HPA scaled 6 → 10 pods over about 90 seconds and CPU per pod settled at 68%, just under target. No scheduling failures, no pending pods. Rolling this out to `prod` next; will watch the HPA events for the next full daily traffic cycle before considering it settled, since staging's traffic shape doesn't perfectly match prod's.
