# Script 17 — broken pagination cursor that skips or repeats rows

[[thinking]]
Cursor pagination bugs almost always trace to one of two things: the cursor encodes a field that isn't unique (ties break unpredictably) or the sort order used to build the cursor doesn't match the sort order used to apply it. Skipping and repeating are actually different symptoms — repeats usually mean the cursor's boundary is inclusive on both sides, skips usually mean a non-unique sort key. The bug report says "sometimes skips, sometimes repeats," which suggests a non-unique key rather than an off-by-one, since that would explain both depending on how ties happen to sort. Let me look at the resolver.

Confirmed — non-unique sort key. `src/graphql/resolvers/comments.ts:58`:

```ts
async function commentsConnection(postId: string, first: number, after?: string) {
  const afterCreatedAt = after ? decodeCursor(after).createdAt : null;

  const rows = await db.query(`
    SELECT id, body, created_at
    FROM comments
    WHERE post_id = $1
      AND ($2::timestamptz IS NULL OR created_at > $2)
    ORDER BY created_at ASC
    LIMIT $3
  `, [postId, afterCreatedAt, first + 1]);

  return buildConnection(rows, first, (row) => encodeCursor({ createdAt: row.created_at }));
}
```

The cursor is `created_at` alone, and `created_at` is not unique — comments imported in a batch (visible in `db/seeds/import_legacy_comments.rb`) share the exact same millisecond timestamp, so `WHERE created_at > $2` either:

- **skips** every row tied with the boundary row that sorted before it in the same page (they share `created_at`, so `>` excludes all of them, including ones never returned), or
- **repeats** rows that tied with the boundary row but sorted *after* it — if the last row returned this page and the first row of the tie group next page share a timestamp, `>` should exclude the whole tie group, but if the app is paginating client-side with a `>=` in a different code path (checked — there's a second, older resolver at `legacy/commentsPage.ts:22` still wired into the REST API using `>=`), that one repeats instead of skips.

Two different bugs sharing one root cause: no tiebreaker in the sort or the cursor.

[[thinking]]
The fix is standard — add a unique, stable secondary sort key (id) to both the ORDER BY and the cursor comparison, so ties resolve deterministically instead of by whatever order Postgres happens to return them in. I want to make sure the comparison becomes a proper compound comparison, not just an added ORDER BY column, because `created_at > $2 AND id > $3` alone would still be wrong at the tie boundary (it'd exclude ties with a smaller id even when created_at differs).

[[response]]
Fixed with a compound cursor and the correct compound comparison — not just `created_at > $2 AND id > $3`, which would wrongly exclude same-timestamp rows with a smaller id when `created_at` actually differs:

```diff
 async function commentsConnection(postId: string, first: number, after?: string) {
-  const afterCreatedAt = after ? decodeCursor(after).createdAt : null;
+  const cursor = after ? decodeCursor(after) : null;

   const rows = await db.query(`
     SELECT id, body, created_at
     FROM comments
     WHERE post_id = $1
-      AND ($2::timestamptz IS NULL OR created_at > $2)
-    ORDER BY created_at ASC
-    LIMIT $3
-  `, [postId, afterCreatedAt, first + 1]);
+      AND ($2::timestamptz IS NULL OR (created_at, id) > ($2, $3))
+    ORDER BY created_at ASC, id ASC
+    LIMIT $4
+  `, [postId, cursor?.createdAt ?? null, cursor?.id ?? null, first + 1]);

-  return buildConnection(rows, first, (row) => encodeCursor({ createdAt: row.created_at }));
+  return buildConnection(rows, first, (row) => encodeCursor({ createdAt: row.created_at, id: row.id }));
 }
```

`(created_at, id) > ($2, $3)` is Postgres row-wise comparison — exactly the semantics needed: strictly after in `created_at`, or equal in `created_at` and strictly after in `id`. `id` is the primary key, so it's a guaranteed unique tiebreaker.
