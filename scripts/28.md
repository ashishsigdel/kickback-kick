# Script 28 — events processing out of order across regions

[[thinking]]
Out-of-order processing that only shows up across nodes, never within a single node's own stream, points at cross-node timestamp comparison rather than a queueing bug — if it were a local ordering bug it would reproduce on one machine. Before I look at the merge/sort code I want to see whether the timestamps themselves are trustworthy, which means checking clock offset between the nodes actually involved.

[[response]]
```
ingest-us-east-1: System time     : 0.000031 seconds fast of NTP time
ingest-eu-west-1: System time     : 0.000019 seconds slow of NTP time
ingest-ap-south-1: System time     : 0.412 seconds fast of NTP time
```

`ap-south-1` is 412ms off — chrony's been failing to sync there. `journalctl -u chronyd` on that box shows repeated `Source ... replaced with better one` and `No suitable source for synchronisation` going back 6 days; it's been free-running off a degraded local clock, drifting since a security group change blocked outbound NTP (port 123/udp) to the pool during a firewall rollout.

That's within the same order of magnitude as the reordering window support flagged (events out of order when `event.timestamp` deltas are under ~500ms), so it's a strong candidate but not proof yet — I want to see how the merge logic actually uses these timestamps before concluding the clock is sufficient explanation.

[[thinking]]
The merge code is the other half of the story. If it orders purely by `event.timestamp` (wall clock, generated at the edge node) rather than a value the ordering can't be fooled by — a Kafka offset, a Lamport/HLC counter — then a 412ms clock skew is exactly enough to reorder events that are genuinely close in real time. I need to confirm that's the sort key before I can call this root-caused.

[[response]]
Confirmed. `internal/merge/reorder.go:58`:

```go
sort.Slice(batch, func(i, j int) bool {
    return batch[i].Timestamp.Before(batch[j].Timestamp)
})
```

`Timestamp` is set at `internal/ingest/handler.go:22` via `time.Now()` on the edge node at receipt — pure wall clock, no correction, no logical component. Any two events within the skew window (412ms, and growing while chronyd stays broken) can sort backwards regardless of true arrival order. This explains the specific complaint: `ap-south-1` events consistently appear to precede events that support confirmed arrived earlier from other regions.

Root cause is infrastructure (blocked NTP), but the ordering code has no defense against it — a single misbehaving clock in one region can silently corrupt global ordering, and nothing detects it. Two separate things need fixing.

[[thinking]]
Fixing the firewall rule stops this specific incident but doesn't stop the next one — some future clock drift, some future NTP outage, will do the same thing again unless the ordering logic is skew-tolerant. I want to propose the durable fix, not just the incident fix, and be clear about which is which.

[[response]]
Two changes, different urgency.

**Immediate (incident):** restored outbound 123/udp to the NTP pool on `ap-south-1`'s security group. `chronyc tracking` now shows sync within 2ms after 20 minutes. This stops new skew from accumulating but does nothing for already-ingested events.

**Structural (root cause):** replaced wall-clock ordering with a hybrid logical clock so future clock drift can't silently reorder events:

```diff
 type Event struct {
-    Timestamp time.Time
+    Timestamp time.Time // retained for display/debugging only
+    HLC       hlc.Timestamp // (wall, counter, nodeID) — used for ordering
 }

 sort.Slice(batch, func(i, j int) bool {
-    return batch[i].Timestamp.Before(batch[j].Timestamp)
+    return batch[i].HLC.Before(batch[j].HLC)
 })
```

HLC still uses wall time as its primary component so ordering stays close to real time under normal conditions, but the logical counter advances on every send/receive and is monotonic per-node regardless of clock skew — a node with a broken clock can no longer reorder events relative to itself, only (bounded) relative to others, and the bound is enforced by the HLC's max-drift check rather than trusted blindly.

Backfilled the last 6 days of `ap-south-1` events by re-deriving order from Kafka partition offset (which is skew-immune) rather than the corrupted timestamp, and replayed the affected downstream aggregations. `go test ./internal/merge/... -run TestHLC -v` → ok, 14 tests. Also added a chrony-offset alert (`>50ms sustained`) so an NTP outage pages someone instead of silently corrupting ordering for six days before anyone notices.
