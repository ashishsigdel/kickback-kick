# Script 88 — stale cache serving old data seconds after a confirmed write

[[thinking]]
"Stale for a few seconds, then correct" rules out a TTL misconfiguration — that would be stale for the full TTL, not a handful of seconds. The two live suspects are a race between the write and the cache invalidation, or a read that repopulates the cache with data read from a replica that hasn't caught up yet. I want to see the actual write path before speculating further.

[[response]]
`app/repositories/product_repo.py:58`:

```python
def update_price(self, product_id, new_price):
    db.session.execute(
        update(Product).where(Product.id == product_id).values(price=new_price)
    )
    db.session.commit()
    cache.delete(f"product:{product_id}")
```

That order looks right — DB commit, then invalidate — so it's not a write-before-commit bug. The suspect is what repopulates the key afterward. `app/services/pricing.py:31`:

```python
def get_price(product_id):
    cached = cache.get(f"product:{product_id}")
    if cached:
        return cached
    product = Product.query.get(product_id)   # reads from the read replica
    cache.set(f"product:{product_id}", product.price, ex=300)
    return product.price
```

The read on cache miss goes to `Product.query.get`, and this app's SQLAlchemy session is bound to a read replica (`config.py:19`, `SQLALCHEMY_BINDS["read"]`). If a request lands in the invalidate→repopulate window before the replica has applied the write, it caches the *old* price for the next 300 seconds.

[[thinking]]
To confirm this is actually replica lag and not something else, I want a number — how far behind is the replica typically, and does it line up with "a few seconds" rather than sub-second. If replica lag is usually 20ms, this theory is wrong and I'm looking at something else, like a second writer or a retry re-caching the old value.

Confirmed — replica lag was 2.8s at the time of the check and the on-call's report says the stale read cleared "after a few seconds," which matches. This is a classic read-your-writes gap: invalidate against primary, repopulate from a replica that hasn't replayed the write yet, and the stale value gets cached for the full 300s TTL instead of just the lag window.

Two independent fixes, and I'd do both:

1. **Repopulate from primary on cache miss**, at least for this table — `Product.query.get(product_id)` → route through the primary-bound session. Removes the race entirely for this path, at the cost of extra primary read load.
2. **Shrink the blast radius if it does race** — set a much shorter TTL for cache-miss repopulation (e.g. 5s) with a background refresh, or better: write-through the new price into the cache in the same request as the DB write, instead of deleting and letting the next reader repopulate it.

[[thinking]]
Write-through avoids the replica read entirely for the common case and is the more complete fix, but it changes the cache-consistency model for every caller of `update_price`, not just this endpoint — worth checking whether other write paths already assume delete-then-lazy-repopulate before I commit to changing the pattern everywhere.
