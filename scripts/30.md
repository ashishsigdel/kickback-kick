# Script 30 — GraphQL resolver fanning out one query per row

[[thinking]]
GraphQL N+1s hide better than REST ones because the symptom is "this query is slow" rather than an obvious loop in application code — the fan-out happens inside the resolver tree, one field resolver invoked per item in a list, and nothing in the handler itself looks wrong. Before reading resolver code I want the actual query log for a single request, since the shape of the duplication tells me exactly which field is at fault.

[[response]]
51 — one query for the feed itself, then one `select * from "users" where "id" = ?` per post. Confirmed N+1.

`src/graphql/resolvers/post.ts:22`:

```ts
Post: {
  author: (post) => db("users").where({ id: post.authorId }).first(),
},
```

Standard field resolver, called once per `Post` object returned by the parent `feed` resolver — GraphQL's execution model invokes it independently for each of the 50 posts, with no batching unless the resolver author does it explicitly. This is the pattern DataLoader exists for, and there's no loader in this codebase yet — `rg "dataloader" -g "*.ts" src/` returns nothing.

Before wiring in DataLoader I want to check whether `author` is the only offender or whether the same pattern repeats elsewhere in the schema — fixing one field and shipping while a sibling field (say, `Post.comments` or `Post.reactions`) has the identical shape would just move the complaint to a different query, and I'd rather catch that now than in the next report.

| --- | --- | --- |

All three key off a foreign ID (`authorId`, `postId`) and hit the same underlying tables (`users`, `reactions`), so a single pair of loaders scoped per-request covers all of them.

[[thinking]]
DataLoader's contract is strict about batching happening within one request only — a loader instantiated at module scope would cache and batch across requests, silently leaking one user's data into another's cache and batching unrelated requests together. It has to be constructed fresh per request in context. Worth being explicit about that in the diff since it's the easiest way to get this wrong.

[[response]]
Added per-request loaders in `src/graphql/context.ts:14`, scoped to the request so nothing leaks or batches across users:

```diff
+import DataLoader from "dataloader";
+
 export function createContext(req) {
+  const userLoader = new DataLoader(async (ids: readonly string[]) => {
+    const rows = await db("users").whereIn("id", ids);
+    const byId = new Map(rows.map((r) => [r.id, r]));
+    return ids.map((id) => byId.get(id) ?? null); // must preserve input order
+  });
   return {
     db,
+    userLoader,
     currentUser: req.user,
   };
 }
```

Resolvers now go through the loader instead of the raw table:

```diff
 Post: {
-  author: (post) => db("users").where({ id: post.authorId }).first(),
+  author: (post, _args, ctx) => ctx.userLoader.load(post.authorId),
 },
```

Same change for `Comment.author` reusing the same `userLoader` — DataLoader dedupes and batches keys within the same tick regardless of which resolver called `.load()`, so posts and comments sharing an author collapse into the same batch. `Post.reactionCounts` got its own loader keyed by `postId` since it hits a different table.

Re-ran the 50-post feed query: `select * from "users"` count went 51 → 1 (`where id in (...)`, deduped to unique author IDs). `Comment.author` on a feed with nested comments went from 250 → 3 queries.

`npm run test:graphql` → 41 passed. Didn't touch caching headers or query complexity limits — this was strictly the batching fix; happy to look at complexity limiting separately if `feed` depth is a concern.
