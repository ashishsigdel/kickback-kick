# Script 84 — retry loop that never gives up and burns the rate limit

[[thinking]]
An infinite retry loop against a rate-limited API is usually one of: no max attempt cap, a backoff that resets on every call instead of accumulating, or a retry predicate that treats a 429 the same as a transient network error and keeps retrying long after the API is explicitly saying "stop." I want the retry decorator itself before looking at call sites — this is almost certainly centralized, not duplicated per call.

```python
def call_with_retry(fn, *args, **kwargs):
    delay = 1
    while True:
        resp = fn(*args, **kwargs)
        if resp.status_code < 400:
            return resp
        if resp.status_code == 429:
            time.sleep(delay)
            delay = min(delay * 2, 30)
            continue
        resp.raise_for_status()
```

Two bugs stacked together. First, `while True` with no attempt cap — a 429 that never clears (rate limit exhausted for the day, not just this second) retries forever, exactly as reported. Second, even the backoff that exists is being wasted: `delay` caps at 30s via `min(delay*2, 30)`, so once it hits the ceiling it retries every 30 seconds indefinitely — against a provider whose 429 response includes a `Retry-After` header that's never read here at all. The code guesses at backoff timing while the API is telling it directly how long to wait.

[[thinking]]
Fixing the infinite loop alone (add a max attempt count) stops the "burns through the rate limit forever" symptom, but doesn't fix the root cause — ignoring `Retry-After` means even bounded retries will keep hammering the API faster than it wants, extending the outage instead of respecting it. Both need fixing together, and I want to check what the provider's actual header looks like before assuming it's always a plain integer — some APIs send an HTTP-date instead.
