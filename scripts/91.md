# Script 91 — a sorting comparator violates transitivity

[[thinking]]
`IllegalArgumentException: Comparison method violates its general contract!` from `Collections.sort` on a large-enough list means the comparator returns inconsistent results for some triple of elements — usually because it composes multiple fields with subtraction (integer overflow) or independent boolean checks that don't compose into a total order. I want the actual comparator before guessing which pattern it is.

[[tool_use]]
name: Bash
input:
  command: rg -n "class.*Comparator|compare\(" src/main/java/com/app/catalog/ProductRanking.java
  description: Find the comparator implementation that's throwing

[[response]]
`ProductRanking.java:22`:

```java
public int compare(Product a, Product b) {
    if (a.isFeatured() && !b.isFeatured()) return -1;
    if (a.getRating() > b.getRating()) return -1;
    if (a.getRating() < b.getRating()) return 1;
    return a.getPrice().compareTo(b.getPrice());
}
```

Two bugs stacked here, both breaking transitivity.

**1. `isFeatured()` check is asymmetric.** It only handles "a featured, b not" — it never checks the reverse case, so `compare(a, b)` and `compare(b, a)` aren't consistent when `b` is featured and `a` isn't: that branch is skipped entirely and falls through to the rating comparison, ignoring `isFeatured` on that side. That alone can produce `compare(x,y) < 0`, `compare(y,z) < 0`, `compare(x,z) > 0` — exactly the kind of triple that trips the contract check.

**2. Featured status and rating are compared in the wrong order for a total order.** Even fixed symmetrically, "featured beats everything, then rating, then price" needs featured to be checked as a complete field comparison before rating is consulted at all — right now a non-featured product with a much higher rating can beat a featured product in some pairings and lose in others depending on which branch fires first, because the featured check is incomplete rather than absolute.

[[thinking]]
The cleanest fix is to stop hand-rolling the composition and build it out of `Comparator.comparing().thenComparing()` chains, which Java's own combinators guarantee are a total order as long as each individual key extractor is well-behaved — that removes the whole class of bug rather than patching this one asymmetry.

[[response]]
Rewrote using `Comparator` composition instead of manual branching:

```diff
-public int compare(Product a, Product b) {
-    if (a.isFeatured() && !b.isFeatured()) return -1;
-    if (a.getRating() > b.getRating()) return -1;
-    if (a.getRating() < b.getRating()) return 1;
-    return a.getPrice().compareTo(b.getPrice());
-}
+private static final Comparator<Product> RANKING =
+    Comparator.comparing(Product::isFeatured).reversed()
+              .thenComparing(Comparator.comparingDouble(Product::getRating).reversed())
+              .thenComparing(Product::getPrice);
+
+public int compare(Product a, Product b) {
+    return RANKING.compare(a, b);
+}
```

`Comparator.comparing(Product::isFeatured)` sorts `false` before `true` by default, so `.reversed()` puts featured first — and crucially, it's evaluated as a complete boolean comparison on both elements symmetrically, not an asymmetric if-branch. `thenComparing` only consults rating when featured status is equal, and only consults price when both are equal, which is exactly the total order that was intended but not actually implemented.

Added a property-based regression test using jqwik — generate random triples of `Product` and assert the contract holds directly, rather than relying on `Collections.sort` happening to hit a bad triple:

```java
@Property
void comparatorIsTransitive(@ForAll List<Product> products) {
    assertThatCode(() -> products.sort(RANKING)).doesNotThrowAnyException();
}
```

`mvn test -Dtest=ProductRankingTest` → 1,000 generated cases, 0 failures. Previously this threw within the first ~40 generated lists once list size exceeded Java's Timsort merge threshold (32 elements), which also explains why it was passing code review and small unit tests for months — the contract violation only surfaces with enough elements to trigger a merge that exposes the inconsistency.
