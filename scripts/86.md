[[response]]
Diff adds `internal/httpsign/sign.go` (client side) and `internal/httpsign/verify.go` (server middleware). Two problems, both real.

**1. Timing side channel — `verify.go:41`**

```go
expected := hex.EncodeToString(mac.Sum(nil))
if expected != r.Header.Get("X-Signature") {
    return errInvalidSignature
}
```

String `!=` short-circuits on the first differing byte, so response latency leaks how many leading bytes of the guess were correct. It's a classic byte-at-a-time oracle — slow over the internet, trivial over a shared VPC with low jitter, which is exactly the deployment here (service mesh, sub-ms RTT). Needs `hmac.Equal`:

```go
if !hmac.Equal(mac.Sum(nil), sig) {
    return errInvalidSignature
}
```

**2. Signature doesn't bind method or path — `sign.go:22`**

```go
mac := hmac.New(sha256.New, secret)
mac.Write(body)
```

Only the body is signed. Two endpoints that happen to take the same (or empty) body — say `GET /v1/accounts/42/balance` and `POST /v1/accounts/42/close` with no body — produce identical signatures. A signature captured off one route is valid on the other.

[[thinking]]
Before I flag the fix, I want to know whether there's any replay protection at all — a timestamp or nonce — because that changes how bad #2 is in practice. If a captured signature is valid forever, binding method+path is urgent even without solving replay. If it's not, the two issues compound.

[[response]]
No replay protection either. The client sets `X-Signature-Timestamp` (`sign.go:19`) but `verify.go` never reads it — grepped for `Timestamp` in the verify path, zero hits. So a captured request+signature pair is valid indefinitely, not just within a window.

Recommended canonical string, covering both gaps in one change:

```go
canonical := strings.Join([]string{
    r.Method,
    r.URL.Path,
    r.Header.Get("X-Signature-Timestamp"),
    sha256Hex(body),
}, "\n")
mac := hmac.New(sha256.New, secret)
mac.Write([]byte(canonical))
```

Plus reject on the server if `abs(now - timestamp) > 300s`. That closes replay-after-5-minutes and the method/path confusion in the same diff, and it's a small change to a file that hasn't shipped yet.

That's a real infra decision (Redis SETNX, TTL matching the window, cache sizing per RPS) rather than a code-review nit, and it's not clear from the diff whether they've budgeted for it. Summary: `hmac.Equal` and the canonical-string fix (method + path + timestamp + body hash) are both required before this merges — cheap now, and this is unreleased so there's no compatibility cost to changing the wire format.

One open question I didn't decide for you: a 5-minute timestamp window still allows replay *within* that window — a captured request can be resent any number of times for 5 minutes. Closing that fully needs a nonce store (e.g. `SETNX sig:<hash> 1 EX 300` in Redis, one write per inbound call) rather than just a wider or narrower clock check. That's extra infra and an extra Redis round-trip per request on every internal call, so it's worth deciding deliberately — do you want replay closed completely, or is the timestamp window an acceptable tradeoff for the call volume this handles?
