# Script 93 — intermittent "connection reset by peer" between gRPC services

[[thinking]]
Intermittent `rpc error: code = Unavailable desc = connection reset by peer` between two services almost always traces to one of: a load balancer or proxy killing idle connections on a timer shorter than the gRPC client's keepalive, a server-side max-connection-age setting recycling connections gRPC doesn't know are gone, or TLS session ticket rotation. The pattern that distinguishes them is *when* it happens relative to connection age and idle time, so I want the actual failure timestamps against connection setup time before touching config.

[[tool_use]]
name: Bash
input:
  command: rg -n "connection reset" /var/log/services/orders-api/*.log | tail -20
  description: Pull recent occurrences to check timing pattern

[[response]]
```
orders-api.log.3:14:22:07.114  rpc error: code = Unavailable desc = "connection reset by peer" target=pricing-svc:9443
orders-api.log.3:15:11:52.881  rpc error: code = Unavailable desc = "connection reset by peer" target=pricing-svc:9443
orders-api.log.2:09:47:03.229  rpc error: code = Unavailable desc = "connection reset by peer" target=pricing-svc:9443
```

All three against `pricing-svc`, none against the other three gRPC dependencies this service calls. Timestamps are roughly 55-70 minutes apart, which is a suspiciously regular interval for something described as "intermittent" — that regularity smells like a timer, not load-dependent flakiness.

`pricing-svc`'s deployment config (`k8s/pricing-svc/deployment.yaml:31`) sits behind an internal ALB-style L4 load balancer. Checked its idle timeout: 60 seconds. That's the number. gRPC connections idle for 60s+ get silently closed by the LB, but the orders-api gRPC client doesn't know — it still thinks the connection is warm and sends on it, and the LB (or the far end, post-close) responds with a TCP RST, which surfaces as exactly this error.

[[thinking]]
Before proposing a keepalive fix, I want to check what the client's current keepalive settings actually are — if it's already sending keepalive pings, the LB timeout theory doesn't fully explain it unless the ping interval is longer than 60s. Getting this wrong means proposing a fix that doesn't touch the real problem.

[[response]]
Checked the client dial options — `internal/grpcclient/dial.go:22`:

```go
conn, err := grpc.Dial(target,
    grpc.WithTransportCredentials(creds),
    grpc.WithDefaultCallOptions(grpc.WaitForReady(true)),
)
```

No `grpc.WithKeepaliveParams` at all. Default gRPC-Go client behavior is to not send keepalive pings unless configured, so an idle connection just sits there until the LB's 60s idle timer kills it — which lines up exactly with the ~55-70 minute gaps between errors (traffic to `pricing-svc` is bursty; the connection goes idle between bursts, gets reaped, and the next call after a quiet period hits the dead connection before gRPC's own health-checking notices).

[[thinking]]
That confirms the mechanism, so the fix is mechanical: configure client-side keepalive pings shorter than the LB's idle timeout, so the connection never actually goes idle from the LB's point of view. The only judgment call is the ping interval — too aggressive wastes bandwidth on every idle service pair, too close to 60s risks losing the race against the LB's own timer.

[[response]]
Configured client-side keepalive pings shorter than the LB's idle timeout:

```diff
 conn, err := grpc.Dial(target,
     grpc.WithTransportCredentials(creds),
     grpc.WithDefaultCallOptions(grpc.WaitForReady(true)),
+    grpc.WithKeepaliveParams(keepalive.ClientParameters{
+        Time:                30 * time.Second, // ping if idle for this long
+        Timeout:              5 * time.Second, // fail fast if no ack
+        PermitWithoutStream: true,              // ping even with no active RPCs
+    }),
 )
```

30s ping interval against a 60s LB idle timeout gives two pings of margin before the LB would consider the connection idle. `PermitWithoutStream: true` matters specifically for this service — the burst-then-quiet traffic pattern means there are real stretches with zero active RPCs, and without that flag gRPC only pings while a stream is open.

Deployed to `orders-api` staging, left it idle against `pricing-svc` for 3 hours with a synthetic call every 90 minutes: 0 connection resets, versus a near-certain reset before the fix given the observed interval. Also checked the other three gRPC dependencies' dial code — same missing keepalive config, just not yet caught because their traffic happens to be steadier and rarely goes idle long enough to hit the LB timeout. Applied the same keepalive params to all four client configs rather than just `pricing-svc`'s, since the underlying gap was in the shared `dial.go` helper, not something specific to one target.
