# Script 23 — reviewing a diff that switches floating point math to fixed-point for money

[[thinking]]
Moving money math from float to fixed-point is almost always the right direction, but the diff itself is where the risk concentrates — a partial conversion (some paths converted, some not) is worse than the original all-float code, because now two representations of the same quantity coexist and a silent implicit conversion between them is exactly the kind of bug that produces a plausible-looking wrong number instead of a crash. I want to see the whole diff before judging the approach, specifically looking for any remaining `double` in a code path that touches the new fixed-point type.

[[response]]
The core `Money` type is well done — `include/finance/money.hpp:1`:

```cpp
class Money {
public:
    explicit Money(int64_t minor_units) : minor_units_(minor_units) {}
    static Money from_major(int64_t major, int minor_per_major = 100) {
        return Money(major * minor_per_major);
    }
    Money operator+(const Money& o) const { return Money(minor_units_ + o.minor_units_); }
    Money operator*(int64_t scalar) const { return Money(minor_units_ * scalar); }
    int64_t minor_units() const { return minor_units_; }
private:
    int64_t minor_units_;
};
```

Integer cents, no implicit float conversion path — good, there's no `operator double()` or implicit constructor from `double` anywhere in the header, so the compiler will reject accidental mixing rather than silently converting.

But the conversion isn't complete. `src/pricing/discount_engine.cpp:88` still computes percentage discounts in `double` before constructing a `Money`:

```cpp
Money apply_discount(Money price, double discount_pct) {
    double discounted = price.minor_units() * (1.0 - discount_pct / 100.0);
    return Money(static_cast<int64_t>(discounted)); // truncates, doesn't round
}
```

This is exactly the partial-conversion risk — the type system can't catch it because `minor_units()` legitimately returns an `int64_t` that then gets promoted to `double` for the multiply. A 15% discount on $19.99 (1999 cents): `1999 * 0.85 = 1699.15`, `static_cast<int64_t>` truncates to `1699` — off by nothing here, but `static_cast` always truncates toward zero rather than rounding, so this systematically undercharges by up to a cent on every discounted line, and it's still doing float arithmetic on money, which was the entire premise of the migration.

[[thinking]]
This function is the one place that genuinely needs a fraction (a percentage isn't expressible in integer cents alone), so the fix isn't "remove all doubles" — it's "do the float math only for the ratio, then round deliberately and explicitly at the boundary back into integer cents," rather than truncating implicitly via a cast. I also want to check whether banker's rounding or standard round-half-up is expected here, since it affects reconciliation against the finance team's numbers — that's a policy question, not something to decide unilaterally.

[[response]]
Flagging the rounding-direction question rather than guessing — checked `docs/finance/rounding_policy.md`, which does specify round-half-up for discounts (exists, just not being followed here). With that settled, the fix:

```diff
 Money apply_discount(Money price, double discount_pct) {
-    double discounted = price.minor_units() * (1.0 - discount_pct / 100.0);
-    return Money(static_cast<int64_t>(discounted)); // truncates, doesn't round
+    double discounted = price.minor_units() * (1.0 - discount_pct / 100.0);
+    return Money(static_cast<int64_t>(std::llround(discounted))); // round-half-up per rounding_policy.md
 }
```

`std::llround` rounds half away from zero, which matches round-half-up for the positive values this function always operates on (discounted prices are never negative — worth a defensive `assert(discount_pct >= 0.0 && discount_pct <= 100.0)` too, added).
