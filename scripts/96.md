# Script 96 — a Spark batch job's output is nondeterministic between reruns

[[thinking]]
Same input, different output, on a distributed engine points at something that depends on partition/task ordering — a `groupBy` followed by an operation that isn't commutative (like taking `first()` of a group), or a UDF touching wall-clock time or random state without a fixed seed, or floating-point summation order changing with shuffle partitioning. I want to see whether the difference is in row *values* or row *order* first, since that split narrows it a lot.

[[tool_use]]
name: Bash
input:
  command: spark-submit jobs/daily_rollup.py --input s3://data/raw/2026-07-21/ --output /tmp/run_a && spark-submit jobs/daily_rollup.py --input s3://data/raw/2026-07-21/ --output /tmp/run_b && diff <(sort /tmp/run_a/*.csv) <(sort /tmp/run_b/*.csv) | head -20
  description: Run the job twice on identical input and diff the sorted output to isolate value drift from ordering

[[response]]
After sorting, most rows match — this isn't row ordering, which rules out "output written in nondeterministic partition order" as the whole story. But 340 of ~50,000 rows genuinely differ in value:

```
< store_042,2026-07-21,revenue,18422.50
> store_042,2026-07-21,revenue,18422.51
< store_119,2026-07-21,revenue,9103.17
> store_119,2026-07-21,revenue,9103.16
```

All the diffs are penny-level on `revenue`, which is a `sum()` aggregation over floating-point `DoubleType` columns in `jobs/daily_rollup.py:28`:

```python
rollup = (
    df.groupBy("store_id", "date")
      .agg(F.sum("line_total").alias("revenue"))
)
```

Floating-point addition isn't associative — `(a + b) + c` can differ from `a + (b + c)` at the last bit. Spark sums each partition's rows in whatever order they land in, and partition-to-task assignment isn't guaranteed identical between runs (depends on executor scheduling, speculative execution, shuffle read order), so the summation order for a given `store_id` group changes run to run, and the rounding error changes with it.

[[thinking]]
Before proposing "switch to decimal," I should confirm this really is float summation order and not, say, `line_total` itself carrying nondeterministic values upstream (a join fanning out differently, or a dedup step that isn't stable). If upstream row *counts* per group also vary between runs, the fix is elsewhere; if row counts match exactly and only the sum differs, it's purely a floating-point ordering artifact and the fix is contained to this aggregation.

[[response]]
Confirmed it's contained here — added a row-count check per group across both runs:

```python
df.groupBy("store_id", "date").count()
```

identical between `run_a` and `run_b` for all 340 affected groups, same `line_total` values feeding in (checked via a hash of the sorted input rows per group). So it's exactly what it looks like: correct rows, correct count, non-associative float summation producing a different last-digit result depending on task execution order. Not a correctness bug in the traditional sense — both outputs are "right" to within float precision — but it fails any downstream check that expects byte-identical output between reruns (this job feeds a reconciliation diff against a legacy system that expects exact match).

[[thinking]]
The clean fix is to do currency math in a type that doesn't have this problem at all rather than trying to control summation order, since pinning order in a distributed shuffle is fragile and would need to survive every future change to partitioning or executor count. Decimal is the standard answer for money; the question is just how invasive the type change is across the pipeline.

[[response]]
Switched `line_total` and the `revenue` aggregate to `DecimalType`, which Spark sums using exact fixed-point arithmetic — no rounding-order dependence:

```diff
-df = spark.read.schema(StructType([
-    ...,
-    StructField("line_total", DoubleType()),
-]))
+df = spark.read.schema(StructType([
+    ...,
+    StructField("line_total", DecimalType(12, 2)),
+]))

 rollup = (
     df.groupBy("store_id", "date")
-      .agg(F.sum("line_total").alias("revenue"))
+      .agg(F.sum("line_total").cast(DecimalType(14, 2)).alias("revenue"))
 )
```

`DecimalType(12, 2)` matches the source data's actual precision (currency to the cent); the wider `(14, 2)` on the aggregate output gives headroom for the sum across many rows without overflow.

Ran 10 back-to-back executions on the same input, diffed all pairs: byte-identical output every time, including the previously-drifting 340 groups. Also re-ran the existing job test suite since changing the schema type touches downstream consumers: `pytest tests/jobs/test_daily_rollup.py -q` → 19 passed. Flagging separately that two other jobs in `jobs/` (`weekly_rollup.py`, `ltv_estimate.py`) aggregate the same `line_total` column via `DoubleType` and would have the identical nondeterminism if anything downstream ever diffs their output byte-for-byte — worth the same fix if that's ever a requirement there too.
