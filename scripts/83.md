# Script 83 — chunked upload corrupts large files

[[thinking]]
Corruption specifically on large files, not small ones, points at something that only kicks in once a file spans multiple chunks — ordering, off-by-one on byte ranges, or a race between chunks uploading in parallel. A single-chunk file can't exhibit an ordering bug, which is a useful first filter. I want the chunk-assembly code, not the upload UI, since that's where bytes actually get concatenated.

[[response]]
Found it, and it's a concurrency bug, not a math bug. `server/upload/assembler.js:38`:

```js
async function receiveChunk(uploadId, chunkIndex, buffer) {
  const filePath = tempPath(uploadId);
  await fs.promises.appendFile(filePath, buffer);
  markReceived(uploadId, chunkIndex);
}
```

The client uploads chunks in parallel — `client/upload.js:52` fires all chunk requests concurrently with `Promise.all` for throughput. `fs.appendFile` on a shared path is not safe against concurrent callers: it opens the file, seeks to end, and writes, but two concurrent `appendFile` calls against the same path can interleave — both open the fd at the same "end of file" position before either has flushed, and whichever write lands second either overwrites bytes from the first or the two writes' bytes get interleaved depending on OS/filesystem buffering. On a small file that finishes in one chunk this never triggers, since there's nothing to interleave against. On a large file split into 40 chunks racing in parallel, some ordering collides and the file comes out with garbled or duplicated byte ranges.

[[thinking]]
`appendFile` is the wrong primitive here regardless of concurrency — even serialized, "append whatever arrives next" only produces correct output if chunks are guaranteed to arrive in order, which parallel upload explicitly doesn't guarantee. The real fix is to write each chunk to its own byte offset rather than appending, so arrival order stops mattering entirely — that fixes both the race and the ordering assumption in one change.

[[response]]
Rewrote to write each chunk at its own fixed offset instead of appending, using a file handle opened once per upload and `write(buffer, 0, length, position)`:

```diff
-async function receiveChunk(uploadId, chunkIndex, buffer) {
-  const filePath = tempPath(uploadId);
-  await fs.promises.appendFile(filePath, buffer);
-  markReceived(uploadId, chunkIndex);
-}
+async function receiveChunk(uploadId, chunkIndex, buffer) {
+  const filePath = tempPath(uploadId);
+  const handle = await getOrOpenHandle(uploadId, filePath);
+  const position = chunkIndex * CHUNK_SIZE;
+  await handle.write(buffer, 0, buffer.length, position);
+  markReceived(uploadId, chunkIndex);
+}
```

`handle.write(..., position)` writes to an explicit offset regardless of what else is happening on the same fd — two concurrent writes to *different* offsets on the same open file handle don't race each other, since there's no shared "current position" being contended. `getOrOpenHandle` caches one open `FileHandle` per `uploadId` so concurrent chunk writes reuse it rather than each racing to open the file fresh (which was a second, smaller bug in the original — no fd reuse at all, every chunk opened and closed its own handle).

One assumption baked into `position = chunkIndex * CHUNK_SIZE`: every chunk except possibly the last must be exactly `CHUNK_SIZE` bytes. Checked the client — `client/upload.js:31` slices with `file.slice(i * CHUNK_SIZE, (i + 1) * CHUNK_SIZE)`, so that holds for all clients hitting this endpoint today. If a future client ever sends variable-size chunks this offset math breaks silently (writes overlap or leave gaps) rather than erroring, so it's a real constraint worth keeping visible, not just true today by accident.

`getOrOpenHandle` caching one `FileHandle` per `uploadId` raises a resource-lifetime question I should settle before shipping this: what closes that handle, and what happens if a client abandons an upload halfway through — does the handle (and its fd) leak for the life of the process?

[[response]]
Checked — `getOrOpenHandle`'s cache had no eviction at all in my first pass, which would leak an open fd per abandoned upload. Added explicit lifecycle: the handle is closed and evicted from the cache either when `finalizeUpload` completes successfully, or by a sweep (`server/upload/gc.js:12`, already existed for cleaning up orphaned temp files) extended to also close any cached handle whose upload hasn't received a chunk in 30 minutes.

```diff

   if (!handles.has(uploadId)) {
-    handles.set(uploadId, await fs.promises.open(filePath, 'a+'));
+    const handle = await fs.promises.open(filePath, 'w+');
+    handles.set(uploadId, { handle, lastWrite: Date.now() });
   }
-  return handles.get(uploadId);
+  const entry = handles.get(uploadId);
+  entry.lastWrite = Date.now();
+  return entry.handle;
 }
```

Also switched the open flag from `'a+'` to `'w+'` — this turned out to matter, not just for cleanliness. Files opened with `O_APPEND` (what `'a+'` maps to) ignore the explicit `position` argument to `write()` on Linux; the kernel forces every write to the current end-of-file regardless of what offset you pass. My rewrite to `handle.write(buffer, 0, length, position)` earlier in this fix would have silently degraded back into append-order-dependent writes under `'a+'`, defeating the whole point of writing to fixed offsets. `'w+'` doesn't set `O_APPEND`, so `position` is honored.

Also added an explicit assertion — `markReceived` now checks all `chunkIndex` values 0..N-1 are present before finalizing, rejecting the upload instead of silently finalizing a file with a missing middle chunk (previously possible if a chunk request failed and the client didn't retry it, since nothing checked for gaps before this).
