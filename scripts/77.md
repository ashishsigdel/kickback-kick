# Script 77 — 4xx spike right after the mobile client update

[[thinking]]
A 4xx spike coinciding with a mobile release is almost always a contract mismatch — the client started sending something the API doesn't expect, or the API expects something the old client sent and the new one dropped. Aggregate error rate won't tell me which; I need the breakdown by status code, route, and app version to know where to look first.

[[tool_use]]
name: Bash
input:
  command: curl -s "https://logs.internal/query" --data-urlencode 'q=status>=400 status<500 | stats count by route, status, client_version since -6h' | jq -r '.[] | "\(.route) \(.status) v\(.client_version) \(.count)"' | sort -k4 -rn | head -15
  description: Break down the 4xx spike by route, status, and app version

[[response]]
Almost entirely one route, one version, one status code:

```
route                    status  version  count
POST /v1/checkout/apply  422     4.9.0    38412
POST /v1/checkout/apply  422     4.8.3    112
GET  /v1/catalog         404     4.9.0    89
```

38k `422`s on `checkout/apply` from client `4.9.0`, essentially none from `4.8.3` (the previous version, still in the field for users who haven't updated). That's a contract break shipped in the 4.9.0 release, not a general capacity or auth issue.

Pulled a sample failing request body from the access log:

```json
{"cartId": "c_88f2", "promoCode": "", "paymentMethodId": "pm_af31"}
```

And the validator, `src/routes/checkout.js:58`:

```js
const applySchema = Joi.object({
  cartId: Joi.string().required(),
  promoCode: Joi.string().pattern(/^[A-Z0-9]{4,12}$/).allow(null),
  paymentMethodId: Joi.string().required(),
});
```

`.allow(null)` permits `promoCode: null` to skip the pattern check, but the client is sending `promoCode: ""` — an empty string, not `null` — when the user hasn't entered a promo code. Joi still runs the pattern check against `""`, which doesn't match, so validation fails with 422. This schema hasn't changed recently; 4.9.0 changed how the app serializes "no promo code entered."

[[thinking]]
Two ways to close this: relax the server schema to also allow empty string, or get the mobile team to send `null` instead of `""`. The server fix is safer to ship immediately — it fixes every already-installed 4.9.0 client without waiting on another app store review cycle, which takes days.

[[response]]
Fixed server-side since it's deployable now and a client-side fix would need another app store release to reach affected users:

```diff
-  promoCode: Joi.string().pattern(/^[A-Z0-9]{4,12}$/).allow(null),
+  promoCode: Joi.string().pattern(/^[A-Z0-9]{4,12}$/).allow(null, ''),
```

`.allow(null, '')` treats both as "no promo code," matching what 4.9.0 actually sends. Deployed; `422` rate on `checkout/apply` for `v4.9.0` dropped from ~6,400/hr to near-zero within the next polling window.

If 4.9.0 is sending `""` for those too, it would be silently accepted — no 422, no signal — where previously it may have sent `null` and hit different (or no) downstream logic. `promoCode` only surfaced because it has a strict format check; a looser field wouldn't show up in error logs at all.

Not fixing that speculatively — I don't have evidence 4.9.0 changed those fields, only that it changed `promoCode`. Flagging it for the mobile team to confirm whether the empty-string-for-absent change was scoped to one field or is a client-wide serialization change, since that determines whether this needs a broader server-side audit.
