[[response]]
Checked for shared module-level state before committing to that plan — this is where splits usually go wrong.

Three module-level objects:

```python
_stripe = StripeClient(settings.STRIPE_KEY)   # :24
_rate_limiter = TokenBucket(100, 60)          # :29
_TIER_CACHE: dict[int, Tier] = {}             # :33
```

`_stripe` and `_rate_limiter` are used by `charges` and `subscriptions`. `_TIER_CACHE` is used by `subscriptions` and `reporting`.

So a naive split creates a cycle. These need to move first:

```
services/billing/_clients.py   # _stripe, _rate_limiter
services/billing/_cache.py     # _TIER_CACHE
```

That becomes step 1, before the package scaffolding — everything else imports from those two and there are no cycles.

One caveat worth naming: `_TIER_CACHE` is currently unbounded and never invalidated. Moving it doesn't change that, but it does make it visible as a shared dependency of two modules, which is a good moment to decide whether it should have a TTL. I'd keep that as a separate change so the refactor stays behavior-preserving.
