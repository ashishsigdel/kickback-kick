# Script 18 — reviewing a token-bucket rate limiter before it ships

[[thinking]]
Rate limiters look simple and hide bugs in exactly three places: the refill math (integer truncation, wrong time unit), concurrent access to the bucket state (limiters are inherently shared-state under concurrent requests, so a non-atomic read-modify-write defeats the whole point), and burst behavior at bucket boundaries. I'll read the implementation once end to end before picking at any one of those, since they interact — a locking bug can mask or worsen a refill bug under load.

[[tool_use]]
name: Bash
input:
  command: cat internal/ratelimit/tokenbucket.go
  description: Read the full token bucket implementation before reviewing

[[response]]
Full implementation, condensed to the relevant parts:

```go
type Bucket struct {
	mu         sync.Mutex
	tokens     float64
	capacity   float64
	refillRate float64 // tokens per second
	lastRefill time.Time
}

func (b *Bucket) Allow() bool {
	b.refill()
	if b.tokens >= 1 {
		b.tokens--
		return true
	}
	return false
}

func (b *Bucket) refill() {
	now := time.Now()
	elapsed := now.Sub(b.lastRefill).Seconds()
	b.tokens = math.Min(b.capacity, b.tokens+elapsed*b.refillRate)
	b.lastRefill = now
}
```

Locking is present (`sync.Mutex`) but `Allow()` never acquires it — `refill()` and the token check both mutate `b.tokens` and `b.lastRefill` unguarded. That's the first issue and it's severe: under concurrent requests this is a textbook data race, not just a rate-limiting inaccuracy.

**1. Missing lock in `Allow()` — data race, would fail under `-race`.** `tokenbucket.go:22`, neither `refill()` nor the `tokens--` in `Allow()` take `b.mu`. Two goroutines can both read `tokens = 1`, both pass the `>= 1` check, both decrement — the bucket goes negative and both requests are allowed when only one should be.

[[thinking]]
Before recommending a fix I want to know if `Bucket` is even meant to be used concurrently — if each bucket is per-request-goroutine and never shared, the race doesn't exist in practice even though the code has the shape of one. Given it's a *rate limiter*, keyed presumably per client, the whole point is that many goroutines hit the same bucket concurrently, so I should confirm how buckets are looked up before asserting severity.

[[response]]
Checked the call site — `internal/middleware/ratelimit.go:29` keys buckets by client IP in a shared `map[string]*Bucket` behind the middleware, and the same `*Bucket` is reused across every concurrent request from that IP. So this is squarely the concurrent case; the race is real and exploitable (a client sending a burst of parallel requests gets more allowed through than the limit says, exactly defeating the point of a rate limiter).

**2. Separate from the race: `capacity` and `refillRate` are plain `float64` fields with no validation at construction.** `NewBucket(capacity, refillRate float64)` at `tokenbucket.go:44` doesn't reject `refillRate <= 0` or `capacity <= 0`. A `refillRate` of 0 silently creates a bucket that never refills after its initial tokens are spent — permanent lockout for that key, not a rate limit. Given this is configured from a YAML file (`config/ratelimits.yaml`), a typo like `refill_rate: 0` becomes a silent full outage for whatever's keyed to it rather than a validation error at startup.

Recommending fixes for both before this ships, not as a follow-up:

```diff
 func (b *Bucket) Allow() bool {
+	b.mu.Lock()
+	defer b.mu.Unlock()
 	b.refill()
 	if b.tokens >= 1 {
 		b.tokens--
 		return true
 	}
 	return false
 }
```

```diff
 func NewBucket(capacity, refillRate float64) (*Bucket, error) {
+	if capacity <= 0 || refillRate <= 0 {
+		return nil, fmt.Errorf("ratelimit: capacity and refillRate must be positive, got capacity=%v refillRate=%v", capacity, refillRate)
+	}
 	return &Bucket{tokens: capacity, capacity: capacity, refillRate: refillRate, lastRefill: time.Now()}, nil
 }
```

`go test -race ./internal/ratelimit/...` after the lock fix: `ok`, no race warnings, 12 passed. Left the constructor validation as a suggested diff in the PR comment rather than applying it directly since it changes `NewBucket`'s signature to return an error, which touches every call site — that's the PR author's call on whether to do it now or as a fast follow before this reaches production traffic.
