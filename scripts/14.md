# Script 14 — reviewing a PR that adds a caching layer to a hot endpoint

[[thinking]]
Caching PRs fail in predictable ways: stale reads after writes, cache stampede on expiry, or keys that don't account for every input that changes the response (pagination, tenant, locale). Before opining I want to see the actual key construction and invalidation path — that's where correctness lives, not the Redis client setup.

Diff adds caching around `CatalogService.getCatalog(tenantId, page)`:

```diff
+    @Autowired
+    private RedisTemplate<String, CatalogPage> redisTemplate;
+
     public CatalogPage getCatalog(String tenantId, int page) {
-        return catalogRepository.fetchPage(tenantId, page);
+        String key = "catalog:" + tenantId;
+        CatalogPage cached = redisTemplate.opsForValue().get(key);
+        if (cached != null) {
+            return cached;
+        }
+        CatalogPage result = catalogRepository.fetchPage(tenantId, page);
+        redisTemplate.opsForValue().set(key, result, Duration.ofMinutes(10));
+        return result;
+    }
+
+    @Transactional
+    public void updateProduct(String tenantId, Product product) {
+        catalogRepository.save(product);
+        redisTemplate.delete("catalog:" + tenantId);
     }
```

Two problems, one correctness and one operational.

**Correctness — cache key ignores `page`.** `CatalogService.java:34`: the key is `"catalog:" + tenantId`, but the cached value is `CatalogPage`, page-specific. Request page 1, cache stores page 1's data under `catalog:acme`. Request page 2 for the same tenant, key hits, and the caller gets page 1's contents back with a `200`. This isn't a rare edge case — any tenant with more than one page of products hits it on the very next request.

**Consistency gap — invalidation isn't transactional with the write.** `updateProduct` at `:47` deletes the cache key after `save()` inside the same `@Transactional` method, but Spring's default propagation commits the DB transaction *after* the method returns, while `redisTemplate.delete` runs synchronously inside it — so the delete can fire, then the DB commit itself could still fail and roll back, leaving the cache correctly empty but a fine window exists the other direction too: a concurrent reader between `save()` and commit visibility can repopulate the cache with pre-commit stale data if isolation level allows a read of the not-yet-committed... actually more directly: a concurrent read landing between the `delete` and the transaction's commit will find a cold cache, re-read from the DB (which may or may not yet reflect the write depending on isolation), and repopulate the cache with whatever it read.

[[thinking]]
The page-key bug is unambiguous and needs to block the PR. The invalidation race is real but subtler, and I don't want to overstate it without being precise about the actual window — I should state it as a narrow, plausible race rather than dress it up as certain corruption, since misjudging severity here would cost credibility on the parts that are certain.

[[response]]
Being precise about severity so the two issues don't get conflated:

**Key collision (`:34`) — blocking.** This isn't a race, it's deterministic wrong data on every multi-page tenant. Fix is to fold `page` into the key:

```diff
-        String key = "catalog:" + tenantId;
+        String key = "catalog:" + tenantId + ":page:" + page;
```

**Invalidation ordering (`:47`) — narrow race, worth fixing but not launch-blocking.** The window is: thread A calls `updateProduct`, saves, deletes the cache key, then before A's transaction commits, thread B calls `getCatalog`, misses the (now-cold) cache, and — depending on isolation level — may read pre-update data and cache it for another 10 minutes. At `READ_COMMITTED` (check `application.yml:18`, confirmed that's what's configured) B genuinely can read the pre-commit row, so this is live, not theoretical.

The standard fix is to invalidate in a post-commit hook rather than inline in the transactional method, so the delete only happens once the write is durably visible:

```java
@TransactionalEventListener(phase = TransactionPhase.AFTER_COMMIT)
public void onProductUpdated(ProductUpdatedEvent event) {
    redisTemplate.delete("catalog:" + event.tenantId() + ":page:*");
}
```

(Note the `:page:*` — with per-page keys, invalidation now needs to clear every page for the tenant, not one key. `SCAN` + delete, or a secondary index of active keys per tenant, rather than `KEYS *` in production.)

[[thinking]]
I've flagged a blocker and a real-but-narrow race. The PR author should also know the TTL choice (10 minutes) is a judgment call I can comment on but shouldn't unilaterally change — that's a product tradeoff about staleness tolerance, not a bug.
