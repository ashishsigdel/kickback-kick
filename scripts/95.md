# Script 95 — a state machine allows an invalid transition under a race

[[thinking]]
A state machine that's correct under single-threaded testing but wrong in production almost always means the guard check and the state write aren't atomic — something reads the current state, decides a transition is legal, and another request slips in between the read and the write. I want to see the transition function itself before assuming where the gap is, since it might be missing a lock entirely or have one that doesn't cover the right critical section.

[[response]]
`app/orders/state_machine.py:9`:

```python
ALLOWED_TRANSITIONS = {
    "pending":   {"paid", "cancelled"},
    "paid":      {"shipped", "refunded"},
    "shipped":   {"delivered", "returned"},
    "delivered": set(),
    "cancelled": set(),
    "refunded":  set(),
}

def transition(order_id, new_state):
    order = Order.objects.get(id=order_id)
    if new_state not in ALLOWED_TRANSITIONS[order.status]:
        raise InvalidTransition(order.status, new_state)
    order.status = new_state
    order.save()
```

No lock, no `select_for_update`, no optimistic version check. The read (`Order.objects.get`) and the write (`order.save()`) are two separate round-trips with application logic in between — classic read-check-write race. If `cancel` and `mark_paid` are both called for the same order within that window, both read `status="pending"`, both pass the `ALLOWED_TRANSITIONS` check, and whichever `save()` lands last wins — silently. The bug report matches exactly: an order ended up `"cancelled"` after a `"paid"` webhook had already fired and the customer had been charged, because the cancel request read the stale pre-payment status.

[[thinking]]
Two standard fixes: a DB row lock (`select_for_update`) held across the check-and-write, or an optimistic check that includes the expected prior state in the `UPDATE`'s `WHERE` clause and fails if zero rows matched. Locking is simpler to reason about but holds a lock for the duration of the transaction, which matters if `transition()` is ever called inside a longer-running transaction elsewhere. I want to check callers before picking.

[[response]]
Checked callers — `transition()` is called from four places (`webhooks/stripe.py`, `api/orders.py` twice, `tasks/shipping.py`), none of them wrap it in a surrounding transaction that does other slow work, so a row lock held for the duration of this one call is cheap. Went with `select_for_update` rather than optimistic retry, since it's a one-line change and this codebase doesn't have a retry-on-conflict convention elsewhere I'd want to introduce just for this:

`select_for_update()` takes a row lock on read; the second concurrent caller blocks until the first transaction commits, then re-reads the *now-current* status and correctly rejects the transition instead of racing past the check.

[[thinking]]
I should verify this actually closes the race rather than just looking correct, since lock scope bugs (locking the wrong row, or a lock that doesn't cover the actual write) are easy to get subtly wrong. A concurrency test that fires both transitions simultaneously and asserts exactly one succeeds is the only way to be confident, not just reading the diff.

[[response]]
Wrote a concurrency test — two threads call `transition(order_id, "paid")` and `transition(order_id, "cancelled")` simultaneously against the same `pending` order, 200 iterations with fresh orders each time:

```python
def test_concurrent_transition_only_one_wins():
    results = run_concurrently([
        lambda: transition(order.id, "paid"),
        lambda: transition(order.id, "cancelled"),
    ])
    succeeded = [r for r in results if not isinstance(r, InvalidTransition)]
    assert len(succeeded) == 1
```

Before the fix: 31/200 iterations had both transitions "succeed" (one silently overwriting the other, no exception raised by either). After the fix: 200/200 iterations have exactly one success and one `InvalidTransition`, matching the intended semantics — the loser is now rejected instead of silently overwritten.
