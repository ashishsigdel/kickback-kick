# Script 80 — metrics went dark for one service after a change

[[thinking]]
One service going silent in Prometheus while others keep reporting points at either a change to that service's `/metrics` endpoint (crash, panic, handler removed) or a change to how Prometheus discovers/scrapes it (relabeling, port, service discovery). The target status page will tell me which side of that line it's on faster than reading application code, so that's where I start.

[[tool_use]]
name: Bash
input:
  command: curl -s http://prometheus.internal:9090/api/v1/targets | jq -r '.data.activeTargets[] | select(.labels.job=="billing-svc") | "\(.health) \(.lastError) \(.scrapeUrl)"'
  description: Check Prometheus's own view of the target — is it being scraped at all, and what error if not

[[response]]
```
down failed: Get "http://10.2.4.18:9102/metrics": dial tcp 10.2.4.18:9102: connect: connection refused  http://10.2.4.18:9102/metrics
```

It's not a scrape config or discovery problem — Prometheus is finding the target fine, the target just isn't listening on 9102 anymore. Checked the recent diff on `billing-svc`:

```diff
-func main() {
-    go func() {
-        http.Handle("/metrics", promhttp.Handler())
-        log.Fatal(http.ListenAndServe(":9102", nil))
-    }()
-    startServer()
-}
+func main() {
+    r := chi.NewRouter()
+    r.Handle("/metrics", promhttp.Handler())
+    r.Mount("/api", apiRoutes())
+    startServer(r)
+}
```

The recent refactor moved off a standalone `net/http` mux on `:9102` onto the app's main `chi` router, which serves on `:8080` — but nothing in the diff updated the Prometheus scrape config, which still points at port `9102`. The metrics handler is registered and working; it's just listening on a port nothing scrapes anymore.

[[thinking]]
Before just repointing the scrape config to `:8080`, I want to check whether serving `/metrics` on the same port as the public API is intentional here or an accident of the refactor — some setups deliberately keep metrics on a separate unauthenticated internal port so `/metrics` isn't reachable through the public-facing listener/load balancer. If `:8080` is internet-facing, merging them could expose metrics externally.

[[response]]
Checked `startServer(r)` — `:8080` is bound behind the internal load balancer only; the public-facing listener is a separate reverse proxy that doesn't forward to this process at all (confirmed in `infra/lb/billing.tf:12`, no `:8080` target group attached to the public ALB). So `:8080` is already internal-only — merging `/metrics` onto it doesn't newly expose anything.

Given that, the simpler fix is repointing Prometheus rather than re-splitting the port in the service:

```diff
 - job_name: billing-svc
   static_configs:
     - targets:
-        - "10.2.4.18:9102"
+        - "10.2.4.18:8080"
```

Applied via the scrape-config repo, `prometheus/targets/billing-svc.yml:4`. Reloaded config (`curl -X POST prometheus.internal:9090/-/reload`), confirmed:

```
up  ""  http://10.2.4.18:8080/metrics
```

Target healthy, `billing_requests_total` series resumed at the current reload timestamp — there's a real gap in the time series for the outage window (about 6 hours), not backfillable since Prometheus doesn't have the samples, but everything from now forward is intact.

[[thinking]]
The immediate outage is fixed, but the underlying condition — a scrape target silently going stale with nothing watching for it — has no reason to be unique to `billing-svc`. Before treating this as a one-off port fix I want to know if any other service has the same drift between its actual listen port and its scrape config, since that's the same failure waiting to happen again somewhere else.

[[response]]
Checked all scrape targets against what each service actually listens on (`for t in $(yq '.[].targets[]' prometheus/targets/*.yml); do ...` cross-referenced against each service's `startServer` call). Two more mismatches:

```
job              scrape config    actual listen port
payments-svc     :9102            :9102   (match)
notifications    :9091            :8080   (MISMATCH — same refactor, three weeks ago)
search-svc       :9102            :9102   (match)
```

`notifications` has been silently dark for three weeks — same root cause, an earlier chi-router migration that moved `/metrics` without updating its scrape target. Nobody noticed because, same as billing, there's no alert on scrape health itself, only on the request/error-rate metrics that stopped updating along with everything else.

[[thinking]]
Two things now: fix the `notifications` scrape target the same way, and this is clearly not a one-off — it's happened twice from the same class of refactor, three weeks apart, with no detection either time. A per-service alerting rule is the right fix, not another manual repoint next time it happens.

[[response]]
Fixed `notifications`' scrape target (`prometheus/targets/notifications.yml:4`, `:9091` → `:8080`), confirmed `up` and metrics resumed the same way as billing.

Added a standard rule rather than a one-off, `prometheus/rules/scrape_health.yml`:

```yaml
- alert: TargetDown
  expr: up == 0
  for: 5m
  labels:
    severity: page
  annotations:
    summary: "{{ $labels.job }} scrape target down for 5m"
```

Applied globally across all jobs rather than per-service, so this pattern of "port changes, scrape config doesn't, nobody notices" alerts within 5 minutes instead of being found by whoever happens to check a dashboard. `promtool check rules prometheus/rules/scrape_health.yml` → `SUCCESS: 1 rules found`.

Not something I'm fixing here: neither refactor that caused this had a step for "did the scrape config move with the port," and that's a process gap, not a code one. Worth adding a line to the service migration checklist, or is the new alert enough of a safety net on its own?
