Read `billing/proration.py` and its three call sites. The public surface is small — one function does the real work:

```python
def prorate(plan, old_price, new_price, days_remaining, days_in_cycle):
```

The contract, as the callers actually depend on it:

1. Returns a **credit** (negative) when downgrading, a **charge** (positive) when upgrading.
2. Rounds to whole cents, **toward the customer** — a half-cent goes to the credit, not the charge.
3. Returns exactly `0` when `days_remaining == 0`, rather than a tiny float.

Point 2 is the interesting one. It's not written down anywhere; it's implied by `math.floor` at `proration.py:44` and it's the sort of thing a refactor would silently flip. That's the test worth having.

Before writing them I want to check the edge cases the callers can actually produce, so I'm not testing inputs that can't occur.

[[thinking]]
The mid-cycle plan change is where the rounding rule bites, and a same-day change is where the zero case comes from. If days_in_cycle can be zero anywhere I need a test for the division, but if the callers guarantee it's positive then testing it invents a contract that doesn't exist.

[[response]]
`days_in_cycle` is always positive — it comes from the plan definition, and all three call sites read it from there. So no division-by-zero test; it would be asserting on an impossible input.

Written to `tests/billing/test_proration.py`:

```python
def test_downgrade_returns_credit():
    assert prorate(PLAN, 5000, 2000, 15, 30) == -1500

def test_upgrade_returns_charge():
    assert prorate(PLAN, 2000, 5000, 15, 30) == 1500

def test_half_cent_rounds_toward_the_customer():
    # 1 day of a 3-day cycle: exactly 0.5c. Must credit, not charge.
    assert prorate(PLAN, 100, 99, 1, 3) == -1

def test_no_days_remaining_is_exactly_zero():
    result = prorate(PLAN, 5000, 2000, 0, 30)
    assert result == 0 and isinstance(result, int)
```

The third one is the test that earns its place — it pins the rounding direction that nothing else records.
