[[response]]
The diff adds `MaxReceiveCount`-based redrive via the queue's native redrive policy, which is the right mechanism — SQS handles the count and move automatically, no app code needed for the counting itself:

```go
redrivePolicy := &sqs.RedrivePolicy{
    DeadLetterTargetArn: dlqArn,
    MaxReceiveCount:     5,
}
```

That's correctly attached to the source queue (`infra/sqs.tf:34`), so any message that fails processing 5 times gets moved to the DLQ by SQS itself, independent of application code. Good — that's the part that's hard to get subtly wrong.

The consumer side, `internal/queue/consumer.go:71`, is where I'd expect the bug if there is one:

```go
func (c *Consumer) handle(msg *sqs.Message) {
    defer c.deleteMessage(msg)
    if err := c.process(msg); err != nil {
        log.Printf("processing failed: %v", err)
        return
    }
}
```

Found it: `defer c.deleteMessage(msg)` runs on *every* return path, including the error path. A message that fails processing gets deleted from the queue immediately after the failed attempt, regardless of whether `process` returned an error. It never gets a second delivery, so `MaxReceiveCount` never has a chance to count to 5 — the message is gone after receive #1, success or failure. The DLQ redrive policy is correctly configured and will never fire, because nothing ever reaches a second attempt.

[[thinking]]
This means every message that ever failed processing before this diff was silently deleted with no DLQ, no redrive, no record — which is worse than what the diff is trying to fix, not a partial improvement on it. I want to confirm this is pre-existing behavior and not something introduced by this diff, since that changes how urgently it needs fixing versus how it gets communicated.

[[response]]
Checked blame on that line — `defer c.deleteMessage(msg)` predates this diff by eight months (`git log -1 --format=%H -L71,71:internal/queue/consumer.go`). This diff added the DLQ infrastructure correctly but didn't touch the one line that makes it unreachable. Every processing failure for the last eight months has deleted the message outright with no DLQ, no retry, no trace beyond that one log line — this diff, as written, doesn't change that at all despite adding DLQ config that looks like it should.

Fix is to only delete on success, and let SQS's own visibility timeout handle re-delivery on failure — that's what lets `MaxReceiveCount` actually count:

```diff
 func (c *Consumer) handle(msg *sqs.Message) {
-    defer c.deleteMessage(msg)
     if err := c.process(msg); err != nil {
         log.Printf("processing failed: %v", err)
         return
     }
+    c.deleteMessage(msg)
 }
```

[[thinking]]
Not deleting on failure means the message stays invisible only until the visibility timeout expires, then comes back for redelivery — but if visibility timeout is shorter than `process()` typically takes, a still-in-flight message could become visible again and get picked up by a second consumer while the first is still working it, which is a different bug than the one I'm fixing. Worth checking that relationship before calling this safe.

[[response]]
Checked — visibility timeout on the source queue is 30s (`infra/sqs.tf:19`), and `process()`'s own p99 from existing metrics is ~800ms, so there's over 35x headroom; a stuck-in-flight double-delivery isn't a realistic risk at current processing times. It would become one if a future change made `process()` slow or blocking (e.g. a synchronous call to a slow downstream), so I added a comment at `consumer.go:68` noting the visibility-timeout-vs-processing-time relationship rather than hardcoding a defensive check that has nothing to actually verify against right now.

[[thinking]]
Removing the unconditional delete means a message that fails will now sit visible again after the visibility timeout expires and get redelivered — but if `process` isn't idempotent, redelivery itself is a new correctness risk this diff introduces where there wasn't one before (because previously it just silently vanished). Need to check that before calling this safe to ship.

[[response]]
Checked `process()` — it's a Stripe refund handler keyed on an idempotency key derived from the message body (`internal/payments/refund.go:22`, `stripe.RefundParams{IdempotencyKey: msg.Body.EventID}`), so redelivery calling `process` again on the same message is safe: Stripe dedupes on that key, it won't double-refund. Confirmed this is true for the other two consumers using this same `handle` path — `internal/payments/payout.go` and `internal/notifications/email.go` — both also key off a stable ID from the message body, not a freshly generated one per attempt. If any consumer generated a new idempotency key per attempt instead of deriving it from message content, this fix would need to land alongside an idempotency fix, not before it.
