# Script 36 — reviewing the bcrypt-to-argon2 migration

[[thinking]]
Password hashing migrations fail in one of two ways: the parameters are wrong (too weak, defeats the purpose of migrating) or the transition logic is wrong (existing bcrypt hashes get orphaned, locking out every current user on the next deploy). The second is the one that causes an incident, so I want to see how existing hashes are handled before I even look at the argon2 parameters themselves.

This is a hard cutover, not a migration. `verify_password` only knows how to check argon2 hashes — every existing user's `stored_hash` in the database is still a bcrypt string (`$2b$...`), and `ph.verify` will raise `InvalidHash` (uncaught here — only `VerifyMismatchError` is caught) the first time any current user tries to log in after this deploys. That's every existing account locked out simultaneously, not a gradual migration.

[[thinking]]
The standard pattern here is dual verification: try the new scheme, fall back to the old scheme's format detected by hash prefix, and if the old scheme succeeds, transparently re-hash with the new scheme so the migration completes itself over time as users log in. I want to check whether `PasswordHasher()`'s default parameters are even reasonable before writing that up, since there's no point recommending a migration path onto weak parameters.

[[response]]
Checked the argon2 defaults being used — `PasswordHasher()` with no arguments uses `argon2-cffi`'s library defaults: `time_cost=3`, `memory_cost=65536` (64MB), `parallelism=4`, type `argon2id`. Those are reasonable OWASP-baseline numbers for id-type argon2, so the parameters aren't the problem — the transition is. Two things to fix:

**1. Dual verification with transparent re-hash on login**, so existing bcrypt users aren't locked out and migrate opportunistically:

```python
import bcrypt
from argon2 import PasswordHasher
from argon2.exceptions import VerifyMismatchError, InvalidHash

ph = PasswordHasher()

def hash_password(password: str) -> str:
    return ph.hash(password)  # all new/changed passwords get argon2 immediately

def verify_password(password: str, stored_hash: str) -> tuple[bool, str | None]:
    """Returns (is_valid, new_hash_to_persist_or_None)."""
    if stored_hash.startswith("$argon2"):
        try:
            return ph.verify(stored_hash, password), None
        except (VerifyMismatchError, InvalidHash):
            return False, None

    # legacy bcrypt path
    if bcrypt.checkpw(password.encode(), stored_hash.encode()):
        return True, ph.hash(password)  # caller persists this, completing the migration for this user
    return False, None
```

**2. Bare `except` gap** — original diff only caught `VerifyMismatchError`; a malformed or truncated hash raises `InvalidHash` instead, which was uncaught and would 500 rather than fail the login cleanly. Fixed by catching both.

Checked the deployment — `auth-service` runs on 512MB containers, and login endpoints are rate-limited to 20 req/s per pod upstream. Worst case concurrent hashing load is bounded well under the container's memory budget (20 × 64MB would be too much if truly simultaneous, but the rate limit is per-pod sustained, not a burst allowance, and `ph.hash`/`ph.verify` calls are fast enough — ~40ms each locally — that concurrent overlap in practice is small). Flagging it rather than resizing anything: if login traffic patterns change (a spike, a credential-stuffing attempt hitting this endpoint hard) this is the parameter that would need tuning down, or the container that would need to size up. Not a blocker for merging the migration, just something to keep in view.
