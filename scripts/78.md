The migration (`db/migrations/0047_add_tenant_rls.sql`) looks right on the policy side:

```sql
ALTER TABLE invoices ENABLE ROW LEVEL SECURITY;
CREATE POLICY tenant_isolation ON invoices
  USING (tenant_id = current_setting('app.tenant_id')::uuid);
```

Applied consistently across `invoices`, `line_items`, `payments` — all keyed off the same `app.tenant_id` session variable, `USING` clause covers both reads and, since no separate `FOR SELECT`, also writes via the default `ALL` scope. That part's solid.

The setup side, `app/db.py:41`:

```python
def get_connection(tenant_id):
    conn = pool.getconn()
    conn.cursor().execute("SET app.tenant_id = %s", (str(tenant_id),))
    return conn
```

`SET` (not `SET LOCAL`) is the issue. `SET` persists for the life of the session, and this pool reuses connections across requests. `SET LOCAL` would reset at transaction end; plain `SET` means if `get_connection` is ever called without immediately being followed by a query, or if a connection is returned to the pool without an explicit reset, the *next* request to borrow that connection inherits the *previous* tenant's `app.tenant_id` until something overwrites it.

[[thinking]]
Whether this is exploitable depends entirely on whether every code path that returns a connection to the pool also resets it, or whether `get_connection` is reliably called fresh per-request in a way that reordering can't break. I need to check the pool's checkin/checkout behavior, not assume — this is the difference between "theoretically fragile" and "actively leaking tenants."

Connections go back to the pool still carrying whatever `app.tenant_id` was last set. The gap is narrower but real: three call sites (`app/jobs/webhook_retry.py:18`, `app/admin/console.py:9`, `app/cli/backfill.py:31`) call `pool.getconn()` directly, bypassing `get_connection()` entirely, never calling `SET app.tenant_id` at all. On a fresh connection that's a hard failure (`current_setting` raises if unset) — safe by accident. On a *recycled* connection, `current_setting('app.tenant_id')` silently returns whatever tenant used that connection last, and the query runs scoped to the wrong tenant with no error.

The real fix is two layers: stop connections from leaving the pool with stale state at all, and make the "no tenant context set" case fail loudly instead of raising only sometimes depending on connection freshness. Belt and suspenders — either one alone still leaves a way to get this wrong.

 def get_connection(tenant_id):
     conn = pool.getconn()
-    conn.cursor().execute("SET app.tenant_id = %s", (str(tenant_id),))

`SET LOCAL` scopes it to the current transaction, so it can't leak across a pooled connection's next use even if a checkin path forgets to clean up.

And, belt-and-suspenders: added `DISCARD ALL` in `app/middleware.py`'s connection-return path, so anything set on the connection — not just tenant context — is wiped before it re-enters the pool.
