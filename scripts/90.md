[[response]]
Before: `POST /reports/generate` built a PDF synchronously and returned it, routinely taking 8-14s and holding a gunicorn worker the whole time. After: the endpoint enqueues `generate_report.delay(report_id)` and returns `202 {"status": "queued", "report_id": ...}` immediately. Reasonable shape overall. Two problems.

**1. Task isn't idempotent — `tasks/report_tasks.py:18`**

```python
@app.task(bind=True, max_retries=3)
def generate_report(self, report_id):
    report = Report.objects.get(id=report_id)
    pdf = build_pdf(report)
    report.file = save_pdf(pdf)
    report.status = "complete"
    report.save()
    send_email(report.owner.email, report.file.url)
```

On a transient failure (S3 timeout in `save_pdf`, DB blip) Celery retries the whole task. If `save_pdf` succeeded but `report.save()` raised, the retry regenerates the PDF *and* re-sends the completion email — a user who already got one email can get two or three depending on how many retries fired before something stuck.

**2. No result the client can poll reliably — `api/views/reports.py:44`**

The 202 response includes `report_id` but there's no endpoint to check task state, only `GET /reports/:id` which returns the `Report` row directly. Client has to poll and infer completion from `status == "complete"`, but there's no way to distinguish "still processing" from "failed after 3 retries" — a permanently failed task looks identical to one still in progress, forever.

[[thinking]]
The email duplication is the more urgent of the two — a user-visible bug that fires under normal transient failures, not just an edge case. Before I recommend a fix I want to check whether idempotency needs to be enforced task-wide (a dedup key) or whether it's enough to make each side effect individually safe to repeat, since those are different amounts of work.

[[response]]
Making each step individually idempotent is enough here and is less invasive than a task-level dedup key:

```diff
 def generate_report(self, report_id):
     report = Report.objects.get(id=report_id)
+    if report.status == "complete":
+        return  # already done by a prior attempt, don't redo side effects
     pdf = build_pdf(report)
     report.file = save_pdf(pdf)
-    report.status = "complete"
-    report.save()
-    send_email(report.owner.email, report.file.url)
+    report.status = "complete"
+    report.save()
+    if not report.email_sent_at:
+        send_email(report.owner.email, report.file.url)
+        report.email_sent_at = timezone.now()
+        report.save(update_fields=["email_sent_at"])
```

The early-return on `status == "complete"` handles the case where `save_pdf` succeeded but the row update failed before retry; the `email_sent_at` guard handles the case where everything succeeded but the retry still fired due to a broker redelivery. Both are cheap checks against fields that already exist or are trivial to add via migration.
