Returning a raw `error` instead of a `status.Error(codes.X, ...)` means gRPC wraps it as `codes.Unknown` on the wire, regardless of what actually went wrong. The old REST handler at `api/handlers/inventory.go:38` (still present, not yet deleted) had explicit mapping — `sql.ErrNoRows` → 404, validation errors → 400, everything else → 500. That distinction is gone in the gRPC path; every failure looks identical to `codes.Unknown` to a client, which means client-side retry logic (which typically retries `Unavailable`/`DeadlineExceeded` but not `NotFound` or `InvalidArgument`) can't tell a "sku doesn't exist" from a "database is down."

Also checked deadline propagation — `s.store.Lookup(ctx, ...)` does pass `ctx` through, so if a client sets a gRPC deadline it does reach the DB call via context cancellation. That part's correct.

```diff
 func (s *InventoryServer) GetStock(ctx context.Context, req *pb.GetStockRequest) (*pb.GetStockResponse, error) {
     item, err := s.store.Lookup(ctx, req.Sku)
     if err != nil {
+        if errors.Is(err, sql.ErrNoRows) {
+            return nil, status.Errorf(codes.NotFound, "sku %q not found", req.Sku)
+        }
+        if errors.Is(err, context.DeadlineExceeded) {
+            return nil, status.Errorf(codes.DeadlineExceeded, "lookup timed out")
+        }
+        return nil, status.Errorf(codes.Internal, "lookup failed: %v", err)
     }
     return &pb.GetStockResponse{Quantity: item.Qty}, nil
 }
```

Also flagging a schema issue while I'm in the `.proto`: `GetStockRequest.sku` is `string`, unversioned, no validation annotation, and `inventory.proto:12` has no reserved field numbers for the fields removed from the original REST payload (`warehouse_id` was dropped, not deprecated). In proto3, reusing a field number later for a different type is a wire-compatibility break that won't error at compile time — it'll just silently misread old messages if anything still encodes with the old schema.
