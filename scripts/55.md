# Script 55 — a segfault traced to a native extension

[[thinking]]
Python itself doesn't segfault — a segfault always means either a C extension, a buggy ctypes/cffi call, or a bug in the interpreter itself (rare, and unlikely to be the first suspect). Since the crash is a hard segfault rather than a Python exception, `pdb` and tracebacks are useless; I need a core dump and a native-level backtrace via `gdb`, not a Python-level stack trace, to see which C frame actually faulted.

Not a normal call depth for a spatial index — a balanced R-tree over 10M points should be ~20-25 levels deep, not 400+. This is unbounded recursion blowing the C stack, manifesting as `SIGSEGV` rather than Python's own catchable recursion limit, since C extension frames don't go through `sys.setrecursionlimit` at all.

[[thinking]]
400+ levels for a structure that should be ~20 means either the tree isn't actually balancing (every insert going down one side) or there's a bug in the split condition that fails to fire, so nodes never split and the "depth" argument keeps incrementing on what should have terminated already. I need `geoindex/_rtree.c:184` and the split logic around it, not just the crash frame, to see which.

[[response]]
`geoindex/_rtree.c:178`:

```c
static PyObject* RTreeNode_insert(RTreeNode *node, Point p, int depth) {
    if (node->count < MAX_ENTRIES) {
        node->entries[node->count++] = p;
        Py_RETURN_NONE;
    }
    RTreeNode *child = choose_subtree(node, p);
    return RTreeNode_insert(child, p, depth + 1);   // line 184
}
```

`choose_subtree` picks the child whose bounding box needs least enlargement to contain `p` — standard R-tree logic. `geoindex/_rtree.c:152`:

```c
if (enlargement < best_enlargement) {
    best_enlargement = enlargement;
    best_child = child;
}
```

Strict `<`, no tiebreaker, `best_child` initialized to `node->children[0]`. Under a tie it keeps the first child — fine on its own. The real bug: when a node's points repeatedly route to the same child via this tiebreak (a cluster of near-duplicate coordinates, common in the 10M dataset from repeated sensor readings), that child never triggers a split. The overflow check lives in `RTreeNode_split`, called from `RTreeNode_insert`'s caller — but the recursive call at line 184 calls `RTreeNode_insert` directly, bypassing that wrapper entirely.

[[thinking]]
I should verify this theory rather than assert it, since C-level bugs are easy to misdiagnose from reading alone — the cleanest verification is a small reproduction with a synthetic dataset of exact duplicate points, small enough to run under a debugger interactively without a 10M-point crash, and confirm the recursion depth grows unboundedly with duplicate count rather than plateauing.

[[response]]
Reproduced in isolation with 50,000 exact-duplicate points inserted into a fresh tree — under `gdb` with a breakpoint on `RTreeNode_insert`, watched `depth` climb past 50,000 with `node->count` never exceeding `MAX_ENTRIES` at the top level, confirming the split path never fires when one child always wins the tiebreak. Near-duplicate GPS points at real-world precision limits produce the same tie condition often enough to hit this at 10M points, not just literal duplicates.

Fix is to trigger the split check inside `RTreeNode_insert` itself right after the recursive call returns, rather than relying on the caller to have done it — and add a hard depth cap that raises a Python exception instead of recursing unboundedly, so a future edge case degrades to a catchable error instead of a segfault:

```diff
 static PyObject* RTreeNode_insert(RTreeNode *node, Point p, int depth) {
+    if (depth > MAX_TREE_DEPTH) {
+        PyErr_SetString(PyExc_RuntimeError, "R-tree exceeded maximum depth; degenerate input?");
+        return NULL;
+    }
     if (node->count < MAX_ENTRIES) {
         node->entries[node->count++] = p;
         Py_RETURN_NONE;
     }
     RTreeNode *child = choose_subtree(node, p);
-    return RTreeNode_insert(child, p, depth + 1);
+    PyObject *result = RTreeNode_insert(child, p, depth + 1);
+    if (result && child->count >= MAX_ENTRIES) {
+        RTreeNode_split(child);
+    }
+    return result;
 }
```

`MAX_TREE_DEPTH` set to 64 — generous for any realistically balanced tree, well below the ~8,000-frame point where the C stack actually faults on this platform, so the Python-catchable error fires long before a segfault could.

Rebuilt the extension (`python setup.py build_ext --inplace`) and re-ran both the 50,000-duplicate reproduction and the original 10M-point dataset:
