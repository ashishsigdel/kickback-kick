# Script 25 — race in a distributed lock letting two workers hold it at once

[[thinking]]
Two workers holding what's supposed to be a mutually exclusive lock at the same time, in a Redis-based lock, has a short list of usual suspects: the lock's TTL expiring while the holder is still working (GC pause, slow I/O) and a second worker acquiring it before the first releases, or the release logic deleting a key that isn't even the current holder's (deleting by key name alone, without checking ownership, lets worker A delete worker B's lock after A's own TTL already expired). The bug report says both workers logged "acquired lock" for the same key within the same processing window — I want the actual lock module before guessing which.

[[response]]
`app/locks/redlock.py:18`:

```python
def acquire(self, key: str, ttl_ms: int = 10_000) -> bool:
    return self.redis.set(key, self.token, nx=True, px=ttl_ms)

def release(self, key: str):
    if self.redis.get(key) == self.token:
        self.redis.delete(key)
```

Acquire is correct — `SET key value NX PX ttl` is atomic, single Redis command, fine. The bug is in `release`: the `GET` and the `DELETE` are two separate round trips, not one atomic operation. Between the `GET` returning a match and the `DELETE` executing, the TTL can expire and a second worker can acquire the same key — worker A's `DELETE` then fires and removes worker B's lock, not its own. That's the textbook Redlock release bug, and it's the one the Redlock spec explicitly calls out (release must be a Lua script, not GET-then-DEL) — this implementation has the acquire half right and skipped the release half.

Confirmed against incident logs: worker A acquired `lock:batch-4471` at `14:02:00.100`, TTL 10s, should expire `14:02:10.100`. A's job ran long — 11.2s, finishing at `14:02:11.300`, past expiry. A `MONITOR` capture during a forced repro shows the key re-acquired by worker B at `14:02:10.104`, 4ms after natural expiry. A's `release()` then runs at `14:02:11.300`.

[[thinking]]
If A's GET correctly reads B's token, the `== self.token` check should fail and A shouldn't delete anything — so before assuming this is a pure timing race, I should check that the token comparison actually protects what it claims to, since a token that isn't unique per acquisition would be a worse bug than the GET/DELETE gap alone.

It's worse than a timing gap — `self.token` is generated **once per `RedlockClient` instance**, not once per `acquire()` call. `redlock.py:9`:


`app/workers/batch_processor.py:22` constructs one `RedlockClient` at worker startup and reuses it across every job that process handles. So the token comparison in `release()` provides no protection against the TTL-expiry-then-reacquire race at all when the *same* worker process (same static token) re-acquires a key later — and worse, it means acquire-time identity and release-time identity, which the Redlock pattern depends on being tied to one specific lock attempt, are actually tied to the whole process lifetime instead.

[[thinking]]
Two independent bugs, and I should fix both rather than the one that happens to explain today's incident, since the token-scoping bug is latent and would eventually cause a worse version of the same failure even after the release-atomicity fix. Order matters: fix the GET-then-DEL race with a Lua script first since that's the standard, well-known correct pattern, then fix token scoping since the atomic release only means anything if the token is actually unique per acquisition.
