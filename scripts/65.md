Top error by a wide margin, starting within 2 minutes of the `new-checkout-summary` flag going to 25% rollout at 13:04:

```json
{"message": "Cannot read properties of undefined (reading 'discountLabel')", "count": 4128}
```

Points at `src/components/CheckoutSummary.tsx:33`:

```tsx
function CheckoutSummary({ cart }: Props) {
  const promo = useFeatureFlag('new-checkout-summary')
    ? cart.activePromotion
    : null;

  return (
    <div>
      {promo && <span>{promo.discountLabel}</span>}
    </div>
  );
}
```

`promo` is only truthy-checked, not shape-checked — if `cart.activePromotion` exists but doesn't have `discountLabel`, this still crashes on render since `promo.discountLabel` is accessed unconditionally once `promo &&` passes. So the real question is what shape `activePromotion` has for the affected users.

[[thinking]]
Since this is a percentage rollout, the affected population is randomly assigned — the bug should hit users at roughly the flag's rollout rate if it's purely a client-side rendering bug independent of user state. If instead it correlates with a particular user segment (e.g. users with an expired promo, or a specific cart type), that points at a data-shape assumption baked into the new component rather than a pure rendering bug, and the fix is different.

[[response]]
Checked the correlation: of the 25% of sessions in the flag's `treatment` bucket, error rate is 34% — not close to 100%, which rules out "every treatment user crashes" and rules in "a subset of `activePromotion` shapes are bad." Cross-referenced with cart data: every crashing session has a cart with an *expired* promotion still attached.

`cart.activePromotion` is set by the backend at `api/cart/serializer.py:41` and isn't cleared when a promotion expires — it's left on the cart with `is_active: false` but the object itself, including `discount_label` (note: snake_case from the API), stays present. The frontend's `activePromotion` type declares `discountLabel` (camelCase) assuming an API client transform that, it turns out, doesn't run on this particular endpoint — `src/api/transforms.ts:8` lists the endpoints it camelCases, and `/cart` isn't one of them.

So `promo.discountLabel` is `undefined` whenever the cart carries an expired promo, which is common — abandoned-cart users with an expired discount code are exactly the population likely to still have `activePromotion` set.

[[thinking]]
Before concluding this is purely a data-shape bug, I want to rule out the flag evaluation itself — if `useFeatureFlag` assigned users inconsistently across renders, that would produce a more chaotic error signature than what's here, so worth a quick sanity check before I commit to the data-shape explanation.

[[response]]
Checked `useFeatureFlag` at `src/hooks/useFeatureFlag.ts:12` — it memoizes the assignment per session ID against a stable bucketing hash, not re-evaluated per render. Confirmed via the error tracker's session replay that affected sessions stay in the `treatment` bucket throughout, no flapping. Flag evaluation is solid; the bug is entirely in what the component does once inside the treatment branch.

[[thinking]]
There are two separate bugs stacked here — the missing camelCase transform, and the component not checking `is_active` before rendering a promo at all — and fixing only one leaves a real defect in place even though it'd stop this specific crash. I want to fix both but be clear they're different fixes for different reasons.

[[response]]
Fixed both.

The immediate crash — component was never checking whether the promotion was still active, which is a bug independent of the naming mismatch:

```diff
-const promo = useFeatureFlag('new-checkout-summary')
-  ? cart.activePromotion
-  : null;
+const promo = useFeatureFlag('new-checkout-summary') && cart.activePromotion?.isActive
+  ? cart.activePromotion
+  : null;
```

The underlying data contract gap — added `/cart` to the transform list so the API client stops silently passing through untransformed keys:

```diff
 const CAMEL_CASE_ENDPOINTS = [
   '/orders',
   '/products',
+  '/cart',
 ];
```

`npm test -- CheckoutSummary` → `18 passed`, including a new case with an expired-promo cart. Deployed behind the same flag, held rollout at 25% for 30 minutes: error rate for `Cannot read properties of undefined` dropped to 0 in that window, versus 34% of treatment sessions before.

One thing I didn't chase down: `/cart` not being in `CAMEL_CASE_ENDPOINTS` means any other component reading raw `/cart` data may have the same snake/camel mismatch masked by optional chaining elsewhere. Worth a broader audit of `/cart` consumers before the next flag ships against this endpoint — want me to do that pass?
