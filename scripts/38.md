CI, from the workflow file: no `TZ` set either, but the runner image (`ubuntu-latest`) defaults to UTC.

That's the split. `tests/billing/invoice.test.ts:22` is the failing test:

```ts
it("buckets the invoice into the correct billing month", () => {
  const invoice = createInvoice({ issuedAt: new Date("2026-07-31T22:00:00Z") });
  expect(invoice.billingMonth).toBe("2026-07");
});
```

`2026-07-31T22:00:00Z` is July 31st in UTC. In `America/Kathmandu` (UTC+5:45) that same instant is already `2026-08-01T03:45`, and in UTC it's still July 31st. If `billingMonth` is derived using local-time methods, the two environments compute different months for the identical timestamp.

[[thinking]]
Before I fix the test I want to know whether the bug is in the test's assumption or in `billingMonth`'s own implementation — if the production code itself uses local time to bucket invoices, that's a real bug affecting real invoices depending on server timezone, not just a test artifact. Fixing only the test while leaving a timezone-dependent production function would hide a worse problem.

[[response]]
It's in production code, not just the test. `src/billing/invoice.ts:41`:

```ts
function billingMonth(issuedAt: Date): string {
  const y = issuedAt.getFullYear();
  const m = issuedAt.getMonth() + 1;
  return `${y}-${String(m).padStart(2, "0")}`;
}
```

`getFullYear()`/`getMonth()` read in the host's local timezone. In production this runs on servers set to UTC, so it happens to be consistent there — but it means the *behavior itself* is timezone-dependent, and the only reason it hasn't caused a real invoice-bucketing bug is that prod and CI (as of whenever this test was added) both defaulted to UTC and nobody ran it from a UTC+5:45 laptop until now. The test isn't catching an environment quirk, it's catching a latent correctness bug that got lucky.

[[thinking]]
The fix is to make billing month derivation explicitly UTC-based rather than implicitly relying on the host's TZ, so behavior is identical everywhere regardless of where it runs — including any future move to a different server timezone or a different CI image.

[[response]]
```diff
 function billingMonth(issuedAt: Date): string {
-  const y = issuedAt.getFullYear();
-  const m = issuedAt.getMonth() + 1;
+  const y = issuedAt.getUTCFullYear();
+  const m = issuedAt.getUTCMonth() + 1;
   return `${y}-${String(m).padStart(2, "0")}`;
 }
```

Also pinned `TZ=UTC` explicitly in `.github/workflows/test.yml`'s test job and in the local `npm test` script (`"test": "TZ=UTC jest"`), even with the code fix — belt and suspenders, so a future date-handling regression fails the same way everywhere instead of depending on whichever machine happens to run it.
