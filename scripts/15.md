# Script 15 — migrating a large table's column type without downtime

[[thinking]]
Changing a column type on a table with real row counts is one of the few Rails migrations that can take an exclusive lock for the whole rewrite — `change_column` on Postgres rewrites the table and holds `ACCESS EXCLUSIVE` for the duration, blocking reads and writes. Before writing the migration I need the row count and whether this table is on the hot write path; those two facts decide whether a naive `change_column` is even survivable in a maintenance window versus needing the expand/contract dance.

[[tool_use]]
name: Bash
input:
  command: psql "$DATABASE_URL" -c "SELECT relname, n_live_tup FROM pg_stat_user_tables WHERE relname = 'orders';" -c "\d orders" | head -30
  description: Check row count and current schema for the orders table

[[response]]
`orders` has 41.2M live rows, and `external_ref` is `character varying(32)` — the column that needs to become `bigint` because the payment provider switched their reference format and the new IDs no longer fit the varchar-as-string comparisons cleanly downstream (app team wants numeric comparisons and indexing).

This table is also on the write path — `app/models/order.rb` shows `after_create` and `after_update` callbacks touching it, and staging logs show ~40 writes/sec at peak. A plain `change_column :orders, :external_ref, :bigint` in Rails compiles to Postgres `ALTER TABLE orders ALTER COLUMN external_ref TYPE bigint`, which rewrites all 41M rows under `ACCESS EXCLUSIVE` — every read and write blocks for the duration. At this table's size that's realistically 5-15 minutes of a fully down `orders` table, which is not an acceptable maintenance window here.

Going with the standard expand/contract pattern instead: add the new column, dual-write, backfill in batches, cut reads over, drop the old column in a later deploy.

[[thinking]]
The trap in expand/contract is doing it in one migration file with one deploy — that reintroduces the same downtime risk because Rails runs the whole migration in a transaction by default (for anything except concurrent index creation) and a giant `UPDATE ... WHERE` backfill in one shot still locks aggressively and bloats the WAL. I need to split this into separate migrations across separate deploys, and the backfill has to run in small batches outside the schema-change transaction entirely.

[[response]]
Four migrations, four deploys, in this order:

**1. Add the new column, nullable, no default (instant — metadata only):**
```ruby
class AddExternalRefBigintToOrders < ActiveRecord::Migration[7.1]
  def change
    add_column :orders, :external_ref_bigint, :bigint
  end
end
```

**2. Deploy app code that dual-writes** — every `Order` create/update sets both `external_ref` and `external_ref_bigint`:
```ruby
before_save do
  self.external_ref_bigint = external_ref.to_i if external_ref.present?
end
```

**3. Backfill existing rows in batches, outside a single transaction:**
```ruby
class BackfillExternalRefBigint < ActiveRecord::Migration[7.1]
  disable_ddl_transaction!

  def up
    Order.unscoped.in_batches(of: 10_000) do |batch|
      batch.where(external_ref_bigint: nil)
           .update_all("external_ref_bigint = external_ref::bigint")
      sleep(0.05) # throttle replication lag on the read replica
    end
  end
end
```
`disable_ddl_transaction!` matters — without it Rails wraps the whole migration in one transaction and every batch's lock is held until the last one commits, which is the exact problem this pattern exists to avoid.

**4. After backfill completes and is verified (`Order.where(external_ref_bigint: nil).count == 0`), add a `NOT NULL` constraint the safe way:**
```ruby
class AddNotNullToExternalRefBigint < ActiveRecord::Migration[7.1]
  def up
    safety_assured do
      execute "ALTER TABLE orders ADD CONSTRAINT external_ref_bigint_not_null CHECK (external_ref_bigint IS NOT NULL) NOT VALID"
      execute "ALTER TABLE orders VALIDATE CONSTRAINT external_ref_bigint_not_null"
    end
  end
end
```
`NOT VALID` first, then `VALIDATE CONSTRAINT` as a separate statement — the `NOT VALID` add takes a brief `ACCESS EXCLUSIVE` but only to write metadata, not to scan; `VALIDATE` does the actual full-table scan but only needs `SHARE UPDATE EXCLUSIVE`, which doesn't block writes.
