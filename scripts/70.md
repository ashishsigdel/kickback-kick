# Script 70 — reviewing a diff adding read replicas and routing reads to them

[[thinking]]
The single hardest thing to get right in a read-replica routing diff is replication lag — any read that must see its own immediately-preceding write is a correctness bug waiting to happen if it gets routed to a replica. I want to find every place the diff routes reads to the replica pool and cross-check each against whether it follows a write in the same logical operation, before looking at the routing mechanism itself.

[[response]]
Routing mechanism, `app/db/routing.py:18`:

```python
@contextmanager
def use_replica():
    conn = replica_pool.get_connection()
    try:
        yield conn
    finally:
        replica_pool.release(conn)
```

Applied at 6 call sites via `rg`. Five look fine — pure read paths like `ProductCatalog.search()` and `ReportService.monthly_summary()` with no preceding write in the request. One is a real bug:

`app/services/order_service.py:104`, inside `create_order()`:

```python
def create_order(self, cart_id):
    order = self._insert_order(cart_id)          # write, primary
    with use_replica() as conn:
        confirmation = self._load_order_for_confirmation(conn, order.id)  # read, replica
    return confirmation
```

Immediately after inserting the order on the primary, it reads it back from a replica to build the confirmation response. If replication lag exceeds the time between the insert and this read — which on most managed Postgres replica setups is commonly single-digit milliseconds but not bounded — `_load_order_for_confirmation` can return nothing, or stale data, for an order that was just created.

[[thinking]]
Before flagging this as a hard blocker I should check what actually happens on a cache miss here — does `_load_order_for_confirmation` raise, return `None` and get handled gracefully, or silently return a wrong/partial object? The severity of "read your own write" bugs varies a lot depending on whether the failure is loud or quiet.

[[response]]
Checked `_load_order_for_confirmation` at `app/services/order_service.py:130` — it does `Order.objects.replica_conn(conn).get(id=order_id)`, and `.get()` on no match raises `Order.DoesNotExist`, which isn't caught anywhere in `create_order()`. So under replication lag, a customer who just placed an order gets a 500 on checkout confirmation, even though the order was created successfully. That's a hard-failure "read your own write" bug, not a silent staleness issue — worse in some ways, since it's loud and would look like the write itself failed.

Confirmed replicas are asynchronous (checked `infra/terraform/rds.tf:41` — standard async read replica, not synchronous), so lag is a real, unbounded-in-the-worst-case possibility, not a theoretical one.

[[thinking]]
The general pattern for this — read-your-write inside the same request — is to route that specific read to the primary rather than exempting the whole request from replica usage, so I don't lose the replica benefit for the parts of the request that don't need read-your-write consistency. Should also check if this is the only place with the pattern before calling it fixed.

[[response]]
Fixed by routing just the post-write read back to the primary, not the whole request:

```diff
 def create_order(self, cart_id):
     order = self._insert_order(cart_id)
-    with use_replica() as conn:
-        confirmation = self._load_order_for_confirmation(conn, order.id)
+    confirmation = self._load_order_for_confirmation(primary_conn(), order.id)
     return confirmation
```

Grepped for the same shape elsewhere — a write followed by a replica-routed read of the thing just written — across all six call sites plus the rest of `app/services/`: `rg -n -B4 "use_replica" app/ | rg -B4 "_insert_|_update_|_create_"`. One more hit, `app/services/inventory_service.py:66`, adjusting stock then reading it back via replica for a response payload. Same fix applied there.

The other four call sites are genuinely lag-tolerant — catalog search and report summaries don't have a preceding write in the same request, so a few milliseconds of staleness is invisible to the user and replica routing is the right call for them.
