# Script 73 — IDOR on the internal invoices API

[[thinking]]
A reported IDOR needs two things before anything else: the exact endpoint and whether the resource lookup is scoped to the current account at all, or scoped-but-bypassable. Those are different bugs with different fixes, so I want to read the controller before speculating about severity.

[[tool_use]]
name: Bash
input:
  command: rg -n "class InvoicesController" -A 30 app/controllers/api/internal/invoices_controller.rb
  description: Read the controller the report points at

[[response]]
Confirmed — this is the simple, bad kind of IDOR: no account scoping at all.

```ruby
class Api::Internal::InvoicesController < ApplicationController
  before_action :authenticate_user!

  def show
    invoice = Invoice.find(params[:id])
    authorize_admin!  # only checks current_user.role == "admin"
    render json: invoice
  end
end
```

`Invoice.find(params[:id])` looks up by primary key across the entire table. `authorize_admin!` at `app/controllers/api/internal/invoices_controller.rb:9` checks that the caller is *an* admin, not that the invoice belongs to *their* account. Any admin at Account A can fetch `GET /api/internal/invoices/4821` and get an invoice belonging to Account B — full name, line items, billing address.

Same pattern on `update` (`:15`) and `destroy` (`:21`) — this isn't read-only exposure, an Account A admin can also edit or delete Account B's invoice.

[[thinking]]
Before writing the fix I want to know how many other internal controllers use the same `authorize_admin!` + `Model.find(params[:id])` pattern — if this is copy-pasted, patching one controller and calling it done leaves the same hole open elsewhere, which is exactly the kind of thing a follow-up pentest finds a week later.

[[response]]
It's copy-pasted. Same shape in three more controllers:

```
app/controllers/api/internal/invoices_controller.rb:8   (show, update, destroy)
app/controllers/api/internal/payouts_controller.rb:11   (show)
app/controllers/api/internal/refunds_controller.rb:14   (show, update)
```

All four scope authorization to role, never to account. Fixing invoices alone and calling the report closed would leave payouts and refunds exploitable via the identical technique.

[[thinking]]
Before writing the actual fix I want to check whether any legitimate feature currently depends on cross-account admin lookup — a support/impersonation tool, for instance, would break outright if I scope every lookup to `current_admin.account` with no escape hatch, and I'd rather find that now than have someone report a regression tomorrow.

[[response]]
Checked for a support-tool call path — `rg -n "current_admin.account.invoices|Invoice.find" app/` turns up only the four controllers already found, no separate impersonation/support flow touching `Invoice`, `Payout`, or `Refund` directly. There's a `SupportSession` model elsewhere in the app, but it operates on `User` records, not billing resources, and doesn't share these controllers. Safe to scope the lookup itself with no exception path needed.

Fix is to scope the lookup itself, not add an after-the-fact ownership check that's easy to forget on the next controller:

```diff
   def show
-    invoice = Invoice.find(params[:id])
-    authorize_admin!
+    authorize_admin!
+    invoice = current_admin.account.invoices.find(params[:id])
     render json: invoice
   end
```

`current_admin.account.invoices.find` raises `ActiveRecord::RecordNotFound` (404) for out-of-account IDs instead of leaking a 200 with someone else's data or a 403 that confirms the ID exists. Applied the same pattern to `update`/`destroy` on invoices, and to `payouts` and `refunds`.

[[thinking]]
This needs a regression test that actually proves cross-account access is blocked, not just that the happy path still works — the whole bug was that the happy path always worked fine.
