# Script 61 — an unbounded queue causing OOM under burst traffic

[[thinking]]
An OOM that only happens under burst traffic, not steady load, points at something that buffers faster than it drains rather than a straightforward memory leak — a leak would show up under steady load too, just slower. I want the heap dump from the actual incident before reading code, since guessing which collection is unbounded from source alone is unreliable when there are several queues in this service.

```
Class                                          Instances   Bytes
java.util.concurrent.LinkedBlockingQueue$Node   4,812,003   192,480,120
com.acme.orders.NotificationEvent               4,811,998   1,058,639,560
```

~1GB held by `NotificationEvent` objects sitting in queue nodes. That points at `NotificationDispatcher`.

`src/main/java/com/acme/orders/NotificationDispatcher.java:22`:

```java
private final BlockingQueue<NotificationEvent> queue = new LinkedBlockingQueue<>();

public void enqueue(NotificationEvent event) {
    queue.put(event); // blocks only when capacity is full — but there is no capacity
}
```

`new LinkedBlockingQueue<>()` with no capacity argument is unbounded — `put()` never blocks, it just keeps allocating nodes. Under normal traffic the single consumer thread (`NotificationWorker`, one thread, per `NotificationDispatcher.java:41`) keeps pace. During a burst — the incident was a marketing email blast that fired ~5M order-confirmation events in 90 seconds — production outpaces the one consumer by roughly 400x and the queue just grows until the heap is gone.

[[thinking]]
Before writing this off as one bad queue, I want to check whether other services in this codebase use the same no-arg `LinkedBlockingQueue` constructor — if it's a copy-pasted idiom, fixing this one instance and shipping leaves the same OOM waiting to happen elsewhere under a different burst trigger.

[[thinking]]
Bounding the queue is necessary but not sufficient — the moment it's bounded, `put()` on a full queue blocks the producer thread, and I need to know whether that producer is a request-handling thread. If it is, a bounded queue turns an OOM into a request-timeout pileup instead, which is better but still a problem worth naming rather than presenting as fully solved.

[[response]]
Checked the call site: `enqueue()` is called from `OrderController.confirmOrder()` on the Tomcat request thread, per `src/main/java/com/acme/orders/web/OrderController.java:67`. So a bounded queue with a blocking `put()` would, under the same burst, stall request threads instead of OOMing the JVM — better failure mode, but it trades one incident type for a thread-pool exhaustion incident.

Went with a bounded queue plus a non-blocking offer that sheds load explicitly instead of blocking silently:

```diff
-private final BlockingQueue<NotificationEvent> queue = new LinkedBlockingQueue<>();
+private static final int MAX_QUEUE_SIZE = 50_000;
+private final BlockingQueue<NotificationEvent> queue = new LinkedBlockingQueue<>(MAX_QUEUE_SIZE);
 
 public void enqueue(NotificationEvent event) {
-    queue.put(event);
+    boolean accepted = queue.offer(event);
+    if (!accepted) {
+        notificationsDropped.increment();
+        log.warn("notification queue full, dropping event orderId={}", event.orderId());
+    }
 }
```

50,000 was sized off the incident: peak observed burst was ~5M events over 90s against one consumer processing ~250/s, so even a fast drain of the backlog takes minutes — the bound needs to survive several seconds of full-rate bursting without immediately shedding, but not hold gigabytes. 50k events at ~220 bytes each is ~11MB, versus the 1GB+ that took the JVM down.

[[thinking]]
Dropping notifications silently, even with a log line and a metric, is a product-visible behavior change — customers wouldn't get order-confirmation emails during a burst. I should be explicit that this is a tradeoff I made, not something to bury in a log line, and flag it rather than assume dropping is acceptable.

[[response]]
`mvn test -Dtest=NotificationDispatcherTest` → `Tests run: 9, Failures: 0`, including a new `dropsEventsWhenQueueFull` test that fills the queue to capacity and asserts `offer()` returns false without blocking.

Load-tested with a synthetic burst matching the incident shape (5M events/90s, single consumer): heap stayed flat at ~40MB retained by the queue, versus growing unbounded before. `notificationsDropped` counter caught ~4.97M of the 5M events during the simulated burst, since the consumer can only drain ~22k in 90s.

Flagging rather than shipping silently: this stops the OOM, but under a burst this size it drops most order-confirmation notifications instead of crashing. That's arguably the right tradeoff — a dropped notification is recoverable, a crashed service isn't — but it's a product call, not just an engineering one. If dropping isn't acceptable, the alternatives are scaling consumer parallelism or moving to a durable queue (SQS/Kafka) that absorbs the burst and drains over minutes instead of holding it in JVM heap. Which direction do you want?
