# Script 2 — debugging a flaky test

[[thinking]]
A test that fails one run in twenty is almost never the test being wrong — it's shared state or a race. Before I read the test itself I want to know whether it fails in isolation or only in a full suite run. That single fact splits the search space in half, so let me get it first.

[[tool_use]]
name: Bash
input:
  command: git log --oneline -8
  description: Check what landed recently around the flaky test

[[response]]
Ran `tests/test_orders.py::test_concurrent_checkout` 200 times in isolation — 200 passed, no failures.

Ran the full suite 20 times — 3 failures, all the same assertion:

```
E   AssertionError: assert 2 == 1
E    +  where 2 = Order.objects.filter(status="paid").count()
```

So it only fails alongside other tests, which rules out the test's own logic and points at leaked state. Narrowing down which test it collides with next.

[[thinking]]
The clean approach is bisection over test ordering rather than reading every test that touches orders. pytest-randomly seeds the order, so if I can find a seed that reproduces reliably I can cut the set in half repeatedly. Much faster than guessing which of forty tests is the culprit.

[[response]]
Found it by bisecting on the random seed.

`-p no:randomly -k "test_refund_flow or test_concurrent_checkout"` fails 10/10. Either test alone passes.

The collision is at `tests/test_orders.py:47`:

```python
@pytest.fixture(scope="module")
def order_factory():
    return OrderFactory(counter=itertools.count(1))
```

`scope="module"` means both tests share one `OrderFactory`, and the factory's `counter` is what generates order IDs. `test_refund_flow` consumes two IDs, so when it runs first, `test_concurrent_checkout` starts at ID 3 instead of 1 — and its assertion filters on IDs it assumed would be 1 and 2.

It passes in isolation because the counter happens to start where the test expects.

[[thinking]]
Two ways to fix this. Change the scope to function, or stop the test depending on specific ID values. Function scope is the one-word fix but it rebuilds the factory per test, and there are sixty tests in this module using it — that could be a real slowdown. Let me check what the factory actually costs to construct before recommending it.

[[response]]
Went with function scope. I checked the cost first since the module has 60 tests using this fixture:

```
module scope:   4.71s
function scope: 4.88s
```

0.17s total — `OrderFactory.__init__` only builds an `itertools.count`, so there's nothing meaningful to amortize. The module scope wasn't buying anything.

```diff
-@pytest.fixture(scope="module")
+@pytest.fixture
 def order_factory():
     return OrderFactory(counter=itertools.count(1))
```

Full suite, 50 runs: 50 passed. Previously 3 failures per 20.

One thing worth flagging: three other fixtures in this file are also `scope="module"` and hold mutable state — `cart_factory:61`, `user_session:78`, `payment_stub:94`. None are failing today, but they're the same shape of bug waiting for an unlucky ordering. Want me to convert those too?
