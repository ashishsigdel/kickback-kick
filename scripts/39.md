[[thinking]]
Flag-gated risky UI changes fail in a couple of characteristic ways that plain code review often misses because the code "looks fine" per branch — the interesting bugs are usually at the seam between server-rendered and client-evaluated state, and in what happens when the flag service itself is unreachable. I want to see how and where the flag is actually evaluated before reviewing the two branches of UI it guards, since evaluation timing determines whether this is even safe to ship as a flag at all.

The redesign itself is solid — `CheckoutSummaryV2.tsx` is well-isolated behind the flag, no shared mutable state with the old component. Two issues before this ships, one of them a real bug.

**1. Hydration mismatch — real bug.** `src/pages/checkout.tsx:18`:

```tsx
export default function CheckoutPage() {
  const showRedesign = useFeatureFlag("checkout-redesign-v2"); // resolves client-side only
  return showRedesign ? <CheckoutSummaryV2 /> : <CheckoutSummaryV1 />;
}
```

`useFeatureFlag` reads from a client-side flag client (`src/flags/client.ts:12`, backed by a fetch to the flag service on mount, default `false` until it resolves). Server-side rendering has no access to that resolved value, so the server always renders `CheckoutSummaryV1`. If the flag resolves to `true` for a given user after mount, React hydrates client-side into `CheckoutSummaryV2` while the server-sent HTML was `V1` — a hydration mismatch. In dev this throws the standard "Text content does not match server-rendered HTML" warning; in prod React discards the server tree and re-renders client-side, which means every flagged-in user sees a visible flash from `V1` to `V2` on every checkout page load, plus the usual hydration-mismatch performance cost of throwing away the SSR work entirely.

**2. Fail-open on flag service errors.** `src/flags/client.ts:24`:

```ts
try {
  const res = await fetch(flagServiceUrl);
  return await res.json();
} catch {
  return { "checkout-redesign-v2": true }; // <-- fails open to the risky variant
}
```

If the flag service is unreachable — the exact scenario a kill switch exists for — every user gets defaulted into the new checkout flow, not the safe old one. That inverts the point of gating a risky change behind a flag: the flag becomes a liability exactly when the infrastructure around it is having a bad day.

[[thinking]]
Before writing up a recommendation I want to check whether resolving the flag server-side is actually feasible here — some flag services support server-side evaluation via an API key and user context passed at request time, which would fix the hydration issue directly rather than needing a workaround on the client. If that's not available, the fix has to be a client-only pattern (skip SSR for this fragment, or gate on a stable server-known signal) instead.

[[response]]
Checked — the flag provider (`LaunchDarkly`-style SDK per `package.json`) does support server-side evaluation via `initialize(sdkKey).variation(flagKey, userContext, default)`, it's just not being used; only the client SDK is wired up. That makes server-side resolution the right fix rather than a client-side workaround.

Recommended changes, not yet applied since this is a design/safety review and the SSR data-fetching change touches more than just this component:

1. **Resolve the flag server-side** in `getServerSideProps` (or equivalent data loader) using the user's session, and pass the resolved boolean as a prop — server and client then agree from the first paint, no hydration mismatch, no flash.
2. **Flip the fail-open default to fail-closed**: `return { "checkout-redesign-v2": false }` on fetch failure. A flag service outage should degrade to the known-safe path, not the one being actively tested.
3. Minor: no cleanup ticket referenced anywhere for removing the flag and the V1 component once the rollout completes — worth a linked issue now so this doesn't become permanent dead-code-behind-a-flag six months from now.

#2 is the one I'd block on — it's a one-line change and it's the difference between "flag service hiccup causes a brief evaluation delay" and "flag service hiccup force-enables the risky variant for 100% of traffic." #1 is a real bug but lower stakes (a visual flash, not a wrong-variant-for-everyone failure mode) and touches the data-fetching layer, so I'd want you to confirm the SSR approach before I change it. Want me to implement the fail-closed default now and leave the SSR restructuring for a follow-up, or handle both together?
