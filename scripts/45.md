
never raises here, so `pr_2`'s failure is currently invisible. Before this diff, the per-item loop would have raised on `pr_2` specifically and logged it; after this diff, batching silently loses that failure. This is the regression the review needs to catch — it's not in the diff's stat output, it's in what the diff doesn't check.

[[thinking]]
The fix needs to preserve the old guarantee — every failed item gets surfaced individually — while keeping the batching win. That means parsing `results`, not just the top-level status code, and deciding what "fail" means for the caller: raise on any per-item error, or return the partial result and let the caller decide. Given this is a price sync where a silently-skipped price update is a real business bug (stale price shown to customers), I'd lean toward surfacing failures loudly rather than swallowing them into a return value.

[[response]]
Suggested change — parse per-item results and raise a batch-aware error listing exactly which items failed, rather than treating the batch as atomic:

```diff
+class BulkSyncError(Exception):
+    def __init__(self, failures: list[dict]):
+        self.failures = failures
+        super().__init__(f"{len(failures)} of batch failed: {[f['id'] for f in failures]}")
+
 def sync_prices(items: list[PriceUpdate]) -> None:
+    all_failures = []
     for batch in chunked(items, 500):
         resp = client.post("/v1/prices/bulk", json={"updates": [i.to_payload() for i in batch]})
         resp.raise_for_status()
+        results = resp.json()["results"]
+        failures = [r for r in results if r["status"] != "ok"]
+        if failures:
+            logger.error("bulk_price_sync.partial_failure", count=len(failures), items=failures)
+            all_failures.extend(failures)
+    if all_failures:
+        raise BulkSyncError(all_failures)
```

This keeps the batching win (500 items → 1 call, `raise_for_status()` still catches transport-level and auth failures) while restoring per-item visibility. Callers that previously relied on "one bad item stops the whole sync" get that back via the raised exception at the end, after all batches attempt — which is actually better than the old loop, since one bad item no longer aborts the remaining 499.

Haven't run this against the sandbox API yet since I don't have write access from here — flagging this as the one thing that needs verification before merge: does `results` preserve the same order as the request's `updates` array, or does it need matching by `id`? I wrote the diff assuming order isn't guaranteed (`r["id"]` in each result) since I couldn't find that documented either way. Worth confirming with the API owner before relying on it.
