# Script 85 — the migration that locked the orders table

[[thinking]]
A migration causing a production outage is a postmortem-shaped task: I need the actual lock, not a guess from reading the migration file cold, because plenty of migrations that *look* dangerous acquire only a brief lock, and plenty that look innocent (an added column with a non-null default, an index without `CONCURRENTLY`) take a full-table `ACCESS EXCLUSIVE` lock for the duration of a rewrite. Postgres's own lock and log data settles that, not intuition.

[[response]]
Outage window matches the migration exactly. The log line:

```
2026-07-22 09:14:02 UTC LOG:  duration: 41230.918 ms  statement: ALTER TABLE orders ADD COLUMN fulfillment_priority integer NOT NULL DEFAULT 0;
```

41 seconds holding a lock on `orders`, and `pg_stat_activity` during that window showed 30+ backends stuck in `Lock` wait state, all ordinary `SELECT`/`UPDATE` traffic against `orders` — that's the request timeout spike matching the incident window.

The migration, `db/migrate/20260722_add_fulfillment_priority.rb`:

```ruby
def change
  add_column :orders, :fulfillment_priority, :integer, null: false, default: 0
end
```

This is the Postgres version-dependent trap: on Postgres 11+, adding a column with a constant default is metadata-only — no rewrite. That fast path landed in 11; checked `SELECT version()` and this instance is on 10. On 10, `ADD COLUMN ... NOT NULL DEFAULT` rewrites the entire table to backfill the default into every row, holding `ACCESS EXCLUSIVE` for the full duration. `orders` has 9.4M rows; that's the 41 seconds.

[[thinking]]
A 41-second exclusive lock is exactly what `lock_timeout` exists to prevent — a well-configured migration session should fail fast rather than hold the lock while blocked backends queue up. Worth checking whether that setting exists at all here before treating this purely as a migration-authoring mistake.

[[response]]
No `lock_timeout` or `statement_timeout` set anywhere — not `config/database.yml`, not the deploy migration task (`lib/tasks/deploy.rake:18`). Postgres defaults both to `0` (wait forever). The migration wasn't just slow — it was structurally allowed to hold an exclusive lock indefinitely, for however long the rewrite took. `lock_timeout` is the backstop that turns "surprise long lock" into "migration fails loudly, nobody's blocked," independent of whether the migration itself is well-written.

[[thinking]]
Knowing it's a full rewrite on PG10 tells me the fix, but there are two different correct patterns depending on constraints: add nullable + backfill in batches + add `NOT NULL` after, or (if PG10 is being upgraded soon anyway) just wait. Since this is a live incident and the column is already partially useful even nullable, the batched approach is the one that doesn't require a version upgrade to ship, so I'll design around what's true today.

[[response]]
Standard three-step split for a NOT-NULL-with-default column addition on a version without the fast path — each step takes a lock briefly instead of one step holding it for the whole rewrite:

```ruby
# migration 1 — instant, metadata only, no rewrite
class AddFulfillmentPriorityToOrders < ActiveRecord::Migration[7.0]
  def change
    add_column :orders, :fulfillment_priority, :integer, default: 0
    # no NOT NULL yet — nullable add is metadata-only even on PG10
  end
end

# migration 2 — backfill in batches, out of band, no long lock
class BackfillFulfillmentPriority < ActiveRecord::Migration[7.0]
  disable_ddl_transaction!
  def up
    Order.in_batches(of: 5_000) do |batch|
      batch.where(fulfillment_priority: nil).update_all(fulfillment_priority: 0)
      sleep(0.1) # let replication/replicas catch up between batches
    end
  end
end

# migration 3 — add the constraint, validated separately from applied
class AddNotNullToFulfillmentPriority < ActiveRecord::Migration[7.0]
  def up
    execute "ALTER TABLE orders ADD CONSTRAINT fulfillment_priority_not_null CHECK (fulfillment_priority IS NOT NULL) NOT VALID"
    execute "ALTER TABLE orders VALIDATE CONSTRAINT fulfillment_priority_not_null"
  end
end
```

`NOT VALID` + separate `VALIDATE CONSTRAINT` is the key move in step 3: adding the constraint itself takes a brief `ACCESS EXCLUSIVE` lock but doesn't scan the table (fast), and `VALIDATE CONSTRAINT` scans to confirm existing rows comply but only takes a `SHARE UPDATE EXCLUSIVE` lock — which blocks other schema changes but not ordinary reads/writes. Splits one 41-second exclusive lock into three steps, none of which block application traffic for more than milliseconds.

[[response]]
Ran all three migrations against a staging replica restored from a production snapshot (9.1M rows in `orders`), with a concurrent read/write load generator running against the table throughout:

```
migration 1 (add nullable column):  duration 8ms,    0 blocked queries
migration 2 (batched backfill):     duration 4m40s,  0 blocked queries (each batch's UPDATE lock released between batches)
migration 3 (NOT VALID + VALIDATE): duration 1.9s,   0 blocked queries
```

No blocked queries across all three, versus the original single migration's 30+ blocked backends for 41 seconds. Total wall time is longer (backfill alone ~5 minutes vs. 41 seconds) but that's the correct tradeoff — slower and non-blocking beats fast and outage-causing.

Applied the three-step migration to production during a low-traffic window, `lock_timeout` set to `5s` for the deploy session going forward as the safety net regardless of migration quality:

```yaml

  lock_timeout: 5s
```

Flagging separately rather than folding into this fix: this instance is on Postgres 10, which reached end-of-life in November 2022 — no security patches for over three years. The three-step migration split works around the missing fast-path optimization, but every future `NOT NULL DEFAULT` column addition will hit the same trap until the instance is upgraded. That's a bigger and separate decision than this incident — is there already a Postgres upgrade planned, or should this outage be the forcing function to schedule one?
