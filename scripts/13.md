# Script 13 — deadlock in a worker pool under load

[[thinking]]
A thread pool that deadlocks only under load and not in the test suite usually means the deadlock needs contention to manifest — two locks acquired in inconsistent order, where low concurrency rarely hits the interleaving. Before reading code I want a thread dump from the stuck process; guessing at lock order from source alone is how you miss the actual cycle.

[[tool_use]]
name: Bash
input:
  command: gdb -p $(pgrep -f target/release/ingest-worker) -batch -ex "thread apply all bt" 2>&1 | head -80
  description: Capture a backtrace of every thread in the hung worker process

[[response]]
Two threads are blocked on each other's locks — classic deadlock, confirmed by the backtrace:

```
Thread 4 (LWP 9214):
#3 std::sync::mutex::Mutex::lock (queue.rs:203)
#4 ingest_worker::pool::Worker::steal_from (pool.rs:118)
#5 ingest_worker::pool::Worker::run (pool.rs:74)

Thread 7 (LWP 9217):
#3 std::sync::mutex::Mutex::lock (metrics.rs:41)
#4 ingest_worker::pool::Worker::record_and_requeue (pool.rs:156)
#5 ingest_worker::pool::Worker::run (pool.rs:74)
```

`pool.rs:118`, `steal_from`, locks `self.queue` then `self.metrics` when a worker's local queue is empty and it steals from a sibling:

```rust
fn steal_from(&self, sibling: &Worker) -> Option<Job> {
    let mut q = sibling.queue.lock().unwrap();
    let job = q.pop_front();
    if job.is_some() {
        let mut m = self.metrics.lock().unwrap(); // metrics after queue
        m.steals += 1;
    }
    job
}
```

`pool.rs:156`, `record_and_requeue`, locks `self.metrics` then `self.queue` — the reverse order:

```rust
fn record_and_requeue(&self, job: Job) {
    let mut m = self.metrics.lock().unwrap(); // metrics first
    m.requeued += 1;
    let mut q = self.queue.lock().unwrap();   // then queue
    q.push_back(job);
}
```

Under low load, workers rarely steal (queues stay non-empty) so `steal_from` almost never runs concurrently with `record_and_requeue`. Under load, queues drain and stealing becomes frequent — that's why this only shows up in production traffic, not the test suite.

[[thinking]]
Two standard fixes: enforce a single global lock order everywhere, or eliminate the need to hold both locks at once by copying data out before acquiring the second lock. A global order is simpler to reason about and to audit later — I should check whether every call site can be made to agree on one order before picking that, since the metrics lock is also taken independently in several places that don't touch the queue at all.

[[response]]
Went with a strict global order (queue before metrics, everywhere) rather than restructuring around lock-free copies — `rg -n "metrics.lock\(\)|queue.lock\(\)" src/pool.rs` shows 6 call sites total, and only these two ever hold both at once, so enforcing an order is a small, auditable change:

```diff
 fn record_and_requeue(&self, job: Job) {
-    let mut m = self.metrics.lock().unwrap(); // metrics first
-    m.requeued += 1;
-    let mut q = self.queue.lock().unwrap();   // then queue
-    q.push_back(job);
+    let mut q = self.queue.lock().unwrap();   // queue first, matches steal_from
+    q.push_back(job);
+    let mut m = self.metrics.lock().unwrap();
+    m.requeued += 1;
 }
```

Added a comment at the top of both lock-acquisition sites pointing at each other, and a `// LOCK ORDER: queue -> metrics` note in `pool.rs:1` above the struct definition so future call sites don't reintroduce the inversion silently.

Reran the load test that reproduced this — `cargo run --release --bin load_test -- --workers 16 --jobs 200000` — previously hung at ~40s every run (5/5). After the fix: 5/5 runs complete, ~38s each, no hang.

[[thinking]]
This kind of bug is exactly what a lock-order checker catches automatically instead of relying on a comment convention that can rot. Worth checking if the project already has `parking_lot`'s deadlock detection feature or something equivalent wired in, since `std::sync::Mutex` gives no diagnostics at all when this happens in production — the process just hangs silently.

[[response]]
Checked `Cargo.toml` — the project uses `std::sync::Mutex`, not `parking_lot`, so there was no deadlock detection at all; the only reason we caught this was a healthcheck timeout paging someone, followed by a manual `gdb` attach. That's a slow, manual path to diagnosis.

Not making this change unprompted since it's a dependency addition, but flagging it: `parking_lot` has an optional `deadlock_detection` feature that runs a background thread checking for lock cycles and logs them with a full backtrace, which would have surfaced this in minutes instead of requiring someone to notice the healthcheck failing and attach a debugger by hand.
