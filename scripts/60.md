Priority queues have a starvation failure mode that FIFO structurally can't have: low-priority work waits forever if high-priority work keeps arriving. That's the thing to check before anything else, because it won't show up in a functional test with ten jobs — it only shows up under sustained load, which is exactly when this queue will actually be under pressure.

The diff swaps a `container/list`-backed FIFO for a `container/heap` ordered by a new `Priority int` field, three levels: `High=0, Normal=1, Low=2`. `internal/queue/queue.go:38`:

```go
func (q *PriorityQueue) Push(job Job) {
    heap.Push(&q.items, job)
}

func (q *PriorityQueue) Pop() (Job, bool) {
    if q.items.Len() == 0 {
        return Job{}, false
    }
    return heap.Pop(&q.items).(Job), true
}
```

`Less(i, j)` at line 61 compares only `Priority` — no tiebreaker on enqueue order or age. Two problems:

1. **No starvation guard.** If `High` jobs keep arriving, `Low` jobs never pop — the heap has no notion of how long an item has waited.
2. **Non-deterministic ordering within a priority level**, since `container/heap` doesn't guarantee FIFO among equal keys. Two `Normal` jobs enqueued in order A, B can pop as B, A. For jobs that are logically independent this is fine; I need to check if any caller assumes same-priority FIFO ordering.

[[thinking]]
The within-priority ordering question is answerable by reading callers rather than guessing — if nothing depends on it, that's a non-issue and I shouldn't invent a fix for it. Starvation is the one I'm fairly confident matters regardless of caller behavior, since the whole point of a job queue is that everything eventually runs.

[[response]]
Checked callers — `internal/worker/dispatch.go:20` enqueues retry jobs at `Normal` priority in the order failures occurred, and nothing reads or depends on that order being preserved on the way back out. So the within-priority nondeterminism is benign here; not flagging it as a blocker.

Starvation is real, though, and it's not hypothetical for this system specifically: `internal/api/uploads.go:88` enqueues every user-triggered upload as `High`, and uploads are bursty — a customer doing a bulk import can push hundreds of `High` jobs in a few seconds. Anything sitting at `Low` (the nightly cleanup jobs, per `internal/scheduler/cleanup.go:14`) would wait indefinitely during that burst, and there's no metric that would surface it as "stuck" versus "queued" — the job doesn't error, it just never gets picked up.

[[thinking]]
Aging is the standard fix, but I want to propose the specific mechanism rather than "add aging" — the implementation choice (bump priority by wait time vs. hard age-out promotion) changes both the code and the guarantee it gives, so I should show the tradeoff rather than pick silently.

[[response]]
Two ways to close the starvation gap:

1. **Effective priority = base priority − (wait time / threshold)**, recomputed on each `Push`/`Pop` cycle. Smooth, no hard cutoff, but requires re-heapifying periodically since priorities drift while items sit in the heap.
2. **Hard promotion**: a background tick every N seconds walks the heap and promotes anything older than a threshold to `High`. Simpler, bounded worst-case wait (N + threshold), easier to reason about and test.

I'd go with #2 — it gives a concrete SLA ("nothing waits more than 2 minutes past its threshold") that's easy to alert on, versus #1's aging function being another thing to tune. Haven't implemented either yet since this changes the queue's guarantees, not just its internals, and that's worth confirming before I write it: is a 2-minute max-wait for `Low` acceptable, or does cleanup need a longer/shorter bound?
