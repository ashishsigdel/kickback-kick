# Script 22 — pod keeps getting OOMKilled

[[thinking]]
OOMKill on a K8s pod splits into two very different investigations: the process is actually leaking or legitimately growing past its request pattern, or the memory limit is simply set too low for a workload that's fine on average but spikes under specific conditions (batch requests, GC not keeping up with allocation rate). The `kubectl describe` event history and exit reason will tell me which shape this is before I go anywhere near application code — a limit set right at the p50 usage with no headroom looks very different from a genuine unbounded climb.

14 restarts in a week, each surviving 2-6 hours before OOMKilled. That's not "limit set too tight for a spike" — a genuinely undersized limit gets killed fast and consistently near startup. Surviving hours and climbing to the wall points at a leak or unbounded growth. Pulled the memory curve from `kubectl top pod` sampled every 5 min over one lifecycle via a quick loop, and it's a straight line from ~90Mi at start to 512Mi at kill, no plateau — consistent with something accumulating without bound rather than the process settling at a higher-but-stable working set.

[[thinking]]
Go services growing without bound under this shape are usually one of: an unbounded cache/map that's never evicted, goroutines that leak (each holding buffers), or — specific to Go — RSS growing because the runtime isn't returning memory to the OS as fast as GOMEMLIMIT/container limit awareness expects, which shows as "OOMKilled" even though live heap objects are actually bounded. I need pprof to tell live heap size apart from RSS before assuming it's a real leak versus a GC-tuning issue against the cgroup limit.

Live heap really is ~390MB and growing — this isn't a GOMEMLIMIT/cgroup-visibility artifact, it's a real leak in `ModelCache.Load`. `internal/scorer/cache.go:44`:

```go
func (c *ModelCache) Load(modelID string) (*Model, error) {
	c.mu.Lock()
	defer c.mu.Unlock()
	if m, ok := c.models[modelID]; ok {
		return m, nil
	}
	m, err := loadModelFromDisk(modelID)
	if err != nil {
		return nil, err
	}
	c.models[modelID] = m
	return m, nil
}
```

`c.models` is a plain `map[string]*Model` with no eviction, no size cap, no TTL. Each `*Model` is ~4-8MB (embedding weights). `modelID` is derived from a request parameter (`internal/handler/recommend.go:38`, `modelID := req.ExperimentVariant`), and the experimentation platform generates a new variant ID per A/B test cohort — checked the experimentation service's config, currently running ~60 concurrent variants and rotating roughly every 6 hours as tests conclude and new ones start. Every rotation adds new entries; nothing ever removes the old ones. That maps directly onto the 2-6 hour time-to-OOM depending on how many variant rotations happen to land in that pod's lifetime.

[[thinking]]
This needs a bounded cache, and given the access pattern (recently-used variants matter, old experiment cohorts don't) an LRU with a size cap is the right shape rather than a raw TTL — a TTL would evict a still-active long-running experiment just because it's old, while LRU naturally keeps whatever's actually being requested. I should size the cap based on the real working set (60 concurrent variants) with headroom, not an arbitrary round number.

At 150 entries × ~8MB worst case, cache ceiling is ~1.2GB live objects — still too high for the current 512Mi limit, so bumped the request/limit to `768Mi`/`1Gi` in `deploy/recommend-svc.yaml:31` to give the bounded cache room, rather than shrinking the cap to fit the old limit and risk thrashing (evicting and reloading from disk) during normal variant rotation.

Deployed to staging, ran the pod for 30 hours (versus the 2-6 hour kill window before): RSS plateaued at ~340MB after roughly 6 hours (matching one full variant-rotation cycle) and stayed flat. Zero restarts. Also exposed the pprof debug mux behind a `NetworkPolicy`-restricted port so this doesn't require a redeploy to diagnose next time.

Haven't load-tested what happens if a single deploy window somehow has more than 150 concurrent variants — LRU eviction under that would mean occasional disk reloads rather than an outage, so it degrades gracefully, but flagging it in case the experimentation platform's concurrency ever grows well past current levels.
