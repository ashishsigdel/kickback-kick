# Script 41 — a nightly cron job that fires twice

[[thinking]]
Duplicate charges from a nightly job usually mean either the schedule itself is duplicated (two beat instances, or a schedule registered twice) or the task isn't idempotent and a retry replays it. Those have completely different fixes, so I want to see how many times the task actually enters the worker before I look at the task body at all.

[[tool_use]]
name: Bash
input:
  command: grep -c "billing.tasks.run_nightly_charges" /var/log/celery/worker.log && grep "run_nightly_charges" /var/log/celery/worker.log | grep "Task accepted" | tail -4
  description: Count how many times the task was accepted by a worker last night

[[response]]
Two accepts, six minutes apart, same night, different task IDs:

```
[2026-07-21 02:00:03] Task accepted: billing.tasks.run_nightly_charges[8a1c...]
[2026-07-21 02:06:11] Task accepted: billing.tasks.run_nightly_charges[f02e...]
```

Different task IDs rules out a retry — Celery keeps the same ID on retry. This is two separate task instances being dispatched, which means the schedule itself is firing twice, not the task replaying.

[[thinking]]
If beat is only dispatching once, the duplicate has to come from either two beat processes running against the same schedule file, or the schedule being registered under two different names that both resolve to the same task. `celery beat` writes a schedule DB file with a PID lock, but that lock doesn't help if two separate deployments both think they're the singleton.

[[response]]
Found two beat processes:

```
$ ps aux | grep "celery beat"
deploy   14221  celery beat -A billing --schedule=/var/celery/beat.db
deploy   22890  celery beat -A billing --schedule=/var/celery/beat.db
```

PID 22890 is orphaned from the last deploy — the systemd unit restarts the worker on release but `celery-beat.service` doesn't have `KillMode=control-group`, so the old beat process survived the restart and kept scheduling alongside the new one. Both point at the same `beat.db` file, but the file-lock (`beat.db.lock`) only prevents concurrent writes to the schedule itself, not two processes independently computing "it's 02:00, fire the task" and both winning.

`deploy/systemd/celery-beat.service:12`:

```ini
[Service]
ExecStart=/opt/venv/bin/celery -A billing beat --schedule=/var/celery/beat.db
Restart=on-failure
```

No `KillMode`, no `ExecStop`. On redeploy, systemd sends the new `ExecStart` but the old process, still holding an open socket, doesn't get SIGTERM cleanly if it's mid-task.

[[thinking]]
Two independent fixes: stop the leak at the process-supervision layer, and make the task itself idempotent so this class of bug stops being customer-visible even if it recurs. I should do both — the systemd fix prevents the immediate cause, but relying on "only one beat process ever runs" as the sole safety net is fragile for something that charges cards.

[[response]]
Applied both fixes.

Supervision — kill the whole cgroup on restart so orphaned children can't survive a redeploy:

```diff
 [Service]
 ExecStart=/opt/venv/bin/celery -A billing beat --schedule=/var/celery/beat.db
 Restart=on-failure
+KillMode=control-group
+TimeoutStopSec=30
```

Idempotency — `billing/tasks.py:118`, the task now claims a per-day lock before charging, using a unique constraint rather than a soft check so it fails closed under a race:

```diff
 @shared_task
 def run_nightly_charges():
-    for account in Account.objects.filter(billing_due=True):
-        charge(account)
+    run_date = timezone.localdate()
+    for account in Account.objects.filter(billing_due=True):
+        created = ChargeRun.objects.filter(
+            account=account, run_date=run_date
+        ).get_or_create(account=account, run_date=run_date)[1]
+        if not created:
+            logger.warning("charge_run.duplicate_skipped", account=account.id, run_date=run_date)
+            continue
+        charge(account)
```

`ChargeRun` has a `unique_together = ("account", "run_date")` constraint, so under a genuine race the second `get_or_create` raises `IntegrityError` rather than both succeeding — caught and treated the same as `created=False`.

Killed the orphaned beat process manually (`kill 22890`), redeployed with the new unit file, confirmed `ps aux | grep beat` shows exactly one process. Ran the task twice back to back in staging: second run logs 340 `duplicate_skipped` entries, zero charges.
