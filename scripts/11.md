# Script 11 — race condition in a payment webhook handler causing duplicate processing

[[thinking]]
Duplicate charges from a webhook usually mean one of two things: the upstream provider is legitimately retrying (delivery isn't exactly-once, by design) and we're not deduping, or two goroutines are handling the same delivery concurrently and both slip past a check-then-act guard. The support ticket mentions the same `evt_` ID appearing twice in our logs within milliseconds, which smells like the second case. Let me find the handler and see how it guards against replay.

Found it — `internal/webhook/handler.go:41`:

```go
func (h *Handler) HandleWebhook(ctx context.Context, evt *stripe.Event) error {
	var exists bool
	err := h.db.QueryRowContext(ctx,
		`SELECT EXISTS(SELECT 1 FROM webhook_events WHERE event_id = $1)`, evt.ID,
	).Scan(&exists)
	if err != nil {
		return err
	}
	if exists {
		return nil // already processed
	}

	if err := h.processCharge(ctx, evt); err != nil {
		return err
	}

	_, err = h.db.ExecContext(ctx,
		`INSERT INTO webhook_events (event_id, processed_at) VALUES ($1, now())`, evt.ID)
	return err
}
```

Classic check-then-act: the `SELECT EXISTS` and the `INSERT` aren't atomic. The provider sends the same event twice inside a ~40ms window when our endpoint is slow to ACK (visible in `logs/webhookd-2026-07-20.log:1183` and `:1184`, both `evt_1Nx7...`, 38ms apart). Two goroutines — this handler runs behind a worker pool of 8 — both read `exists=false` before either finishes the INSERT, so both call `processCharge`.

`webhook_events.event_id` at `db/schema.sql:112` has no unique constraint, so nothing at the DB level would even reject the second insert.

[[thinking]]
An in-process mutex would close the race for a single instance, but this service runs 3 replicas behind the load balancer — the provider has no guarantee both retries land on the same pod. The guard has to live in the database, as an atomic operation, not in application memory. `INSERT ... ON CONFLICT DO NOTHING` is the standard shape for this; I need to check whether `processCharge` is safe to attempt and then roll back if the insert loses the race, or whether I should reserve the row first.

[[response]]
Reserving first, so a losing goroutine never calls `processCharge` at all — cheaper and avoids any charge-then-undo path.

```sql
-- db/migrations/0047_webhook_events_unique_event_id.sql
ALTER TABLE webhook_events
  ADD CONSTRAINT webhook_events_event_id_key UNIQUE (event_id);
```
