# Script 3 — tracking down a performance regression

[[thinking]]
An endpoint that went from 80ms to 1.2s after a deploy gives me a bounded search: whatever changed is in that diff. Guessing at likely causes before reading it wastes time. Let me get the commit range first, then profile a single request so I'm working from a measurement rather than a hunch.

[[tool_use]]
name: Bash
input:
  command: git log --oneline -12
  description: Get the commit range for the regression

[[response]]
Diffed the deploy range — 14 commits, 3 of which touch `api/reports.py`.

Profiled one request to `/reports/summary` with cProfile:

```
ncalls  tottime  cumtime  filename:lineno(function)
  1     0.001    1.187   api/reports.py:88(summary)
841     0.004    1.102   db/query.py:31(fetch_one)
841     1.081    1.081   {method 'execute' of 'Cursor'}
```

841 queries for one request. That's a classic N+1 — the endpoint used to issue 3.

Tracing where the 841 comes from now.

[[thinking]]
841 is a suspiciously specific number. If it's one query per row, there should be about 840 rows in whatever the outer loop iterates, plus the outer query itself. That pattern usually means someone replaced a join with an attribute access inside a loop — the code reads more cleanly, which is exactly why it slips through review.

[[response]]
That's exactly what happened. Commit `a3f91c2`, "simplify report aggregation":

```diff
-rows = db.fetch_all(SUMMARY_JOIN_SQL, org_id)
-for row in rows:
-    total += row.line_total
+orders = Order.objects.filter(org=org_id)
+for order in orders:
+    total += order.customer.tier.multiplier * order.amount
```

840 orders in the test org. `order.customer` is lazy, so each iteration fires a query — and `.tier` fires another on the ones not already cached.

The rewrite is genuinely more readable, which is why it passed review. It just moved the join from SQL into a Python loop.

[[thinking]]
I don't want to simply revert to the raw SQL — the readability gain was real, and reverting means the next person re-introduces this. `select_related` keeps the readable form and collapses it back to one query. Let me verify the query count actually drops rather than assuming the ORM does what I expect.

[[response]]
Fixed with `select_related`, keeping the readable loop:

```diff
-orders = Order.objects.filter(org=org_id)
+orders = Order.objects.filter(org=org_id).select_related("customer__tier")
```

Measured before and after with `django-silk`:

| | Queries | p50 | p95 |
| --- | --- | --- | --- |
| Before | 841 | 1187ms | 1402ms |
| After | 1 | 74ms | 96ms |

Slightly faster than the original 80ms, since the ORM's generated join is narrower than the old hand-written `SUMMARY_JOIN_SQL`.

I also added a regression guard at `tests/test_reports.py:203`:

```python
def test_summary_query_count(django_assert_num_queries):
    with django_assert_num_queries(1):
        client.get("/reports/summary")
```

That fails loudly if someone reintroduces a lazy access in this loop — which is the actual failure mode here, not slowness in general.
