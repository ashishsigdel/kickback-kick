Diff adds `src/schemas/order.ts` and a generic `validate(schema)` middleware applied to the write endpoints. Overall direction is right — this was previously unvalidated (`req.body` cast straight to the handler's type with no runtime check). Three things I'd flag before merge.

**1. `z.coerce.number()` on `quantity` — `src/schemas/order.ts:12`**

```ts
export const createOrderSchema = z.object({
  productId: z.string().uuid(),
  quantity: z.coerce.number().int().positive(),
  note: z.string().max(500).optional(),
});
```

`z.coerce.number()` runs `Number(input)` before validating, so `quantity: "5e2"` coerces to `500`, `quantity: true` coerces to `1`, `quantity: ""` coerces to `0` then fails `.positive()` — inconsistent, surprising inputs pass. For a JSON API where the client should be sending a real number, coercion mainly exists for form/query-string inputs where everything arrives as a string. I'd use `z.number().int().positive()` here and let genuinely malformed input 400 rather than silently reinterpreting it.

**2. Schema doesn't strip or reject unknown keys** — plain `z.object` defaults to *stripping* unknown keys silently, which is usually fine, but worth confirming it's intentional here rather than assumed, since it means a client sending `{"productId": ..., "isAdmin": true}` won't error, it'll just quietly drop `isAdmin`. That's safe as long as nothing downstream re-merges `req.body` instead of the parsed result — worth checking.

**3. Validation errors return the raw Zod issue array — `src/middleware/validate.ts:19`**

```ts
} catch (e) {
  if (e instanceof z.ZodError) {
    res.status(400).json({ errors: e.issues });
  }
}
```

`e.issues` includes `path` (fine, useful to clients) but also Zod's internal `code` values like `invalid_type`, `too_small`, and for refined schemas can include the validator function's internal message verbatim. Not a security leak at the level of stack traces, but it's an implementation detail becoming a public API contract — if the schema internals change (e.g. swapping `.uuid()` for a custom regex later), the error shape clients depend on changes with it, silently.

[[thinking]]
This one's a design call rather than a strict bug — the fix is straightforward (map to a stable shape), but I want to state it as a recommendation rather than a blocker, since coupling the wire format to Zod internals is bad practice but not broken today.

[[response]]
Recommend mapping to a stable error shape rather than passing Zod's format through:

```ts
res.status(400).json({
  errors: e.issues.map(i => ({ field: i.path.join('.'), message: i.message })),
});
```

That decouples the public contract from Zod's internal issue format — you can change validation libraries or add refinements later without a client-visible break.

Summary: `quantity` coercion and the error-shape leak are worth fixing before this ships publicly; the unknown-key stripping I checked and it's already handled correctly. Not blocking on my end for #1 or #3 since neither is a security hole, just a forward-compatibility one — your call whether that's in scope for this diff or a fast follow.
