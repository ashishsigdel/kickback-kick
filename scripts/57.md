# Script 57 — reviewing a diff that adds distributed tracing spans

[[thinking]]
Tracing diffs are easy to approve on skim because they look additive — just spans wrapping existing calls. The real risk is cardinality and span lifecycle: a span that never ends, or one tagged with a high-cardinality attribute, is invisible in code review and expensive in production. I want to check attribute usage before reading the happy-path logic.

[[tool_use]]
name: Bash
input:
  command: git diff main -- internal/billing/processor.go | rg -n "StartSpan|SetAttributes|End\(\)"
  description: Check every span's start/end pairing and attributes in the diff

[[response]]
15 `StartSpan` calls added, 15 `End()` calls — count matches, but pairing isn't the same as correctness. Two problems on read-through:

**1. Span never ends on the error path** — `internal/billing/processor.go:112`

```go
ctx, span := tracer.Start(ctx, "billing.charge_card")
result, err := gateway.Charge(ctx, req)
if err != nil {
    span.RecordError(err)
    return nil, err   // span.End() never called
}
span.End()
```

`defer span.End()` isn't used here — it's a manual call after the happy path only. Every failed charge leaks a span that never closes, which means it never gets exported and never shows up in the trace, but it does sit in the SDK's active-span bookkeeping until GC.

**2. High-cardinality attribute** — `internal/billing/processor.go:118`

```go
span.SetAttributes(attribute.String("card.last_four", req.CardLastFour))
```

Card-last-four isn't unbounded like an ID, but combined with `attribute.String("customer.email", req.Email)` two lines down, this is now tagging spans with PII. That's a compliance problem independent of cardinality.

[[thinking]]
Before I flag the attribute as a blocker I should check whether the collector config redacts or drops these attributes downstream — if there's a processor stripping PII before export, this might already be handled and I'd be raising a non-issue.

[[response]]
Checked the collector config at `deploy/otel-collector.yaml` — there's an `attributes` processor, but it only redacts `attribute.String("password", ...)` and `attribute.String("ssn", ...)`. `customer.email` and `card.last_four` pass through untouched to the exporter, which ships to a third-party tracing backend.

So both issues stand. Fixing the span leak is mechanical:

```diff
 ctx, span := tracer.Start(ctx, "billing.charge_card")
+defer span.End()
 result, err := gateway.Charge(ctx, req)
 if err != nil {
     span.RecordError(err)
     return nil, err
 }
-span.End()
```

`defer` also covers a third path I noticed — `gateway.Charge` can panic on a nil client during test setup, and without `defer` that path leaked a span too.

[[thinking]]
The PII attribute is a product/compliance call, not something I should silently strip or silently ship — better to state the fix options and let them pick, since removing `customer.email` might be relied on for debugging support tickets against traces.

[[response]]
`go test ./internal/billing/... -run TestTracing -v` → `PASS`, and a manual trace export against the local collector now shows the error-path span present where before it was missing entirely.

For the PII attribute, I didn't change it. Two ways to fix, different tradeoffs:

1. Hash `customer.email` (`attribute.String("customer.email_hash", sha256(...))`) — keeps traces correlatable across a customer's requests without shipping the raw address.
2. Drop it and rely on `customer.id`, which is already on the span at `processor.go:120` and is sufficient for support lookups.

`card.last_four` I'd leave as-is — four digits isn't enough to reconstruct a card number and it's genuinely useful for debugging declined-charge disputes. Let me know which way you want the email attribute to go before this merges.
