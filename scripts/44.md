[[response]]
Confirmed: page 1 with `pageSize=10` returns 9 rows, IDs 1–9. Row 10 is missing from page 1 and never appears on page 2 either — it's just gone, not shifted.

`internal/api/pagination.go:38`:

```go
func Paginate(page, pageSize int) (offset, limit int) {
    offset = (page - 1) * pageSize
    limit = pageSize - 1
    return offset, limit
}
```

`limit = pageSize - 1` is the bug. Someone hand-wrote a fencepost adjustment here, and it's wrong in the specific way that returns one fewer row than requested on every single page, not just the last one.

[[thinking]]
Before I touch it I want the blame on that line — a `-1` that specific usually isn't a typo, it's someone working around a different bug (maybe a duplicate-boundary-row issue between pages) and overcorrecting. If that's true, just deleting the `-1` might reintroduce whatever it was compensating for.

[[response]]
`git blame internal/api/pagination.go` on that line:

```
a41c9d2f (R. Okafor 2025-11-03) limit = pageSize - 1  // avoid dup row at page boundary
```

Found the referenced issue in the commit message — `git show a41c9d2f`:

> Fixes duplicate row appearing at start of page N+1 when page N's last row matches page N+1's first row under concurrent inserts.

So the `-1` was a workaround for a *different* bug: `OFFSET`-based pagination duplicating a row when a new row is inserted mid-scan (the same class of drift `OFFSET` pagination always has under concurrent writes). Trimming the limit by one doesn't fix that — it just under-fetches every page to reduce the visible frequency of the symptom, while permanently dropping one legitimate row per page as a side effect. That's a worse trade, not a fix.

[[thinking]]
The correct fix here is the one that actually addresses the root cause the original author was fighting: don't use `OFFSET` for a paginated feed that's read concurrently with inserts. Cursor-based pagination on a monotonic column removes the duplicate-row problem structurally, so the workaround limit can come out entirely rather than being patched.

[[response]]
Replaced offset pagination with cursor pagination keyed on `id` (already indexed, already monotonic — orders are append-only):

```diff
-func Paginate(page, pageSize int) (offset, limit int) {
-    offset = (page - 1) * pageSize
-    limit = pageSize - 1
-    return offset, limit
-}
-
-func (r *OrderRepo) List(page, pageSize int) ([]Order, error) {
-    offset, limit := Paginate(page, pageSize)
-    return r.db.Query(
-        `SELECT id, customer_id, total_cents, created_at FROM orders
-         ORDER BY id LIMIT $1 OFFSET $2`, limit, offset,
-    )
-}
+func (r *OrderRepo) List(afterID int64, pageSize int) ([]Order, error) {
+    return r.db.Query(
+        `SELECT id, customer_id, total_cents, created_at FROM orders
+         WHERE id > $1 ORDER BY id LIMIT $2`, afterID, pageSize,
+    )
+}
```

API contract changes from `?page=N` to `?after=<lastId>`, with `next_cursor` in the response body for the client to carry forward. Updated `internal/api/handlers/orders.go:71` and the OpenAPI spec at `api/openapi.yaml:210` to match.
