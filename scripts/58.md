```python
def to_local(utc_dt, tz_name):
    tz = pytz.timezone(tz_name)
    local = utc_dt.replace(tzinfo=pytz.utc).astimezone(tz)

```

This looks reasonable and is the standard `pytz` pattern for converting *into* a zone from UTC — that part's actually fine. The bug isn't here.

[[response]]
Found it upstream at `app/scheduling.py:58`, in the function that builds the schedule slots before they're ever passed to `to_local`:

```python
def build_slots(base_date, tz_name):
    tz = pytz.timezone(tz_name)
    slots = []
    for h in range(24):
        naive = base_date.replace(hour=h)
        slots.append(tz.localize(naive))   # localize() computes offset once
    return slots
```

`pytz`'s `localize()` picks the UTC offset based on the naive wall-clock time alone, using whatever the standard offset is for that moment in isolation — it doesn't know it's generating a full day's sequence. For `2026-03-08`, hour 2 doesn't exist locally (clocks jump 02:00 → 03:00), so `localize()` guesses `is_dst=False` by default and silently produces `01:00-06:00` again instead of raising or skipping — which is exactly the duplicate I saw.

This is a known `pytz` foot-gun: `localize()` and naive-datetime arithmetic don't compose safely across a transition. The fix is to stop constructing naive local times at all and build the slots from a UTC anchor instead.

[[thinking]]
Before rewriting `build_slots`, I want to check whether any caller depends on the return type being pytz-specific — some pytz methods like `normalize()` behave differently on pytz-attached datetimes than on stdlib `zoneinfo`-attached ones, and swapping the underlying tzinfo implementation could silently break a comparison or format call elsewhere.

[[response]]

None call `.normalize()` or any other pytz-specific method on the returned datetimes — they only compare and `.isoformat()` them, both of which `zoneinfo`-backed datetimes support identically. Safe to swap the implementation without touching any caller.

[[thinking]]
The safest fix is to not use pytz's localize-then-arithmetic pattern at all — zoneinfo with UTC-anchored construction sidesteps the ambiguity entirely rather than requiring me to special-case the skipped hour. Worth checking what Python version this needs before proposing it.
