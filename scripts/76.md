[[thinking]]
A gradual rise in miss rate that isn't tied to a deploy usually means eviction, not a cache-population bug — TTLs expiring is roughly constant-rate, but eviction under memory pressure is bursty and correlates with `used_memory` approaching `maxmemory`. I want Redis's own counters before touching application code, since `INFO stats` will tell me directly whether this is eviction or something else without any guessing.

`used_memory` is sitting essentially at `maxmemory` (6.44GB vs 6.44GB), and `evicted_keys` is climbing continuously, not just at deploy times. With `allkeys-lru`, once memory is full every new write evicts the least-recently-used key regardless of that key's own TTL or importance — so keys with a 24h TTL that were only accessed once get evicted in minutes if the instance is full, well before they'd naturally expire.

[[thinking]]
The question now is what's actually filling the memory — a real growth in working set, or something writing keys that shouldn't be cached at all (e.g. no TTL, or unbounded value sizes). `allkeys-lru` will evict indiscriminately including hot keys, so I want to see what's actually being written before deciding whether to raise `maxmemory`, change the policy, or fix a leak.

`session:render:report-a91f` at ~4MB stands out against an average of ~2.3KB. Grepped the write path: `services/report_cache.go:71` caches full rendered report HTML under the `session:render:*` prefix, same key space as small session tokens, and — checked the write call — with no TTL set at all:

```go
func CacheRenderedReport(rdb *redis.Client, sessionID string, html []byte) error {
    key := fmt.Sprintf("session:render:%s", sessionID)
    return rdb.Set(ctx, key, html, 0).Err()  // 0 = no expiry
}
```

`0` means no expiry — these multi-megabyte entries sit in memory forever unless evicted, competing for LRU slots against actual session tokens that are supposed to be the hot working set. That's the real driver: it's not organic growth, it's one code path writing large, immortal keys into a cache designed for small, short-lived ones.

[[thinking]]
Giving this its own TTL stops the leak going forward, but won't fix the instance being full right now, and `allkeys-lru` policy will keep evicting real session tokens under pressure regardless. Two separate actions: fix the write path, and decide whether policy/sizing also needs to change so a future mis-cached key can't repeat this.

[[response]]
Fixed the write path — 15 minute TTL, matching how long a rendered report is realistically re-requested:

```diff
 func CacheRenderedReport(rdb *redis.Client, sessionID string, html []byte) error {
     key := fmt.Sprintf("session:render:%s", sessionID)
-    return rdb.Set(ctx, key, html, 0).Err()  // 0 = no expiry
+    return rdb.Set(ctx, key, html, 15*time.Minute).Err()
 }
```

Also moved it off the `session:` prefix to `render:cache:` — sharing a namespace with actual session tokens was what let one bad key space starve the other under `allkeys-lru`. Deployed and watched `used_memory` over 20 minutes: dropped from 6.44GB to 4.1GB as the un-TTL'd keys aged out under the old prefix (they'll fall off entirely once the last one's LRU'd or the namespace is empty).

`go test ./services/... -run TestReportCache -v` → PASS, TTL asserted via `TTL` command in the test.
