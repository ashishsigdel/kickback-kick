# Script 69 — unusually slow startup after a dependency bump

[[thinking]]
Spring Boot startup regressions after a dependency bump are usually component scanning, autoconfiguration doing more work than before, or a new library initializing something eager (a connection pool, a cache warm, a schema check) that used to be lazy. The Actuator startup endpoint gives an ordered, timed breakdown for free — much better starting point than bisecting the dependency bump commit by commit.

[[tool_use]]
name: Bash
input:
  command: |-
    curl -s localhost:8080/actuator/startup | jq '.springBootVersion, (.timeline.events | sort_by(-.duration) | .[:8] | .[] | {name: .startupStep.name, duration}))' 2>&1 || mvn spring-boot:run -Dspring-boot.run.arguments=--debug | rg -i "took|ms$" | sort -t' ' -k1 -rn | head -20
  description: Get the ordered startup timing breakdown from Actuator, falling back to debug log timing

[[response]]
Actuator wasn't reachable in time (that's how slow this is), fell back to the debug log timing. Startup went from 4.2s to 31.8s after bumping `spring-boot-starter-parent` 2.7.9 → 3.2.1 and `com.acme:shared-security` 4.1.0 → 5.0.0. Top contributor by far:

```
2026-07-22 09:41:02.114 DEBUG --- ConfigurationClassPostProcessor : Fully validated bean definitions
2026-07-22 09:41:29.887 DEBUG --- HikariDataSource : HikariPool-1 - Start completed.
```

27.7s between those two lines, and in between the log is entirely `com.acme.security` — the `shared-security` library. Grepped that library's autoconfiguration:

`~/.m2/repository/com/acme/shared-security/5.0.0/shared-security-5.0.0.jar!/META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports` lists `JwksKeyRotationAutoConfiguration`, which didn't exist in 4.1.0.

[[thinking]]
New autoconfiguration classes that are slow at startup are usually doing network I/O eagerly — fetching a remote JWKS keyset, warming a cache, calling a health endpoint — rather than deferring it to first use. I want the actual source of that autoconfiguration class rather than guessing, since "eager network call" and "expensive in-memory computation" have completely different fixes.

[[response]]
Decompiled/checked the shared-security release notes for 5.0.0 — confirmed: `JwksKeyRotationAutoConfiguration` eagerly fetches the JWKS keyset from the identity provider's `/.well-known/jwks.json` *during bean initialization*, to pre-warm a local key cache "for faster first-request verification." Their changelog frames this as a feature.

The catch: it's a synchronous HTTP call with a 30s timeout, made during context refresh, on every single application boot including local dev startup — and in this environment the IdP endpoint is reachable but slow (fronted by a WAF with aggressive rate limiting that this call regularly gets throttled by, judging by the `429` visible in a manual `curl -v` against the same URL). That accounts for the ~28s.

`src/main/resources/application.yml` doesn't override the new `acme.security.jwks.eager-warmup` property this version introduces — confirmed by checking `JwksKeyRotationAutoConfiguration`'s `@ConditionalOnProperty`, which defaults it to `true`.

[[thinking]]
If a Kubernetes readiness probe or startup health indicator asserts the JWKS cache is populated before marking the pod ready, disabling eager warmup could let pods report ready before they can actually verify a token — a race that wouldn't show up in a local test but would matter the moment a freshly-ready pod takes real traffic.

[[response]]
Checked the health indicator this library registers — `JwksHealthIndicator` reports `UP` unconditionally once the bean exists; it doesn't check whether the cache has been populated. Readiness isn't gated on cache warmth either way. Also confirmed `deploy/k8s/service.yaml:44`'s `readinessProbe` only checks `/actuator/health`, not anything JWKS-specific, so this fix doesn't introduce a new race with the probe.

[[thinking]]
Turning off eager warmup is a one-line config change, but I want to check what "faster first-request verification" actually bought before just disabling it — if the alternative is the first real request paying a multi-second penalty instead, that's a tradeoff to name, not just start-up time moved somewhere invisible.

[[response]]
Checked what happens with warmup off: the key cache populates lazily on first JWT verification instead of at boot. Measured the cost — first authenticated request after a cold start now takes ~340ms extra (one JWKS fetch, same call, just deferred) instead of that cost being paid by every deploy regardless of whether a request ever needs those keys during boot.

```diff
 acme:
   security:
     jwks:
+      eager-warmup: false
       rotation-interval: 1h
```

That 340ms one-time cost on first real request is a much better trade than 28s on every boot, especially since this service autoscales and boot time directly affects how fast new pods come up under load — 28s of extra boot time per pod during a scale-up event is a real availability cost, not just a developer-experience annoyance.

`mvn spring-boot:run` → ready in 4.6s, back near the pre-bump baseline (0.4s over is the new `shared-security` library's smaller, non-network initialization work — acceptable). Verified JWT verification still works correctly on a cold instance: first request pays the deferred fetch, subsequent requests hit the warmed cache. `mvn test -Dtest=SecurityAutoConfigurationTest` → `Tests run: 11, Failures: 0`.

Didn't file this upstream yet, but defaulting `eager-warmup` to `true` seems like a questionable choice for a shared library used across services with different scaling profiles — worth raising with whoever owns `shared-security`.
