Read through `auth/session.py` and `auth/tokens.py`. Structure is sound — the split between issuing and verifying is clean, and I like that verification has no database dependency.

Three things stood out, worst first:

**1. Token comparison is not constant-time** — `auth/tokens.py:84`

```python
if token != expected:
    raise InvalidToken()
```

`!=` on `str` short-circuits at the first differing byte, which leaks prefix length through timing. Over enough samples that's a practical forgery path.

```python
import hmac
if not hmac.compare_digest(token, expected):
    raise InvalidToken()
```

**2. Refresh tokens never expire** — `auth/session.py:112`

`issue_refresh()` sets `exp` only when `ttl` is passed, and the two call sites both omit it. So a leaked refresh token is valid forever.

I'll keep digging into the session store before I make a call on the third one.

[[thinking]]
Now the session store. The thing I want to know is whether logout actually invalidates anything server-side, or just drops the cookie. If it's cookie-only, then that refresh token finding gets a lot worse — there'd be no way to revoke at all. Let me trace what `logout()` actually touches.

[[response]]
Confirmed the third issue, and it compounds the second.

**3. Logout is client-side only** — `auth/session.py:198`

```python
def logout(response):
    response.delete_cookie("sid")
```

Nothing is revoked server-side. Combined with non-expiring refresh tokens, a token captured once stays valid permanently — logout gives the user no way to cut it off.

Minimum fix is a revocation set checked during verification:

```python
def logout(response, sid):
    revoked.add(sid, ttl=REFRESH_TTL)
    response.delete_cookie("sid")
```
