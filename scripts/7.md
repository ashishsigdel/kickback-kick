Found it — a module-level cache with no eviction.

`worker/handlers.py:19`

```python
_RESULT_CACHE = {}

def handle(job):
    result = compute(job)
    _RESULT_CACHE[job.id] = result   # never removed
    return result
```

Every job id ever seen stays resident, along with its result object. That's a monotonic leak keyed on something unbounded, which matches the shape of your graph — linear growth, no sawtooth, unaffected by GC.

The `lru_cache` decorators elsewhere in that file are fine; they're bounded at 128 entries by default. This one is a bare dict, so it has no ceiling at all.

I want to see whether anything actually reads from that cache before I propose removing it.

[[thinking]]
This matters. If nothing reads it, deleting it is the whole fix and it's trivially safe. If something does read it, I need to preserve the hit behaviour while bounding the size, and the eviction policy becomes a real decision rather than a detail.

[[response]]
One reader, and it changes the fix.

`worker/handlers.py:47` checks the cache for retries within the same job batch:

```python
if job.id in _RESULT_CACHE:
    return _RESULT_CACHE[job.id]
```

So it isn't dead code — it's deduplicating retries. Deleting it outright would recompute on every retry, which is a real regression under load.

The right shape is a bounded cache with the same lookup semantics:

```python
from functools import lru_cache

@lru_cache(maxsize=4096)
def _cached_compute(job_id, payload_hash):
    return compute_by_id(job_id)
```

4096 entries is well above your observed batch size of ~300, so the dedup behaviour is unchanged in practice while memory is capped.

Before I claim this is fixed I should be honest that a leak fix is only demonstrated by running it long enough to see the curve flatten. A unit test proves the cache bounds; it does not prove the leak is gone in production. I should say exactly that rather than overstate.

What this does and doesn't establish:

- **Established:** the cache cannot exceed 4096 entries, and retry dedup still hits for anything in the recent window.
- **Not established:** that this was your *only* leak. The test proves a bound on one structure; it can't prove RSS flattens in production.

The check worth running is your existing worker under normal load for a few hours, watching whether RSS plateaus. If it still climbs linearly there's a second retainer, and I'd want `tracemalloc` snapshots one cycle apart rather than more static reading.
