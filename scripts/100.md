# Script 100 — a three-way merge corrupts files under a specific edit pattern

[[thinking]]
A three-way merge that silently corrupts output — as opposed to flagging a conflict — is more concerning than a crash, because users won't notice until later. The report says it only happens with edits near the top and bottom of a file simultaneously, which sounds like a line-numbering or hunk-boundary bug rather than a content-diffing bug. I want a minimal reproduction before reading the merge algorithm itself, since the exact shape of the failing case will point straight at the bug.

[[tool_use]]
name: Bash
input:
  command: printf 'a\nb\nc\nd\ne\n' > base.txt && printf 'X\nb\nc\nd\ne\n' > ours.txt && printf 'a\nb\nc\nd\nY\n' > theirs.txt && ./bin/vcmerge base.txt ours.txt theirs.txt
  description: Minimal repro — non-overlapping edits at the start and end of the same file

Actual output:


`theirs`' edit to the last line silently vanished — no conflict marker, no error, just dropped. Since this didn't even warn, it's the dangerous kind of bug: a user would ship a merged file missing changes with no signal anything went wrong.

`internal/merge/diff3.go:88`, the hunk-matching loop:

```go
for _, h := range oursHunks {
    if overlaps(h, theirsHunks[i]) {
        conflicts = append(conflicts, mergeConflict(h, theirsHunks[i]))
    } else {
        applyHunk(result, h)
    }
    i++
}
```

`i` increments unconditionally on every iteration of the *ours* loop, but it's indexing into `theirsHunks` — a completely different slice that doesn't necessarily have the same length or the same alignment as `oursHunks`. This loop only ever applies hunks from `oursHunks`; `theirsHunks` entries are only ever used for the overlap check, never applied on their own when they don't overlap anything from `ours`.

Before proposing a fix I want to understand the intended algorithm shape — this looks like it's missing an entire branch (apply theirs-only hunks) rather than having one line wrong, so I should check whether there's a separate pass elsewhere that's supposed to handle theirs-only hunks and is itself broken, versus this loop being structurally incomplete by design.

[[response]]
Checked for a separate pass — there isn't one. `diff3.go` has exactly one merge loop and it's structured entirely around `oursHunks`, using `theirsHunks[i]` only as a same-index lookup for the overlap check. That's the actual bug: it assumes `oursHunks[i]` and `theirsHunks[i]` are the same edit region by position, which is only true when both sides touch the same regions in the same order — coincidentally true for a lot of real-world small diffs (hence this shipping unnoticed), false the moment one side has an edit the other doesn't, like this repro.

Correct three-way merge needs to walk *both* hunk lists and union them: any hunk present in `ours` but not overlapping anything in `theirs` gets applied; same for `theirs`-only hunks; anything overlapping in both goes to conflict. Rewrote around a merged, sorted hunk boundary list instead of parallel iteration:

```diff
-for _, h := range oursHunks {
-    if overlaps(h, theirsHunks[i]) {
-        conflicts = append(conflicts, mergeConflict(h, theirsHunks[i]))
-    } else {
-        applyHunk(result, h)
-    }
-    i++
-}
+allHunks := mergeSortedByPosition(oursHunks, theirsHunks)
+for _, group := range groupOverlapping(allHunks) {
+    switch {
+    case group.touchedByBoth():
+        conflicts = append(conflicts, mergeConflict(group.ours, group.theirs))
+    case group.touchedByOursOnly():
+        applyHunk(result, group.ours)
+    case group.touchedByTheirsOnly():
+        applyHunk(result, group.theirs)
+    }
+}
```

`groupOverlapping` buckets hunks by line-range overlap regardless of which side they came from or their index, which removes the same-index assumption entirely.
