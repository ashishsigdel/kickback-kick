# Script 34 — cache stampede on the storefront's hottest key

[[thinking]]
A spike of identical slow queries hitting Postgres at the exact moment a popular cache key expires is the signature of a stampede, not a capacity problem — capacity problems degrade gradually, stampedes look like a cliff at a fixed interval. Before touching the cache code I want to confirm the periodicity matches the TTL exactly, since that's what separates "cache expiry caused this" from "something else happens to correlate."

[[response]]
`TTL storefront:homepage:featured` → 187 (seconds), and the slow-query log shows bursts of 40-60 identical `featured_products` queries landing within the same one-second bucket, roughly every 300 seconds — matching a 300s TTL exactly. This is a stampede: the key expires, and every request that arrives in the gap before it's repopulated falls through to Postgres independently instead of one request rebuilding it for everyone.

`views/homepage.py:34`:

```python
def get_featured_products():
    cached = cache.get("storefront:homepage:featured")
    if cached is not None:
        return cached
    products = expensive_featured_query()  # ~340ms, joins 4 tables, scores + ranks
    cache.set("storefront:homepage:featured", products, timeout=300)
    return products
```

Homepage traffic peaks around 900 req/s. In the ~340ms window between expiry and the first successful `cache.set`, every one of those requests sees a miss and runs the full query independently — the slow log bursts are ~50-strong, roughly `900 * 0.34 / 5` scaled down by request distribution across app workers, which is in the right ballpark once you account for Postgres itself slowing under the burst (each query gets slower as more pile on, which is why the burst window stretches past the theoretical 340ms).

[[thinking]]
Two standard fixes here: a distributed lock so only one process recomputes while others wait or serve stale, or probabilistic early expiration so the key gets refreshed before it actually expires and no request ever sees a hard miss. Locking adds latency for waiters and a failure mode if the lock holder dies; early recomputation avoids both but needs the recomputation to happen off the request path or it just shifts the stampede earlier. I want to pick based on how tolerant this data is of staleness, since that determines which trade-off is acceptable.

[[response]]
`featured_products` is homepage merchandising — a few seconds of staleness is a non-issue, but a 340ms query pile-on 50-wide is a real Postgres load problem. That tolerance makes probabilistic early recomputation the better fit over locking: it never makes a request wait, and it spreads recomputation out instead of synchronizing it.

Implemented XFetch-style early expiration — store a computed "delta" (how long the value took to build) alongside the value, and probabilistically treat the key as expired before its real TTL, with probability increasing as it approaches actual expiry:

```python
import random, math, time

def get_featured_products():
    entry = cache.get("storefront:homepage:featured")
    if entry is not None:
        value, delta, expiry = entry["value"], entry["delta"], entry["expiry"]
        # XFetch: recompute early with rising probability as we approach expiry
        beta = 1.0
        if time.time() - beta * delta * math.log(random.random()) < expiry:
            return value
        # else: fall through and recompute, but everyone else still got `value` above

    start = time.time()
    products = expensive_featured_query()
    delta = time.time() - start
    cache.set("storefront:homepage:featured", {
        "value": products, "delta": delta, "expiry": time.time() + 300,
    }, timeout=300 + 30)  # grace window past logical expiry, so a slow recompute doesn't cause a hard miss
    return products
```

The key never hard-expires from Redis's perspective until 330s (the +30 grace window) — the 300s is now a logical expiry checked in application code, so even if recomputation is somehow delayed, in-flight requests still get the (now slightly stale) cached value instead of falling through to Postgres. Only one or a small handful of requests near the boundary probabilistically decide to recompute; everyone else keeps serving the cached value uninterrupted.

Load-tested by replaying the peak traffic pattern (900 req/s, 10 minutes, so ~2 natural expiries) against a staging Redis+Postgres pair:
