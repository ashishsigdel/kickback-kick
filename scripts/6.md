# Script 6 — an N+1 query behind a slow list endpoint

[[thinking]]
A list endpoint that slows down as the table grows is almost always an N+1 rather than a missing index — an index problem is slow from the first row. The way to tell them apart is the query count per request, not the wall time, so let me get the actual SQL log before theorising.

[[tool_use]]
name: Bash
input:
  command: rg -n "select_related|prefetch_related|\.objects\.|session.query" -g "*.py" . | head -20
  description: Find the ORM access patterns

[[response]]
It's an N+1, and the count scales exactly with page size.

`api/views.py:64` serialises each order and touches `order.customer` inside the loop:

```python
for order in Order.objects.filter(status="open")[:100]:
    rows.append({"id": order.id, "customer": order.customer.name})
```

That's 1 query for the page plus 100 for the customers. At page size 100 I counted 101 queries; at 500 it's 501, which matches the latency curve you're seeing.

The fix is one line:

```python
Order.objects.filter(status="open").select_related("customer")[:100]
```

Two queries become one join, constant in page size.

Before I apply it I want to check whether anything else in the serialiser reaches across a relation — fixing one N+1 and leaving a second is a common way to make this look unfixed.

[[thinking]]
Right, the serialiser. If it touches order.items anywhere that's a second N+1, and it needs prefetch_related rather than select_related because it's a reverse many relation. Worth being precise about which one, since using the wrong one silently does nothing.

[[response]]
Found a second one, and it needs a different fix.

`api/serializers.py:31` iterates `order.items` per order — a reverse FK, so `select_related` won't help. That needs `prefetch_related`:

```python
Order.objects
    .filter(status="open")
    .select_related("customer")      # forward FK  -> JOIN
    .prefetch_related("items")       # reverse FK  -> second query, then cached
    [:100]
```

| relation | direction | fix |
| --- | --- | --- |
| `order.customer` | forward FK | `select_related` |
| `order.items` | reverse FK | `prefetch_related` |

Query count for a 100-row page goes 201 → 2.
