# Script 42 — reviewing a proposed index strategy for a slow report

[[thinking]]
Before judging the proposed indexes I need the actual plan, not a guess based on the schema. A reporting query that scans a big table is often bottlenecked on something the proposal doesn't even touch — a bad join order, a sort spilling to disk, a function on the filtered column that defeats any index regardless of which one exists. Get `EXPLAIN ANALYZE` first, form opinions second.

[[tool_use]]
name: Bash
input:
  command: psql "$REPORTING_DB_URL" -c "EXPLAIN (ANALYZE, BUFFERS) SELECT r.region, date_trunc('month', o.created_at) AS month, sum(o.total_cents) FROM orders o JOIN regions r ON r.id = o.region_id WHERE o.created_at >= now() - interval '18 months' GROUP BY 1,2 ORDER BY 1,2;"
  description: Get the real plan for the reporting query as it runs today

[[response]]
Plan confirms the pain point, and it's not what the proposal targets.

```
GroupAggregate  (cost=... rows=214 width=48) (actual time=4102.331..4980.221 rows=204 loops=1)
  ->  Sort  (cost=... ) (actual time=4098.902..4301.114 rows=1,812,004 loops=1)
        Sort Key: r.region, (date_trunc('month', o.created_at))
        Sort Method: external merge  Disk: 187,344kB
        ->  Hash Join  (actual time=41.220..3210.885 rows=1,812,004 loops=1)
              ->  Seq Scan on orders o  (actual time=0.021..1802.442 rows=1,812,004 loops=1)
                    Filter: (created_at >= (now() - '18 mons'::interval))
                    Rows Removed by Filter: 6,204,118
              ->  Hash  (actual time=8.011..8.011 rows=40 loops=1)
                    ->  Seq Scan on regions r
Planning Time: 0.412 ms
Execution Time: 4980.981 ms
```

Two real costs: the seq scan on `orders` reading 8M rows to keep 1.8M (1802ms), and a disk sort for the group-by (1090ms of the total, `Sort Method: external merge`). `work_mem` is 64MB on this instance, well under the 187MB the sort needs.

[[thinking]]
The proposal is a single index: `CREATE INDEX ON orders (region_id)`. That helps the join but does nothing for the seq scan's actual filter (`created_at`) or the sort. I want to check column order and whether a composite index could serve the filter, the join, and let the sort ride an index order instead of spilling — that's three birds if the column order is right, not three separate indexes.

[[response]]
The proposed index — `CREATE INDEX ON orders (region_id)` — only addresses the join, and the join was never the expensive part (`Hash` on `regions` took 8ms). It would leave the seq scan and the disk sort untouched, which is 90% of the runtime.

What I'd propose instead:

```sql
CREATE INDEX CONCURRENTLY idx_orders_created_region
  ON orders (created_at, region_id)
  INCLUDE (total_cents);
```

Reasoning for the column order:

| goal | how this index serves it |
| --- | --- |
| filter `created_at >= now() - 18mo` | leading column, range scan instead of seq scan |
| join to `regions` | `region_id` present, index-only join possible |
| avoid heap fetch per row | `total_cents` in `INCLUDE`, so the scan never touches the heap |

It won't eliminate the sort — `date_trunc('month', ...)` isn't the same key as raw `created_at`, so Postgres can't use index order for the group-by directly. That's a separate, smaller fix: bump `work_mem` for this reporting connection (`SET work_mem = '256MB'` scoped to the reporting role, not global) so the sort stays in memory instead of spilling to disk.

[[thinking]]
I should validate this against the real data rather than assert it, especially since `CREATE INDEX CONCURRENTLY` on an 8M-row table isn't free and I don't want to recommend it twice. Testing on a replica first is the right call here, not production.

[[response]]
Ran it on the read replica (not production — this table is 8M rows and `CONCURRENTLY` still takes a write-lock-free but I/O-heavy pass):

```

```

Re-ran `EXPLAIN ANALYZE` against the replica with the index and `work_mem` bumped to 256MB:

```
GroupAggregate  (actual time=210.442..298.107 rows=204 loops=1)
  ->  Sort  (actual time=208.991..221.340 rows=1,812,004 loops=1)
        Sort Method: quicksort  Memory: 198,432kB
        ->  Index Only Scan using idx_orders_created_region on orders o
              (actual time=0.031..142.884 rows=1,812,004 loops=1)
              Heap Fetches: 0

```

4980ms → 299ms, roughly 16x. Heap fetches at 0 confirms the `INCLUDE` is doing its job.

I'd hold off on the single-column `region_id` index from the original proposal — it wouldn't help this query and it's another index for every `orders` write to maintain. If there's a different query it was meant for, worth naming it explicitly rather than adding it speculatively.
