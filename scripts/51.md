# Script 51 — reviewing an async/await refactor for swallowed errors

[[thinking]]
A refactor from `.then()` chains to `async/await` almost always looks cleaner and almost always hides the same class of bug: a call that returns a promise but isn't awaited doesn't block execution and, worse, if it rejects, that rejection can go entirely unhandled or get attributed to the wrong place. Grepping for every async function call in the diff and checking each one is awaited is the mechanical way to catch this — eyeballing a big diff for "missing await" is exactly the kind of thing that's easy to miss by reading top to bottom.

[[tool_use]]
name: Bash
input:
  command: git diff main -- src/services/checkout.ts | grep -nE "^\+.*(await |\.\w+\()" | grep -v "await " | grep -E "\w+Async\(|save\(|send\(|create\(|update\(|delete\(|find\("
  description: Find calls to likely-async methods in the diff that are missing an await

[[response]]
Three call sites in the diff look like calls to async functions with no `await` in front of them.

`src/services/checkout.ts:104`:

```ts
async function finalizeOrder(order: Order): Promise<void> {
  updateInventory(order.items);   // <- no await
  await sendConfirmationEmail(order);
  order.status = "completed";
  await orderRepo.save(order);
}
```

Confirmed `updateInventory` is itself `async` (`src/services/inventory.ts:22`, returns `Promise<void>`). Without `await`, `finalizeOrder` doesn't wait for it — it fires, and execution moves straight to `sendConfirmationEmail`. Two consequences: order confirmation can be sent before inventory is actually decremented (a race if two orders for the last unit complete close together), and if `updateInventory` rejects, that rejection is an unhandled promise rejection completely detached from `finalizeOrder`'s call stack — no catch block anywhere in this function will ever see it.

[[thinking]]
Before flagging this as a bug I should check whether it's actually intentional — some refactors deliberately fire-and-forget a non-critical side effect for latency reasons, and if that's the case here the fix isn't "add await," it's "the missing await is fine, but it needs an explicit `.catch()` so the rejection doesn't vanish silently." I need to know whether inventory update is meant to block order completion or not before prescribing the fix.

[[response]]
Checked git blame and the surrounding logic — `updateInventory` isn't a fire-and-forget side effect, it's inside the same function that later checks `order.status = "completed"` and persists it, and there's no comment or test suggesting it's meant to run in the background. The two other flagged calls are less severe but the same shape:

`src/services/checkout.ts:118` — `logAuditEvent(order.id, "checkout")`, also async, not awaited. This one probably is fine to fire-and-forget (audit logging shouldn't block the response), but it currently has no `.catch()`, so a rejection here becomes an unhandled rejection too — just a lower-stakes one.

`src/services/checkout.ts:131`, inside a `.map()`:

```ts
items.map(async (item) => {
  await reserveStock(item);
});
```

This is the classic `.map()`-with-async trap — `.map()` doesn't await anything, it returns an array of unresolved promises and the calling code (`checkout.ts:130`) doesn't do anything with that array. All the `reserveStock` calls fire concurrently and none of them are waited on by the caller, so `finalizeOrder` can proceed to `orderRepo.save(order)` before any stock is actually reserved.

[[thinking]]
Three fixes, three different intents: `updateInventory` needs a real `await` because completion genuinely depends on it; `logAuditEvent` needs an explicit `.catch()` since fire-and-forget is fine but silent-drop isn't; the `.map()` needs `Promise.all` since the intent was clearly "reserve all items," not "reserve items eventually, who knows when."

[[response]]
```diff
 async function finalizeOrder(order: Order): Promise<void> {
-  updateInventory(order.items);
-  await sendConfirmationEmail(order);
+  await updateInventory(order.items);
+  await sendConfirmationEmail(order);
   order.status = "completed";
   await orderRepo.save(order);
 }
```

```diff
-  logAuditEvent(order.id, "checkout");
+  logAuditEvent(order.id, "checkout").catch((err) =>
+    logger.warn("audit_log.failed", { orderId: order.id, err })
+  );
```

```diff
-  items.map(async (item) => {
-    await reserveStock(item);
-  });
+  await Promise.all(items.map((item) => reserveStock(item)));
```

For the `.map()` case I used `Promise.all` rather than a `for...of` loop with sequential `await`, since the original clearly intended concurrent reservation (independent items, no ordering dependency) — a sequential loop would work but would needlessly serialize N independent network calls.

`npm test -- checkout` → 31 passed, including a new test asserting `finalizeOrder` rejects (rather than silently completing) when `updateInventory` throws — that's the regression test for the bug that was here, since the old code had no way to fail this test even with a broken inventory call.
